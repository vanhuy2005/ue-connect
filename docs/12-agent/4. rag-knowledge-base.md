# RAG Knowledge Base

## 1. Purpose

This file defines how UEConnect documentation should be indexed and retrieved for AI agents.

The goal is to make agents retrieve the right context before writing code, docs, tests, API specs, or UI.

---

## 3. Chunking Strategy

Chunk markdown by headings:

H1 = document identity
H2 = major section
H3 = detailed rule

Recommended chunk size:

600-1200 tokens

Overlap:

100-150 tokens

Do not split tables randomly.

## 4. Metadata

Each chunk should include:

file_path
module
title
section_heading
priority
status
last_updated
source_type
related_feature

Example:

{
  "file_path": "docs/03-product/feature-specs/messaging.md",
  "module": "03-product",
  "source_type": "feature_spec",
  "related_feature": "messaging",
  "priority": "P0"
}
## 5. Retrieval Rules

For any task, retrieve:

## 1. source-of-truth map
## 2. feature spec
## 3. state machine
## 4. API contract if endpoint involved
## 5. DB schema/table spec if data involved
## 6. design/page spec if UI involved
## 7. security docs if privacy/admin/moderation involved
## 8. quality docs if test involved
## 6. Ranking Priority

Boost these files:

STATE-MACHINE-SOURCE-OF-TRUTH.md
feature-specs/*.md
api.yaml
schema.md
table-specifications.md
architecture-decision-records/*.md
design-system.md
page-specs/*.md
error-codes.md

Lower priority:

old drafts
examples
assets
appendix
legacy docs
## 7. Retrieval Output Requirement

Agent must cite retrieved docs in its working summary:

Docs consulted:
- docs/03-product/feature-specs/messaging.md
- docs/06-database/schema.md
- docs/07-api/api.yaml

## 8. Staleness Rule

If retrieved docs conflict:

state machine > feature spec > API > database > architecture > design > overview

If still unclear, agent must create a conflict note.

# Cách triển khai tự động update docs thật sự

Bạn có 3 mức triển khai.

## Mức 1: Manual nhưng có agent checklist

Dễ nhất, phù hợp hiện tại.

Workflow:

```txt
Bạn giao task cho agent
→ Agent đọc source-of-truth-map.md
→ Agent làm code/docs
→ Agent liệt kê docs cần update
→ Bạn review

Ưu điểm:

dễ làm
không cần setup phức tạp
giảm hallucination

Nhược điểm:

phụ thuộc discipline
Mức 2: Có script kiểm tra docs trong CI

Thêm vào repo:

scripts/validate-docs.js
scripts/check-api-yaml.js
scripts/check-mermaid.js
scripts/check-doc-frontmatter.js

CI check:

api.yaml valid không
link docs có hỏng không
frontmatter có thiếu không
file quan trọng có tồn tại không
CHANGELOG có update không nếu folder lớn thay đổi

Cái này rất nên làm. Máy móc ít nhất còn biết la đều đặn, không như con người chỉ nhớ lúc production cháy.

Mức 3: Agent tự tạo docs update khi PR thay đổi

Dùng workflow:

git diff
→ detect changed files
→ map changed files to docs
→ agent proposes doc patch
→ human review
→ merge

Ví dụ mapping:

app/Models/Message.php changed
→ check docs/03-product/feature-specs/messaging.md
→ check docs/06-database/schema.md
→ check docs/07-api/api.yaml

routes/api.php changed
→ check docs/07-api/api.yaml
→ check docs/07-api/error-codes.md

database/migrations/* changed
→ check docs/06-database/*
