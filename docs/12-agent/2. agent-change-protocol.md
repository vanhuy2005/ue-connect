# Agent Change Protocol

## 1. Purpose

This file defines how AI agents must handle changes in UEConnect.

No implementation change is complete until impacted documentation is checked.

---

## 2. Change Classification

Every task must be classified as one of:

```txt
feature_change
api_change
database_change
ui_change
security_change
business_rule_change
bug_fix
refactor
documentation_only
## 3. Required Documentation Updates
### 3.1. If API changes

Update:

docs/07-api/api.yaml
docs/07-api/api-overview.md if global rule changes
docs/07-api/error-codes.md if new error code added
related feature API file

Also check:

frontend consumers
tests
acceptance criteria
### 3.2. If Database changes

Update:

docs/06-database/schema.md
docs/06-database/table-specifications.md
docs/06-database/erd.mmd
docs/06-database/migration-strategy.md if migration rule changes

Also check:

models
migrations
seeders
factories
tests
API schemas
### 3.3. If Feature behavior changes

Update:

docs/03-product/feature-specs/{feature}.md
docs/03-product/state-machines/*
docs/02-requirements/acceptance-criteria.md
docs/02-requirements/edge-cases.md
docs/02-requirements/traceability-matrix.md
### 3.4. If UI changes

Update:

docs/04-design/page-specs/*
docs/04-design/ui-states/*
docs/04-design/design-system.md if token/component changes
docs/03-product/sitemap.md if route/page changes
### 3.5. If Security changes

Update:

docs/08-security/*
docs/07-api/api.yaml
docs/03-product/feature-specs/safety-reporting.md
docs/03-product/feature-specs/moderation.md
## 4. Changelog Rule

Update CHANGELOG.md when the change affects:

feature behavior
API contract
database schema
security policy
admin workflow
state machine
release scope

No changelog needed for typo-only changes.

## 5. ADR Rule

Create an ADR when the change introduces or changes:

tech stack
auth strategy
database choice
realtime strategy
permission architecture
storage architecture
AI integration
deployment model

ADR path:

docs/05-system-architecture/architecture-decision-records/
## 6. Conflict Rule

If implementation and docs disagree:

## 1. Do not silently follow code.
2. Check source-of-truth priority.
## 3. Create a conflict note.
4. Propose exact docs/code update.
## 5. Do not invent missing business rules.
