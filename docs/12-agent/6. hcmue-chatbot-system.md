# HCMUE AI Chatbot — System Architecture

## 1. Purpose

Tài liệu này mô tả kiến trúc kỹ thuật chi tiết của hệ thống **HCMUE Academic Chatbot** tích hợp trong UEConnect.

Hệ thống là một **production-grade Retrieval-Augmented Generation (RAG)** pipeline, được thiết kế để trả lời các câu hỏi học thuật về chương trình đào tạo, học phần, quy chế học vụ và sổ tay sinh viên của Trường Đại học Sư phạm TP.HCM (HCMUE).

Phạm vi tài liệu này bao gồm:

- Kiến trúc pipeline 10 bước (end-to-end)
- Cơ chế query routing và intent classification
- Data ingestion pipeline (PDF → Chunking → Embedding → Qdrant)
- LLM provider strategy và fallback chain
- Vector store schema (Qdrant)
- Database tables (SQL Server)
- Evaluation & quality thresholds
- Environment variables

## 3. Query Routing

The `QueryRouterService` classifies each incoming question into one of five routes:

| Route | Triggered When | Retrieval Strategy |
|---|---|---|
| `structured_db` | Questions about specific program, credits, required/elective courses, graduation requirements | SQL Server queries via Eloquent repositories (faculties, majors, training_programs, curriculum_courses) |
| `rag` | Policy questions, student handbook, academic regulations, general HCMUE information | Qdrant dense vector search + BGE-M3 embeddings |
| `hybrid` | Questions needing both program structure context AND policy explanation | Both paths run concurrently, results merged before composing |
| `none/clarification` | Missing required entity (cohort or major not detected) | Return targeted clarification question to user |
| `none/unsupported` | Question is outside academic domain scope | Return scope rejection message |

**Fallback rule:** If `structured_db` finds no result, the pipeline automatically falls back to `rag` route and re-runs retrieval.

## 5. Data Ingestion Pipeline

### 5.1 Training Programs (Structured Path)

```
database/AI/Chuongtrinhdaotao/
└── {Year} - Khóa {N}/
    └── Khoa/
        └── {Tên Khoa}/
            └── Ngành/
                └── {Tên Ngành}/
                    ├── Chuandaura/         ← Chuẩn đầu ra (PDF)
                    └── Chuongtrinhkhung/   ← Chương trình khung (PDF)

TrainingProgramImportService
    ├── Reads folder structure → extracts Cohort / Faculty / Major metadata
    ├── TrainingProgramPdfParser → extracts raw text from PDFs
    ├── TrainingProgramStructuredExtractor → LLM parses PDF into structured data
    └── Persists to SQL Server:
        ├── admission_cohorts (khóa tuyển sinh)
        ├── faculties (khoa)
        ├── majors (ngành)
        ├── training_programs (CTĐT per cohort + major)
        ├── curriculum_courses (học phần trong CTĐT)
        ├── knowledge_source_documents (PDF catalog)
        └── document_chunks (indexed metadata + Qdrant point_id)
```

### 5.2 Student Handbook & Regulations (RAG Path)

```
PDF Upload (via SourceDocumentUploadService)
    │
    ▼
AcademicDocumentTextExtractor
    → Extract raw text (PDF parser / OCR fallback)
    → PdfTextQualityAnalyzer → score text quality
    │
    ▼
AcademicDocumentChunker
    → Split by semantic sections (H1/H2 headings, article boundaries)
    → Target chunk size: 600–1200 tokens
    → Overlap: 100–150 tokens
    │
    ▼
AcademicMetadataExtractor
    → Tag each chunk: document_type, knowledge_type,
      loai_tai_lieu, khoa_hoc, nganh, article, page_start
    │
    ▼
EmbeddingService → BgeM3EmbeddingProvider
    → Generate 1024-dimensional BGE-M3 dense vector
    │
    ▼
QdrantVectorStore
    → Upsert vector + payload to collection: hcmue_knowledge
    │
    ▼
DocumentChunkRepository
    → Store chunk metadata + qdrant_point_id in SQL Server
```

## 6. Vector Store Schema (Qdrant)

```json
{
  "collection_name": "hcmue_knowledge",
  "vectors_config": {
    "size": 1024,
    "distance": "Cosine"
  },
  "payload_schema": {
    "document_id":    "integer   — FK to knowledge_source_documents",
    "document_name":  "string    — Human-readable document title",
    "document_type":  "string    — training_program | student_handbook | regulation",
    "knowledge_type": "string    — chuong_trinh_dao_tao | so_tay_sinh_vien | quyet_dinh",
    "loai_tai_lieu":  "string    — chuandaura | chuongtrinhkhung | handbook_chapter",
    "khoa_hoc":       "string    — Cohort year (e.g. '2024 - Khóa 50')",
    "nganh":          "string    — Major name",
    "article":        "string    — Article/section reference (e.g. 'Điều 5')",
    "page_start":     "integer   — Source PDF page number",
    "chunk_index":    "integer   — Position within document"
  }
}
```

## 7. LLM Provider Strategy

| Priority | Provider | Model | Use Case |
|---|---|---|---|
| 1 (default) | Google Gemini | `gemini-2.0-flash` | Primary cloud inference — fast, multimodal |
| 2 (local) | Ollama | `gemma4:e2b` | On-device inference for 8GB RAM machines |
| 3 (cloud fallback) | OpenRouter | Configurable | Secondary cloud if Gemini unavailable |
| Embedding | BGE-M3 | `bge-m3` (1024-dim) | Dense retrieval — multilingual, Vietnamese-optimized |
| Embedding (alt) | Gemini Embedding | `gemini-embedding-001` (1024-dim) | Cloud embedding fallback |

### 7.1 Resilience Mechanisms

| Mechanism | Description |
|---|---|
| **Multi-key rotation** | `GeminiKeyManager` rotates across `GEMINI_API_KEY_1/2/3` to distribute quota load |
| **Ollama → Gemini fallback** | If Ollama fails, auto-switches to Gemini (`OLLAMA_FALLBACK_ENABLED=true`, `OLLAMA_FALLBACK_PROVIDER=gemini`) |
| **Structured DB → RAG fallback** | If `StructuredRetrievalService` returns empty, pipeline retries with `RagRetrievalService` |
| **Citation grounding** | `CitationVerifierService` cross-checks answer claims against retrieved sources |
| **Hallucination guard** | `HallucinationGuardService` strips claims with no source support |

## 8. Evaluation & Quality Thresholds

All retrieval scores and LLM answers are logged for continuous improvement:

| Variable | Value | Purpose |
|---|---|---|
| `AI_MIN_RETRIEVAL_SCORE` | `0.55` | Minimum Qdrant cosine score to include a chunk |
| `AI_RETRIEVAL_TOP_K` | `8` | Max chunks retrieved from Qdrant |
| `AI_RERANK_TOP_K` | `5` | Chunks kept after reranking |
| `AI_TOTAL_CREDITS_TOP_K` | `20` | Credit query result limit |
| `AI_VERIFICATION_LIKELY_MATCH_THRESHOLD` | `0.85` | Auto-approve confidence score |
| `AI_VERIFICATION_MANUAL_REVIEW_THRESHOLD` | `0.65` | Queue for human review |
| `AI_VERIFICATION_SUSPICIOUS_THRESHOLD` | `0.45` | Flag as suspicious |
| `OLLAMA_EVALUATION_PASS_THRESHOLD` | `0.70` | Ollama answer pass score |
| `OLLAMA_CITATION_PASS_THRESHOLD` | `0.75` | Ollama citation pass score |

## 9. Database Tables (SQL Server)

### 9.1 Chat & AI Tables

| Table | Purpose |
|---|---|
| `chat_sessions` | User chat sessions (session_id, user_id, title) |
| `ai_questions` | Per-turn question log (original, normalized, intent, route, confidence) |
| `ai_answers` | Final answers (provider, model, token usage, latency_ms, status) |
| `ai_retrieved_chunks` | RAG chunks used per question (chunk_id, score, rerank_score) |
| `ai_structured_queries` | Structured DB query plans and result counts |

### 9.2 Knowledge Base Tables

| Table | Purpose |
|---|---|
| `knowledge_source_documents` | Document catalog (name, type, file path, ingestion status) |
| `document_chunks` | Chunked text segments with qdrant_point_id reference |
| `knowledge_batches` | Batch ingestion job tracking and status |

### 9.3 Academic Catalog Tables

| Table | Purpose |
|---|---|
| `admission_cohorts` | Year/cohort catalog (e.g. "2024 - Khóa 50") |
| `faculties` | Faculty catalog (khoa) |
| `majors` | Major catalog (ngành), linked to faculty |
| `training_programs` | CTĐT per cohort + major (total credits, years, type) |
| `curriculum_courses` | Courses within each CTĐT (code, name, credits, type, semester) |
