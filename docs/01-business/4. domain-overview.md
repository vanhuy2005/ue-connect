# Domain Overview

## 1. Purpose

File này mô tả tổng quan các **business domain** của UEConnect.

Mục tiêu:

- Chia UEConnect thành các domain nghiệp vụ rõ ràng.
- Xác định trách nhiệm của từng domain.
- Xác định domain nào là core, supporting, operational, future-ready.
- Làm rõ mối quan hệ giữa các domain.
- Làm nền cho use cases, user flows, functional requirements, database design, system architecture và service/module planning.
- Tránh việc các feature bị thiết kế rời rạc, trùng trách nhiệm hoặc đặt tên lung tung.

Domain overview trả lời câu hỏi:

```txt
UEConnect gồm những vùng nghiệp vụ nào, mỗi vùng chịu trách nhiệm gì, và chúng liên kết với nhau ra sao?
````

`domain-overview.md` mô tả bản đồ nghiệp vụ bên trong sản phẩm. Không có bản đồ này, team sẽ tự phát hiện domain bằng bug report. Một phương pháp học tập đau đớn nhưng phổ biến.

---

## 2. Domain Overview Summary

UEConnect là một verified student social platform dành riêng cho HCMUE.

Sản phẩm có các domain chính:

```txt
01. Identity & Account Domain
02. Verification Domain
03. Role & Permission Domain
04. Profile Domain
05. Social Feed Domain
06. Discovery & Connection Domain
07. Messaging Domain
08. Mentor & Growth Domain
09. Community / Club Domain
10. Notification Domain
11. Safety & Reporting Domain
12. Moderation Domain
13. Admin Operations Domain
14. Audit & Compliance Domain
15. Privacy & Settings Domain
16. Content & Media Domain
17. Search & Recommendation Domain
18. PWA / Client Experience Domain
19. Analytics & Metrics Domain
20. Support / Help Domain
```

Các domain này không ngang nhau về độ ưu tiên.

Có domain là lõi sản phẩm, có domain hỗ trợ, có domain vận hành, có domain chuẩn bị cho tương lai. Nếu xem tất cả đều quan trọng như nhau, đó không phải enterprise thinking. Đó là panic planning.

# 5. Domain 01 — Identity & Account Domain

## 5.1. Definition

```txt
Identity & Account Domain quản lý tài khoản người dùng, trạng thái tài khoản, đăng ký, đăng nhập và vòng đời account.
```

Domain này là cửa vào của UEConnect.

## 5.2. Responsibilities

- Register.
- Login.
- Logout.
- Forgot password.
- Reset password.
- Session management.
- Account status.
- Account lifecycle.
- Account restriction.
- Account deactivation/delete direction.
- Linking account to verification.
- Redirect user based on account/verification status.

## 5.3. Core Concepts

```txt
User
Account
Account Status
Session
Authentication
Password Reset
Account Lifecycle
```

## 5.4. Main Actors

| Actor   | Interaction            |
| ------- | ---------------------- |
| Guest   | Register, login        |
| Student | Login, manage account  |
| Alumni  | Login, manage account  |
| Advisor | Login, manage account  |
| Admin   | Manage user account    |
| System  | Enforce account status |

## 5.5. Key Business Rules

- Guest chưa đăng nhập chỉ xem được landing/auth/public info.
- User cần account để submit verification.
- Account chưa verified không được dùng full product.
- Account có thể ở trạng thái active, pending, rejected, suspended, banned, deactivated.
- Account bị suspended/banned bị giới hạn quyền.
- Account lifecycle quan trọng cần audit nếu do admin action.

## 5.6. Example States

```txt

active
rejected_verification
need_more_information
suspended
banned
deactivated
deleted
```

## 5.7. Related Features

- Public landing.
- Signup.
- Login.
- Account status page.
- Verification entry.
- Profile setup redirect.
- Admin user detail.

## 5.8. Related Requirements

- Authentication requirement.
- Session security.
- Account status access control.
- Password reset.
- Account suspension handling.

## 5.9. Related Data Entities

```txt

```

# 6. Domain 02 — Verification Domain

## 6.1. Definition

```txt
Verification Domain quản lý quy trình xác thực user thuộc HCMUE và xác định primary system role ban đầu.
```

Đây là domain nền tảng của trust.

## 6.2. Responsibilities

- Collect verification information.
- Validate MSSV format.
- Enforce MSSV uniqueness.
- Accept evidence upload.
- Support hcmue.edu.vn email as verification signal.
- Create verification request.
- Track verification status.
- Allow resubmission.
- Support admin review.
- Approve/reject/request more information.
- Protect evidence.
- Audit verification actions.

## 6.3. Core Concepts

```txt
Verification Request
Verification Evidence
MSSV / Student ID
Verified UEer
Verification Status
Evidence Type
Admin Review
```

## 6.4. Main Actors

| Actor   | Interaction                         |
| ------- | ----------------------------------- |
| Student | Submit MSSV/evidence                |
| Alumni  | Submit alumni evidence              |
| Advisor | Submit advisor evidence             |
| Admin   | Review/approve/reject               |
| System  | Check duplicate MSSV, update status |

## 6.5. Key Business Rules

- Một MSSV chỉ được liên kết với một tài khoản duy nhất.
- Student cần MSSV nếu có thể xác định.
- Alumni/advisor có thể dùng evidence khác phù hợp.
- Evidence là dữ liệu nhạy cảm.
- Evidence không public.
- Admin action trong verification phải audit.
- Reject cần reason.
- Need more information cần chỉ rõ user cần bổ sung gì.
- Verified user được cấp primary role tương ứng.

## 6.7. Related Features

## 6.8. Related Requirements

# 10. Domain 06 — Discovery & Connection Domain

## 10.1. Definition

```txt
Discovery & Connection Domain giúp user khám phá UEers phù hợp, gửi lời chào và tạo connection có ngữ cảnh.
```

Đây là domain tạo giá trị kết nối, nhưng cũng là domain dễ bị hiểu sai thành dating nếu thiết kế bậy. Và thiết kế bậy thì thật sự không cần nhiều công sức, chỉ cần một chữ “match” là đủ phá sản phẩm.

## 10.2. Responsibilities

- Show discovery profile cards.
- Filter UEers.
- Explain shared context.
- Send greeting.
- Accept/decline greeting.
- Cancel greeting.
- Create connection after accepted greeting.
- Manage connection states.
- Save profile.
- Report profile.
- Block user.
- Enable messaging after connection.
- Avoid dating language/interaction.

## 10.3. Core Concepts

```txt
Discovery
Discovery Profile Card
Greeting
Connection
Connection State
Shared Context
Profile Save
Discovery Filter
```

## 10.4. Main Actors

| Actor           | Interaction                                 |
| --------------- | ------------------------------------------- |
| Student         | Discover/send greeting/connect              |
| Alumni          | May be discoverable depending on visibility |
| Advisor/Mentor  | Discoverable in mentor context              |
| Admin/Moderator | Review reported profile                     |
| System          | Suggest/filter profiles                     |

## 10.5. Key Business Rules

- Discovery không dùng match/swipe/crush/hot/dating.

- Discovery card phải có context học tập/cộng đồng.
- Messaging không mở tự do, cần accepted connection hoặc accepted mentor request.
- Blocked users không được tiếp tục tương tác trực tiếp.
- Profile visibility ảnh hưởng discovery.
- Report profile phải vào moderation.

# 27. Domain Boundary Rules

## 27.1. Identity vs Verification

Identity & Account trả lời:

```txt
User có account không? Account đang ở trạng thái nào?
```

Verification trả lời:

```txt
User có được xác thực là thành viên phù hợp của HCMUE không?
```

Không trộn hai domain này. Một user có account nhưng chưa verified là chuyện bình thường.

## 27.2. Role vs Permission

Role trả lời:

```txt
User là ai trong hệ thống?
```

Permission trả lời:

```txt
User được làm gì?
```

Ví dụ:

```txt
Student + Club Manager permission
Alumni + Mentor permission
Advisor + Content Moderator permission
```

Đừng nhét tất cả vào role. Đó là cách bảng `users.role` biến thành nghĩa địa logic.

## 27.3. Discovery vs Messaging

Discovery trả lời:

```txt
User tìm ai và gửi lời chào thế nào?
```

Messaging trả lời:

```txt
Khi đã đủ điều kiện, user trò chuyện thế nào?
```

Messaging không được tự mở chỉ vì user thấy profile.

## 27.4. Report vs Moderation

Report trả lời:

```txt
User báo cáo vấn đề thế nào?
```

Moderation trả lời:

```txt
Admin/moderator xử lý báo cáo đó thế nào?
```

Report không tự xử lý nội dung. Moderation mới tạo action.

## 27.5. SQL Server Audit vs Application Audit Log

SQL Server Audit trả lời:

```txt
Database/server level event nào xảy ra?
```

Application Audit Log trả lời:

```txt
Nghiệp vụ nào đã được thực hiện bởi ai, với lý do gì?
```

Không dùng SQL Server Audit để thay app audit. Database không tự biết “admin reject vì minh chứng mờ”. Nó chỉ biết con người lại sửa dữ liệu.
