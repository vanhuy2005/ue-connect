# Business Rules

## 1. Purpose

File này định nghĩa **business rules** chính thức của UEConnect.

Mục tiêu:

- Chốt luật nghiệp vụ cốt lõi của sản phẩm.
- Làm rõ user nào được làm gì, khi nào, trong điều kiện nào.
- Làm rõ các rule về verification, role, permission, profile, feed, discovery, messaging, mentor, community, report, moderation, audit và privacy.
- Làm source of truth cho requirements, use cases, user flows, page specs, database design, backend policy và test cases.
- Ngăn việc mỗi module tự hiểu luật theo một kiểu riêng, vì không gì vui bằng một hệ thống mà frontend cho phép, backend từ chối, database thì silently accept.

Business rules không phải gợi ý. Đây là những quy tắc mà sản phẩm phải tuân thủ.

Nếu requirement mâu thuẫn với business rules, requirement sai.

Nếu UI flow mâu thuẫn với business rules, UI flow sai.

Nếu database design mâu thuẫn với business rules, database design đang chuẩn bị tạo nghiệp.

---

## 2. Business Rule Principles

Tất cả business rules của UEConnect phải tuân thủ các nguyên tắc nền:

```txt
01. HCMUE-only
02. Verified-first
03. One account per verified identity
04. One primary system role per user
05. Permission is separate from role
06. Social but not dating
07. Messaging must be permission-based
08. Community must have governance
09. Safety is mandatory
10. Admin actions must be auditable
11. Sensitive data must be protected
12. MVP must not become a super app
````

# BR-A. Product Scope Rules

## BR-A001 — UEConnect is HCMUE-only

### Rule

```txt
UEConnect chỉ phục vụ cộng đồng HCMUE.
```

### Explanation

Sản phẩm không phải public social network và không phải multi-university platform.

### Applies To

- Registration.
- Verification.
- Landing copy.
- Discovery.
- Community.
- Mentor.
- Admin policy.

### Requirement Implication

- Verification phải xác định user có liên hệ hợp lệ với HCMUE.
- Landing page phải nói rõ UEConnect dành cho HCMUE.
- Không thiết kế tenant/multi-university ở MVP.

### Exceptions

Không có trong MVP.

---

## BR-A002 — UEConnect is not a dating app

### Rule

```txt
UEConnect không được thiết kế, đặt tên, viết copy hoặc vận hành như dating app.
```

### Forbidden Terms

```txt
match
swipe
crush
hot
dating
ghép đôi
tán tỉnh
người yêu
hẹn hò
```

### Preferred Terms

```txt
UEer
Gửi lời chào
Kết nối
Bạn cùng khoa
Cùng học
Mentor
Cộng đồng
CLB
Khám phá
```

### Requirement Implication

- Discovery không dùng dating language.
- Profile card không appearance-first.
- CTA chính là `Gửi lời chào`, không phải `Match`.
- UI không dùng Tinder-like copy.

### Rationale

Nếu app trường học bị hiểu là dating app, trust chết rất nhanh. Một cái chết có thể tránh bằng vài dòng glossary, thật bi hài.

---

## BR-A003 — UEConnect is not a student portal

### Rule

```txt
UEConnect không thay thế student portal, hệ thống học vụ, điểm số, đăng ký môn hoặc học phí.
```

### Out of Scope

```txt
Đăng ký môn học
Quản lý điểm chính thức
Thanh toán học phí
Lịch học chính thức
Hồ sơ học vụ chính thức
```

### Requirement Implication

- Không tạo module học vụ chính thức trong MVP.
- Không thiết kế dashboard kiểu portal hành chính.
- Product UI phải social-first, không portal-first.

---

## BR-A004 — UEConnect is not a super app

### Rule

```txt
UEConnect không được mở rộng thành super app sinh viên trong MVP.
```

### Out of Scope

```txt
Marketplace
Payment
Job board đầy đủ
Livestream
Stories / Reels
Dating
Anonymous confession
Multi-university
Native mobile app
```

### Requirement Implication

- Feature mới phải map với product vision hoặc roadmap.
- Nếu feature không phục vụ verified social, connection, mentor, community, safety hoặc governance thì không vào MVP.

# BR-B. Account & Identity Rules

## BR-B001 — User must have an account before verification

### Rule

```txt
User phải tạo account trước khi submit verification.
```

### Applies To

- Register.
- Verification.
- Account status.

### Requirement Implication

- Guest chỉ có thể xem landing, register, login.
- Verification request phải belong to one account.

---

## BR-B002 — Account access depends on account status

### Rule

```txt
Quyền truy cập của user phụ thuộc vào account status và verification status.
```

### Account Statuses

```txt
registered
pending_verification
active
need_more_information
rejected_verification
suspended
banned
deactivated
deleted
```

### Access Matrix Summary

| Account Status          | Access                                       |
| ----------------------- | -------------------------------------------- |
| `registered`            | Submit verification                          |
| `pending_verification`  | View account status                          |
| `need_more_information` | Resubmit evidence                            |
| `rejected_verification` | View reason / support / resubmit if allowed  |
| `active`                | Full product access based on role/permission |
| `suspended`             | Limited access                               |
| `banned`                | No product access                            |
| `deactivated`           | No active access                             |
| `deleted`               | No access                                    |

### Requirement Implication

- Backend phải enforce account status.
- UI phải show rõ lý do bị giới hạn.
- Suspended/banned user không được post, comment, message, greeting.

---

## BR-B003 — One person should not create multiple verified accounts with same identity anchor

### Rule

```txt
Một verified identity anchor không được liên kết với nhiều account.
```

### Identity Anchors

```txt
MSSV
Verified HCMUE email
Verified alumni evidence
Verified advisor evidence
```

### Requirement Implication

- MSSV phải unique.
- Duplicate identity phải tạo conflict state.
- Admin cần công cụ xử lý duplicate case.

---

## BR-B004 — User account lifecycle changes must be traceable

### Rule

```txt
Các thay đổi quan trọng của account lifecycle phải có audit log.
```

### Audited Actions

```txt
account.suspended
account.banned
account.reactivated
account.deactivated
account.deleted
verification.approved
verification.rejected
```

### Requirement Implication

- Admin action nghiêm trọng cần reason.
- Audit log phải lưu actor, target, action, previous state, new state, timestamp.

# BR-C. Verification Rules

## BR-C001 — Full product access requires verification

### Rule

```txt
User phải được verified mới được dùng đầy đủ UEConnect.
```

### Unverified Users Cannot

```txt
Create post
Comment
Send greeting
Use discovery fully
Send message
Join community
Submit mentor request
Become Club Manager
Become Mentor
```

### Unverified Users Can

```txt
View account status
Submit verification
Resubmit evidence
View limited help/support
Logout
```

---

## BR-C002 — Verification determines primary system role

### Rule

```txt
Primary system role ban đầu được xác định thông qua verification.
```

### Primary Roles

```txt
Student
Alumni
Advisor
Admin
```

### Requirement Implication

- User không tự chọn role cuối cùng nếu admin review không approve.
- Admin approval phải set role phù hợp.
- Role mismatch có thể dẫn đến reject hoặc need more information.

---

## BR-C003 — Student verification requires MSSV when applicable

### Rule

```txt
Student verification phải thu thập MSSV nếu user là sinh viên hiện tại và có MSSV.
```

### Requirement Implication

- Student verification form cần field MSSV.
- MSSV phải validate format nếu có rule format.
- MSSV phải check uniqueness.

### Exception

Nếu trường hợp đặc biệt chưa có MSSV, hệ thống có thể cho evidence khác nhưng phải cần admin review kỹ hơn.

---

## BR-C004 — MSSV must be unique

### Rule

```txt
Một MSSV chỉ được liên kết với một account verified duy nhất.
```

### Requirement Implication

- Database cần unique constraint hoặc unique filtered index cho MSSV active/verified.
- Nếu MSSV đã tồn tại, user không được submit duplicate như bình thường.
- Admin cần conflict handling.

### Edge Cases

| Case                                   | Handling                              |
| -------------------------------------- | ------------------------------------- |
| User nhập nhầm MSSV                    | Cho sửa trước khi approved            |
| MSSV đã bị account khác dùng           | Chặn và hướng dẫn support/recovery    |
| Admin phát hiện duplicate sau approval | Suspend/review conflict, audit action |
| User cố tạo account mới                | Unique rule chặn duplicate            |

---

## BR-C005 — Verification evidence is required unless strong institutional verification exists

### Rule

```txt
User cần cung cấp evidence xác thực, trừ khi có phương thức xác thực đáng tin hơn như email trường hoặc admin-confirmed identity.
```

### Allowed Evidence Examples

```txt
Thẻ sinh viên
Bảng điểm
Giấy nhập học
Bằng tốt nghiệp
Email hcmue.edu.vn
Giấy tờ hoặc tài liệu số chứng minh liên hệ với HCMUE
```

### Requirement Implication

- Evidence upload cần hỗ trợ nhiều file types hợp lệ.
- UI phải có examples.
- Evidence phải có privacy explanation.

---

## BR-C006 — Verification status must be explicit

### Rule

```txt
Mỗi verification request phải có status rõ ràng.
```

### Statuses

```txt
draft
pending
approved
rejected
need_more_information
suspended
```

### Requirement Implication

- Account status page phải hiển thị status.
- User không được bị kẹt trong trạng thái mơ hồ.
- Need more information phải có instruction.

---

## BR-C007 — Rejection must include reason

### Rule

```txt
Admin reject verification phải nhập reason.
```

### Requirement Implication

- Reject action không được submit nếu không có reason.
- User phải xem được reason phù hợp.
- Reason phải đủ rõ để user biết có thể sửa gì.

---

## BR-C008 — Need more information must explain what is missing

### Rule

```txt
Nếu admin yêu cầu bổ sung thông tin, hệ thống phải lưu và hiển thị rõ user cần bổ sung gì.
```

### Requirement Implication

- Admin action modal cần field note.
- User có resubmit flow.
- Resubmission phải giữ lịch sử request.

---

## BR-C009 — Verification actions must be audited

### Rule

```txt
Mọi admin action trong verification phải có audit log.
```

### Audited Events

```txt
verification.approved
verification.rejected
verification.need_more_information
verification.suspended
verification.resubmitted
```

# BR-D. Role & Permission Rules

## BR-D001 — Each user has one primary system role

### Rule

```txt
Mỗi user chỉ có một primary system role.
```

### Roles

```txt
Student
Alumni
Advisor
Admin
```

### Requirement Implication

- Không dùng nhiều primary roles cho một user.
- Role thể hiện identity chính của user.
- Additional capability phải dùng permission.

---

## BR-D002 — Permission is separate from primary role

### Rule

```txt
Permission là quyền thao tác cụ thể, không phải identity chính.
```

### Examples

```txt
mentor_access
manage_club
moderate_content
review_verification
view_audit_log
manage_permissions
```

### Requirement Implication

- Student có thể có `manage_club`.
- Alumni có thể có `mentor_access`.
- Advisor có thể có `mentor_access`.
- Admin có thể cấp/revoke permissions.

---

## BR-D003 — User cannot grant permission to themselves

### Rule

```txt
User không được tự cấp permission cho chính mình.
```

### Requirement Implication

- Permission grant chỉ do admin có quyền thực hiện.
- Sensitive permission changes cần audit.
- Nếu có request permission flow, admin vẫn phải approve.

---

## BR-D004 — Club Manager is a permission/assignment, not primary role

### Rule

```txt
Club Manager không phải primary role. Đây là permission hoặc assignment do admin cấp.
```

### Requirement Implication

- `club_manager` phải gắn với community cụ thể.
- User có thể là Student + Club Manager.
- Permission cần scope: community_id.

---

## BR-D005 — Mentor is a capability, not necessarily a primary role

### Rule

```txt
Mentor nên được mô hình hóa như permission/capability, không phải primary role riêng trong MVP.
```

### Requirement Implication

- Alumni/Advisor/Senior Student có thể có mentor permission.
- Mentor profile chỉ active khi user có mentor permission.
- Admin có thể revoke mentor permission.

---

## BR-D006 — Permission changes must be audited

### Rule

```txt
Mọi grant/revoke permission quan trọng phải có audit log.
```

### Audited Events

```txt
permission.granted
permission.revoked
club_manager.assigned
club_manager.removed
mentor_access.granted
mentor_access.revoked
moderator_access.granted
moderator_access.revoked
```

---

## BR-D007 — Access control must be enforced server-side

### Rule

```txt
Mọi permission quan trọng phải được enforce ở backend, không chỉ ẩn nút trên UI.
```

### Requirement Implication

- Laravel Policy/Gate/Middleware cần dùng cho action quan trọng.
- UI hide button chỉ là convenience, không phải security.
- Test case phải kiểm tra unauthorized action.

# BR-E. Profile Rules

## BR-E001 — Verified users should complete basic profile before full engagement

### Rule

```txt
Sau khi verified, user nên hoàn thiện profile cơ bản trước khi dùng đầy đủ các tính năng social.
```

### Basic Profile Fields

```txt
Display name
Avatar optional
Faculty
Major
Cohort
Bio short optional
Interests
Learning goals optional
```

### Requirement Implication

- Có profile setup flow.
- Có thể allow skip một số optional fields.
- Discovery quality phụ thuộc profile completion.

---

## BR-E002 — Profile must not expose full MSSV publicly

### Rule

```txt
Profile public không được hiển thị đầy đủ MSSV nếu không có lý do nghiệp vụ rõ.
```

### Requirement Implication

- MSSV chỉ dùng cho verification/admin.
- Nếu hiển thị, chỉ mask một phần hoặc không hiển thị.
- Admin view có thể xem nếu có permission.

---

## BR-E003 — Verified badge only appears for approved users

### Rule

```txt
Verified badge chỉ hiển thị khi user đã approved verification.
```

### Requirement Implication

- UI không hiển thị badge cho pending/rejected user.
- Badge không được imply “an toàn tuyệt đối”.
- Copy nên là `Đã xác thực UEer` hoặc `Verified UEer`.

---

## BR-E004 — Profile visibility must be configurable

### Rule

```txt
User phải có khả năng kiểm soát mức độ hiển thị profile/discovery ở mức cơ bản.
```

### Requirement Implication

- Có privacy settings.
- User có thể tắt discovery visibility nếu cần.
- Search/discovery phải tôn trọng visibility.

---

## BR-E005 — Profile should be context-first, not appearance-first

### Rule

```txt
Profile UEConnect phải ưu tiên context học tập/cộng đồng hơn ngoại hình.
```

### Context Examples

```txt
Khoa
Ngành
Khóa
Interest
Learning goals
Community
Mentor topics
Shared context
```

### Requirement Implication

- Discovery card không được chỉ tập trung avatar/photo.
- Shared context phải có chỗ hiển thị.
- Tránh visual giống dating profile.

# BR-F. Feed / Post / Comment Rules

## BR-F001 — Only verified active users can create posts

### Rule

```txt
Chỉ verified active users mới được tạo post.
```

### Requirement Implication

- Pending/suspended/banned user không được post.
- Backend phải check status.
- UI phải explain nếu user chưa đủ điều kiện.

---

## BR-F002 — Only verified active users can comment

### Rule

```txt
Chỉ verified active users mới được comment/reply.
```

### Requirement Implication

- Comment composer bị disable nếu user không đủ quyền.
- Backend validate trước khi lưu comment.

---

## BR-F003 — User can edit/delete own content

### Rule

```txt
User có thể edit/delete post/comment của chính mình nếu content chưa bị khóa bởi moderation policy.
```

### Requirement Implication

- Owner check required.
- Edited state có thể được lưu.
- Deleted by owner phải khác deleted by moderation.

---

## BR-F004 — Moderator/Admin can hide or delete content based on policy

### Rule

```txt
Admin/moderator có thể hide/delete content nếu vi phạm policy.
```

### Requirement Implication

- Content status cần hỗ trợ moderation state.
- Moderator action cần reason.
- Action cần audit.

---

## BR-F005 — Feed must not become official-only portal

### Rule

```txt
Feed phải là không gian social content-first, không chỉ là bảng thông báo hành chính.
```

### Requirement Implication

- Feed hỗ trợ student posts, questions, community posts, mentor/alumni insights.
- Official announcement nếu có chỉ là một content type, không dominate MVP.

---

## BR-F006 — Report action must be available on user-generated content

### Rule

```txt
Post/comment user-generated phải có report action.
```

### Requirement Implication

- Report menu cần xuất hiện ở post/comment.
- Report tạo item cho moderation queue.

---

## BR-F007 — MVP supports limited comment nesting

### Rule

```txt
MVP chỉ hỗ trợ comment và reply một cấp.
```

### Requirement Implication

- Không làm nested comment vô hạn.
- Database có thể hỗ trợ parent_comment_id nhưng giới hạn depth.

```txt
Chỉ verified active users mới được dùng Discovery đầy đủ.
```

```txt
Discovery card phải ưu tiên hiển thị ngữ cảnh chung giữa users.
```

```txt
Cùng khóa
Cùng interest
Cùng community
Cùng learning goal
Mentor topic liên quan
```

### Requirement Implication

- Suggestion logic cần có explainable reason.
- Card UI cần chỗ cho shared context.

---

## BR-G003 — Main discovery action is Greeting

### Rule

```txt
CTA chính trong Discovery là `Gửi lời chào`.
```

### Forbidden Alternatives

```txt
Match
Swipe
Crush
Hot
Like để match
```

---

```txt
Mỗi greeting phải có state rõ ràng.
```

```txt
not_sent
sent
pending
accepted
declined
cancelled
blocked
```

### Requirement Implication

```txt
Receiver có quyền accept hoặc decline greeting.
```

```txt
Greeting có thể kèm message ngắn nhưng phải bị giới hạn độ dài và kiểm soát spam.
```

### Requirement Implication

- Max length cần được định nghĩa trong requirements.
- Content có thể bị keyword flag/report.
- Rate limit cần cân nhắc.

---

## BR-G007 — Discovery must respect privacy and blocks

### Rule

```txt
Discovery không được hiển thị user đã tắt discovery visibility hoặc user đã block/been blocked theo policy.
```

### Requirement Implication

- Discovery query phải check privacy settings.
- Block relationship phải loại khỏi suggestions.

# BR-H. Connection & Messaging Rules

## BR-H001 — Messaging is permission-based

### Rule

```txt
User không được nhắn tin tự do với bất kỳ ai nếu chưa đủ điều kiện.
```

### Messaging Conditions

```txt

```

## Example

Nếu `Greetings Sent` tăng nhưng `Block Rate` cũng tăng cao:

```txt
Safety thắng growth.
```

Nếu user muốn nhắn tin tự do nhưng messaging permission chưa đủ:

```txt
Access control thắng convenience.
```

Nếu feature mới hay nhưng kéo app thành dating vibe:

```txt
Product scope thắng novelty.
```

Thật sốc khi phải nói: không phải feature nào “nghe vui” cũng đáng làm.

# 7. Business Rules to Database Direction

Business rules imply the following database constraints/directions:

```txt
users has one primary_role
student_id / MSSV uniqueness rule
verification_requests belongs to user
verification_evidences belongs to verification_request
permissions are many-to-many with users
permission_grants are audited
profiles belongs to user
posts/comments have status
greetings have state
connections have state
conversations enforce participants
messages have status
mentor_requests have status
communities have status
community managers are scoped assignments
reports have target_type and target_id
moderation_actions link to report/content/user
audit_logs store actor/action/target/previous/new/reason
blocks prevent direct interaction
notifications have read/unread
```

Database design chưa chốt ở đây, nhưng những rule này sẽ ép database phải nghiêm túc. Database không thích mơ hồ. Nó chỉ giả vờ im lặng rồi nổ ở production.
