# Role Permission Matrix

## 1. Purpose

File này định nghĩa **role permission matrix** cho UEConnect.

Mục tiêu:

- Làm rõ từng loại user được phép làm gì.
- Tách rõ `primary role` và `permission`.
- Tránh thiết kế phân quyền lộn xộn.
- Làm nền cho Laravel Policy, Gate, Middleware, database schema, admin UI và test cases.
- Đảm bảo UEConnect vận hành theo hướng verified-first, permission-based, safety-first và audit-ready.

Trong UEConnect:

```txt
Role = danh tính chính của user trong hệ thống.
Permission = quyền thao tác cụ thể được hệ thống hoặc admin cấp.
Scope = phạm vi áp dụng của permission.
````

Ví dụ:

```txt
Một user có primary role là Student.
User đó có thể được admin cấp permission manage_club cho Community A.
User đó vẫn là Student, nhưng có quyền quản lý Community A.
User đó không được quản lý Community B nếu không có permission tương ứng.
```

Đây là khác biệt nhỏ, nhưng nếu làm sai thì cả backend sẽ biến thành một khu rừng `if else`. Và không ai muốn đi lạc trong đó, trừ bug.

---

## 2. Core Principles

```txt
01. Mỗi user chỉ có một primary system role.
02. Permission tách khỏi primary role.
03. Permission phải được enforce server-side.
04. Permission nhạy cảm phải được audit.
05. Permission có thể có scope.
06. Admin không nên mặc định có mọi quyền nếu không cần.
07. Club Manager không phải primary role.
08. Mentor không phải primary role.
09. Moderator không nhất thiết là primary role.
10. User không được tự cấp quyền cho chính mình.
```

---

## 3. Primary System Roles

UEConnect có 4 primary system roles chính:

| Role      | Description                             | Assigned By                   | Notes                                       |
| --------- | --------------------------------------- | ----------------------------- | ------------------------------------------- |
| `student` | Sinh viên HCMUE hiện tại                | Verification approval         | User chính của sản phẩm                     |
| `alumni`  | Cựu sinh viên HCMUE                     | Verification approval         | Có thể được cấp mentor permission           |
| `advisor` | Cố vấn học tập / người hỗ trợ học thuật | Verification approval / Admin | Có thể được cấp mentor permission           |
| `admin`   | Người vận hành hệ thống                 | System/Admin                  | Không nên dùng bừa, cần permission chi tiết |

## 3.1. Role Rule

```txt
Mỗi user chỉ có đúng một primary role tại một thời điểm.
```

## 3.2. Role Transition Direction

Một số role có thể chuyển theo vòng đời:

```txt
Student → Alumni
```

Role transition phải:

- Có rule rõ.
- Có admin hoặc system approval nếu cần.
- Có audit log.
- Không làm mất lịch sử hoạt động.

## 4. Permission Types

Permission là quyền thao tác cụ thể.

| Permission               | Description                                             | Scope                   |
| ------------------------ | ------------------------------------------------------- | ----------------------- |
| `mentor_access`          | Cho phép user tạo mentor profile và nhận mentor request | Global hoặc topic-based |
| `manage_club`            | Cho phép quản lý community/CLB cụ thể                   | `community_id`          |
| `moderate_community`     | Cho phép moderation trong community cụ thể              | `community_id`          |
| `moderate_content`       | Cho phép moderation content rộng hơn                    | Global hoặc module      |
| `review_verification`    | Cho phép duyệt verification requests                    | Global                  |
| `manage_users`           | Cho phép xem/quản lý user account                       | Global                  |
| `manage_permissions`     | Cho phép grant/revoke permission                        | Global                  |
| `approve_community`      | Cho phép approve/reject community                       | Global                  |
| `view_audit_log`         | Cho phép xem audit logs                                 | Global                  |
| `manage_system_settings` | Cho phép chỉnh setting hệ thống nhạy cảm                | Global                  |
| `view_admin_dashboard`   | Cho phép truy cập admin dashboard                       | Global                  |

## 5. Derived Operational Roles

Các “role” dưới đây không phải primary role. Chúng là cách gọi theo permission/capability.

| Operational Role      | Implemented As                         | Description                         |
| --------------------- | -------------------------------------- | ----------------------------------- |
| Mentor                | `mentor_access` permission             | Người có thể nhận mentor request    |
| Club Manager          | `manage_club` scoped permission        | Người quản lý community/CLB cụ thể  |
| Community Moderator   | `moderate_community` scoped permission | Người kiểm duyệt community cụ thể   |
| Content Moderator     | `moderate_content` permission          | Người xử lý report/content rộng hơn |
| Verification Reviewer | `review_verification` permission       | Người duyệt xác thực                |
| User Manager          | `manage_users` permission              | Người quản lý trạng thái user       |
| Permission Manager    | `manage_permissions` permission        | Người cấp/thu hồi quyền             |
| Audit Viewer          | `view_audit_log` permission            | Người xem audit log                 |
| System Admin          | Multiple high-level permissions        | Người quản trị hệ thống cấp cao     |

## 6. Account Status vs Permission

Permission chỉ có hiệu lực khi account status hợp lệ.

| Account Status          | Can Use Permissions? | Notes                                         |
| ----------------------- | -------------------: | --------------------------------------------- |
| `registered`            |                   No | Chưa verified                                 |
| `pending_verification`  |                   No | Chỉ xem status                                |
| `need_more_information` |                   No | Chỉ resubmit                                  |
| `rejected_verification` |                   No | Bị từ chối                                    |
| `active`                |                  Yes | Có thể dùng theo role/permission              |
| `suspended`             |           Limited/No | Tùy policy, thường không được thao tác social |
| `banned`                |                   No | Không được dùng sản phẩm                      |
| `deactivated`           |                   No | Tài khoản ngưng hoạt động                     |
| `deleted`               |                   No | Không truy cập                                |

Rule:

```txt
Nếu account không active, permission không được dùng cho các action social/admin thông thường.
```

Exception có thể có cho admin/system recovery, nhưng phải cực kỳ rõ và có audit. Vì mở exception mơ hồ là cách hệ thống tự đào hố rồi nhảy vào.

# 10. Scoped Permission Model

## 10.1. Why Scope Matters

Không phải permission nào cũng global.

Ví dụ:

```txt
User A là Club Manager của CLB Tiếng Anh.
User A không được quản lý CLB Công nghệ nếu không được cấp quyền.
```

Nếu không có scope, hệ thống sẽ trao nhầm quyền. Một lỗi nhỏ thôi, kiểu “quản lý nhầm cả trường”.

## 10.2. Scoped Permission Examples

| Permission            | Scope Required | Example                  |
| --------------------- | -------------- | ------------------------ |
| `manage_club`         | Yes            | `community_id = 12`      |
| `moderate_community`  | Yes            | `community_id = 12`      |
| `mentor_access`       | No / Optional  | Global mentor capability |
| `moderate_content`    | Optional       | Global or module         |
| `review_verification` | No             | Global                   |
| `manage_users`        | No             | Global                   |
| `view_audit_log`      | No             | Global                   |
| `approve_community`   | No             | Global                   |

## 10.3. Suggested Permission Grant Fields

# 13. Critical Permission Rules

## PERM-RULE-001 — No Self-grant

```txt
User cannot grant permission to themselves.
```

Exception:

```txt
Only possible through controlled system bootstrap or database seed in development.
Production must avoid self-grant.
```

---

## PERM-RULE-002 — Permission Grant Requires Reason

```txt
Sensitive permission grant/revoke must include reason.
```

Applies to:

```txt
mentor_access
manage_club
moderate_content
review_verification
manage_users
manage_permissions
approve_community
view_audit_log
manage_system_settings
```

---

## PERM-RULE-003 — Permission Changes Must Be Audited

```txt
All sensitive permission grant/revoke actions must create audit log.
```

Audit fields:

```txt
actor_id
target_user_id
permission_key
scope_type
scope_id
action
reason
previous_state
new_state
timestamp
```

---

## PERM-RULE-004 — Permission Revocation Takes Immediate Effect

```txt
When permission is revoked, related capability must stop immediately.
```

Examples:

```txt
Revoked mentor_access → user no longer receives new mentor requests.
Revoked manage_club → user no longer sees management tools.
Revoked review_verification → user cannot open verification queue.
```

---

## PERM-RULE-005 — Account Status Overrides Permission

```txt
Suspended/banned/deactivated users cannot use granted permissions unless explicitly allowed by recovery policy.
```

Example:

```txt
A suspended Club Manager cannot manage community.
A banned Mentor cannot receive mentor requests.
```

---

## PERM-RULE-006 — Scope Must Be Checked

```txt
Scoped permission must check both permission key and scope.
```

Example:

```txt
manage_club for community_id = 1 does not allow managing community_id = 2.
```

---

## PERM-RULE-007 — Admin UI Visibility Is Not Authorization

```txt
Hiding buttons is not enough. Backend must enforce permission.
```

The UI can lie by accident. Backend should be less dramatic.

# 17. Authorization Test Cases

Every permission-sensitive feature should have tests.

## 17.1. Verification Tests

```txt
Unverified user can submit verification.
Student cannot view verification queue.
Admin without review_verification cannot view verification queue.
Verification reviewer can approve verification.
Reject verification without reason fails.
Approve verification creates audit log.
```

## 17.2. Profile Tests

```txt
User can edit own profile.
User cannot edit another profile.
Full MSSV is hidden from public profile.
Admin with permission can view MSSV.
Discovery respects hidden profile visibility.
```

## 17.3. Feed Tests

```txt
Verified user can create post.
Pending user cannot create post.
Suspended user cannot create post.
User can delete own post.
User cannot delete another user post.
Moderator can hide reported post.
Hide post creates audit log.
```

## 17.4. Discovery Tests

```txt
Verified student can view discovery.
Pending user cannot view discovery.
User can send greeting to visible user.
User cannot send duplicate pending greeting.
User cannot send greeting to blocked user.
Greeting accept creates connection.
```

## 17.5. Messaging Tests

```txt
Connected users can message.
Non-connected users cannot message.
Accepted mentor request unlocks conversation.
Blocked user cannot message.
Suspended user cannot message.
Non-participant cannot view conversation.
```

## 17.6. Mentor Tests

```txt
User without mentor_access cannot create mentor profile.
Mentor can set availability.
Unavailable mentor cannot receive new request.
Student can submit mentor request with topic and question.
Request without topic fails.
Accepted mentor request unlocks messaging.
```

## 17.7. Community Tests

```txt
Verified user can view approved community.
Pending community is not public.
Club Manager can edit assigned community.
Club Manager cannot edit other community.
Admin can assign Club Manager.
Assignment creates audit log.
```

## 17.8. Admin Tests

```txt
Normal user cannot access admin dashboard.
Admin without manage_permissions cannot grant permission.
Permission grant requires reason.
Permission grant creates audit log.
Audit viewer requires view_audit_log.
Audit logs cannot be edited from UI.
```

# 18. Permission Matrix Acceptance Criteria

File này được xem là đủ dùng khi:

- [ ] Có định nghĩa primary roles.
- [ ] Có định nghĩa permissions.
- [ ] Có phân biệt role và permission.
- [ ] Có operational roles như Mentor, Club Manager, Moderator.
- [ ] Có account status override.
- [ ] Có high-level matrix.
- [ ] Có domain-specific matrix.
- [ ] Có permission definitions.
- [ ] Có scoped permission model.
- [ ] Có admin permission bundles.
- [ ] Có critical permission rules.
- [ ] Có action audit matrix.
- [ ] Có database direction.
- [ ] Có route protection direction.
- [ ] Có authorization test cases.
- [ ] Bám business rules: one primary role, permission separate, server-side enforcement, audit required.

# 19. Final Role Permission Statement

```txt
UEConnect dùng mô hình một primary role và nhiều permission có thể scoped. Student, Alumni, Advisor và Admin là identity chính. Mentor, Club Manager, Moderator, Verification Reviewer, Permission Manager và Audit Viewer là capability/permission được cấp thêm. Mọi action nhạy cảm phải kiểm tra server-side, có reason nếu cần và được audit.
```

---

# 20. Short Version

```txt
Role trả lời “user là ai”. Permission trả lời “user được làm gì”. Scope trả lời “user được làm ở đâu”. Audit trả lời “ai đã làm gì, khi nào, vì sao”. Thiếu một trong bốn thứ này thì phân quyền sẽ bắt đầu kể chuyện kinh dị.
```

---

# 21. Role Permission North Star

```txt
Mọi quyền trong UEConnect phải bảo vệ mục tiêu: user verified được kết nối và tham gia cộng đồng an toàn, trong khi admin vận hành verification, moderation, permission và audit một cách rõ ràng, có kiểm soát.
```
