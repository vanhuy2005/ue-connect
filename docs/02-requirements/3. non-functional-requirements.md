# Non-functional Requirements

## 1. Purpose

File này mô tả toàn bộ **non-functional requirements** của UEConnect.

Functional requirements trả lời:

```txt
Hệ thống làm gì?
````

Non-functional requirements trả lời:

```txt
Hệ thống phải làm tốt đến mức nào?
An toàn ra sao?
Nhanh ra sao?
Dễ dùng ra sao?
Dễ bảo trì ra sao?
Dữ liệu nhạy cảm được bảo vệ thế nào?
Admin action có truy vết được không?
PWA có đủ giống app thật không?
```

UEConnect không chỉ là một social app cho sinh viên HCMUE. Đây là một verified student social platform có xử lý:

```txt
MSSV
Verification evidence
Profile
Post
Comment
Message
Mentor request
Community
Report
Moderation
Admin action
Audit log
```

Vì vậy, non-functional requirements không phải phần phụ. Đây là phần giúp app không biến thành “demo chạy được trên máy em”, một câu nói từng khiến nhiều server lạnh gáy.

# 4. NFR-SEC — Security Requirements

## NFR-SEC-001 — Authentication Security

### Priority

P0

### Requirement

UEConnect must protect authentication flows including register, login, logout, forgot password and session handling.

### Details

System must:

* Hash passwords securely.
* Never store plain-text passwords.
* Validate login credentials safely.
* Prevent leaking whether an email exists during forgot password flow if possible.
* Invalidate session on logout.
* Protect authenticated routes.
* Redirect users based on account status.

### Acceptance Criteria

* Passwords are hashed using framework best practice.
* Logged-out users cannot access authenticated pages.
* Pending/rejected/suspended/banned users are routed correctly.
* Authentication errors do not expose sensitive internal details.

### Related Functional Requirements

```txt
REQ-AUTH-001
REQ-AUTH-002
REQ-AUTH-003
REQ-AUTH-004
REQ-AUTH-005
```

---

## NFR-SEC-002 — Server-side Authorization

### Priority

P0

### Requirement

All protected actions must be authorized on the server side.

### Details

UI hiding is not security. UEConnect must enforce authorization through backend logic such as Laravel Policy, Gate, Middleware or service-level checks.

Protected actions include:

```txt
Create post
Comment
Send greeting
Accept greeting
Send message
Submit mentor request
Manage community
Review verification
Moderate content
Grant/revoke permission
View audit log
View sensitive evidence
```

### Acceptance Criteria

* Direct HTTP request to protected action fails without permission.
* Unauthorized user receives safe 403/redirect behavior.
* QA can test unauthorized access for each protected domain.
* Admin-only actions cannot be executed by normal users.

### Related Business Rules

```txt
BR-D007
BR-N001
```

---

## NFR-SEC-003 — Role and Permission Enforcement

### Priority

P0

### Requirement

UEConnect must enforce one primary role plus additional permissions.

### Details

Primary roles:

```txt
Student
Alumni
Advisor
Admin
```

Permissions:

```txt
mentor_access
manage_club
moderate_content
review_verification
manage_permissions
view_audit_log
approve_community
manage_users
```

System must not rely only on role names for sensitive actions.

### Acceptance Criteria

* A Student can become Club Manager only through scoped permission.
* An Alumni can become Mentor only through mentor permission.
* Admin actions require specific permissions, not just account existence.
* Permission change is audited.

---

## NFR-SEC-004 — CSRF Protection

### Priority

P0

### Requirement

All state-changing form submissions must be protected against CSRF.

### Details

Applies to:

```txt
Register
Login
Verification submit
Profile update
Post/comment create
Greeting action
Message send
Mentor request
Community join
Report/block
Admin actions
Permission changes
Moderation actions
```

### Acceptance Criteria

* State-changing requests without valid CSRF token are rejected.
* Laravel CSRF protection is enabled for web routes.
* API-style routes use appropriate token/session protection.

---

## NFR-SEC-005 — Input Validation and Sanitization

### Priority

P0

### Requirement

All user inputs must be validated and sanitized.

### Input Areas

```txt
Register fields
Verification fields
Profile fields
Post content
Comment content
Greeting message
Message content
Mentor request
Community description
Report description
Admin notes
Search query
```

### Acceptance Criteria

* Required fields are enforced.
* Length limits exist for text inputs.
* Invalid data returns clear validation errors.
* Stored/displayed content does not execute scripts.
* User-generated HTML is not rendered unsafely.

---

## NFR-SEC-006 — XSS Protection

### Priority

P0

### Requirement

UEConnect must prevent cross-site scripting from user-generated content.

### Details

Risk surfaces:

```txt
Post
Comment
Profile bio
Greeting message
Message
Mentor request
Community description
Admin notes
```

### Acceptance Criteria

* User-generated content is escaped by default.
* Blade templates do not render raw HTML unless explicitly sanitized.
* Rich text, if added later, must use a safe sanitizer.
* Script injection attempts are displayed as text or rejected.

---

## NFR-SEC-007 — SQL Injection Protection

### Priority

P0

### Requirement

All database queries must be protected from SQL injection.

### Details

System should use:

```txt
Laravel Eloquent
Query Builder parameter binding
Prepared statements
Validated search/filter inputs
```

### Acceptance Criteria

* No raw SQL uses untrusted string concatenation.
* Search/filter queries use safe binding.
* Admin search by MSSV/email/name is safely parameterized.

---

## NFR-SEC-008 — File Upload Security

### Priority

P0

### Requirement

All uploaded files must be validated and stored safely.

### Applies To

```txt
Verification evidence
Avatar
Post media
Community media future
Resource files future
```

### Acceptance Criteria

* File type is validated.
* File size is limited.
* File extension alone is not trusted.
* Evidence files are stored privately.
* Executable file upload is rejected.
* Upload error returns clear message.
* File path is not predictable/public for sensitive files.

---

### Priority

P0

### Requirement

Admin pages must require strict authentication and authorization.

### Admin Areas

```txt
Admin dashboard
Verification queue
Moderation queue
User management
Permission management
Community approval
Audit viewer
System settings
```

### Acceptance Criteria

* Normal verified user cannot access admin URLs.
* Admin permission is checked server-side.
* Sensitive admin actions require reason/confirmation.
* Admin action creates audit log if business-critical.

---

## NFR-SEC-010 — Rate Limiting

### Priority

P1

### Requirement

UEConnect should apply rate limiting to high-risk actions.

### High-risk Actions

```txt
Login attempts
Register attempts
Forgot password requests
Verification resubmissions
Greeting sending
Message sending
Report submitting
Search requests
File uploads
```

### Acceptance Criteria

* Excessive attempts are temporarily blocked.
* User sees friendly message.
* Rate limit helps reduce spam and abuse.
* Rate limit does not block normal use cases too aggressively.

---

## NFR-SEC-011 — Secure Error Handling

### Priority

P0

### Requirement

System errors must not expose sensitive implementation details.

### Acceptance Criteria

* Production errors do not show stack traces.
* Database errors do not expose SQL details.
* User-facing errors are clear but safe.
* Internal errors are logged for developers/admins.

---

## NFR-SEC-012 — Secure Configuration

### Priority

P0

### Requirement

Environment secrets and security configuration must be protected.

### Details

Secrets include:

```txt
Database credentials
Mail credentials
Storage keys
Pusher/Reverb credentials if used
App key
OAuth keys if added later
```

### Acceptance Criteria

* Secrets are stored in environment variables.
* Secrets are not committed to Git.
* Production debug mode is disabled.
* `.env` is not exposed publicly.

# 5. NFR-PRIV — Privacy & Sensitive Data Requirements

## NFR-PRIV-001 — Evidence Privacy

### Priority

P0

### Requirement

Verification evidence must be treated as sensitive data.

### Details

Evidence can include:

```txt
Thẻ sinh viên
Bảng điểm
Giấy nhập học
Bằng tốt nghiệp
Email/screenshot evidence
Legal or academic documents
```

### Acceptance Criteria

* Evidence is never public.
* Evidence is viewable only by authorized admin/reviewer.
* Evidence access is protected by permission.
* Evidence should not appear in analytics events.
* Evidence storage path is private.

### Related Rules

```txt
BR-P001
```

---

## NFR-PRIV-002 — MSSV Protection

### Priority

P0

### Requirement

MSSV must be protected as sensitive identity data.

### Acceptance Criteria

* Full MSSV is not shown on public profile.
* Full MSSV is not shown in Discovery.
* Full MSSV is available only to authorized admin contexts.
* MSSV is used for uniqueness and verification.
* Search by MSSV is admin-only.

### Related Rules

```txt
BR-C004
BR-P002
```

---

## NFR-PRIV-003 — Message Privacy

### Priority

P0

### Requirement

Private messages must only be accessible to conversation participants and authorized moderation/admin contexts.

### Acceptance Criteria

* Non-participant cannot view conversation.
* Message body is not included in analytics events.
* Reported message review is permission-protected.
* Message preview in notifications respects privacy direction.

---

## NFR-PRIV-004 — Report Privacy

### Priority

P0

### Requirement

Reports must be handled carefully to protect reporter, target and context.

### Acceptance Criteria

* Report details are visible only to authorized moderator/admin.
* Reporter identity visibility follows policy.
* Reports are not public.
* Report content is not included in broad analytics payload.

---

## NFR-PRIV-005 — Analytics Data Minimization

### Priority

P0

### Requirement

Analytics events must avoid sensitive content.

### Forbidden Analytics Payload

```txt
Raw evidence files
Full MSSV if not strictly required
Private message body
Full report description
Admin private notes
Sensitive document metadata beyond what is needed
```

### Acceptance Criteria

* Events use IDs, context and event names.
* Event payloads are privacy-aware.
* Analytics can calculate KPIs without storing sensitive text.

---

## NFR-PRIV-006 — User-facing Privacy Explanation

### Priority

P0

### Requirement

User must understand why sensitive data is collected.

### Applies To

```txt
Verification evidence upload
MSSV collection
Profile visibility
Report/block
Messaging privacy
```

### Acceptance Criteria

* Verification flow explains evidence usage.
* Privacy note appears before upload.
* Account status page links to help/privacy explanation.
* Copy is clear and non-legalistic.

---

## NFR-PRIV-007 — Data Retention Direction

### Priority

P2

### Requirement

Before real deployment, UEConnect must define data retention policy.

### Data Types

```txt
Verification evidence
MSSV
Messages
Reports
Audit logs
Deleted posts/comments
Admin notes
Analytics events
```

### Acceptance Criteria

* MVP documentation acknowledges retention direction.
* Production deployment must define retention period.
* Deletion/deactivation policy must be documented.

---

## NFR-PRIV-008 — Privacy by Default

### Priority

P1

### Requirement

Sensitive information should be hidden by default.

### Acceptance Criteria

* Profile does not expose sensitive fields by default.
* Discovery respects visibility settings.
* Admin sensitive views are permission-gated.
* User can control discovery visibility.

# 6. NFR-PERF — Performance Requirements

## NFR-PERF-001 — General Page Load Performance

### Priority

P1

### Requirement

Core pages should load fast enough for daily mobile use.

### Core Pages

```txt
Home feed
Discovery
Messages
Mentor
Community
Profile
Notification center
Verification status
Admin queues
```

### Target Direction

For MVP, use reasonable performance budget:

```txt
Initial page response should feel fast on normal mobile network.
Avoid loading all records at once.
Use pagination/lazy loading where needed.
```

### Acceptance Criteria

* Core list pages use pagination or limit.
* Large media is not loaded unnecessarily.
* Loading state appears when data is loading.

---

## NFR-PERF-002 — Feed Performance

### Priority

P0

### Requirement

Home feed must support efficient loading.

### Acceptance Criteria

* Feed loads initial posts with pagination.
* Feed does not load all posts at once.
* Hidden/deleted content is filtered efficiently.
* Media does not block text content loading completely.

---

## NFR-PERF-003 — Discovery Performance

### Priority

P1

### Requirement

Discovery must return profile suggestions efficiently while respecting privacy and blocks.

### Acceptance Criteria

* Discovery excludes blocked/private/unverified users.
* Discovery query supports filters.
* Discovery uses indexed fields where appropriate:

  * faculty.
  * major.
  * cohort.
  * interests.
  * visibility status.

---

## NFR-PERF-004 — Messaging Performance

### Priority

P1

### Requirement

Messaging must feel responsive for basic conversations.

### Acceptance Criteria

* Sending message shows immediate `sending` state.
* Sent/failed state updates clearly.
* Conversation history loads with pagination.
* Future realtime can be added without redesigning data model.

---

## NFR-PERF-005 — Admin Queue Performance

### Priority

P0

### Requirement

Admin queues must be usable even as data grows.

### Queues

```txt
Verification queue
Moderation queue
Community approval queue
User management
Audit viewer
```

### Acceptance Criteria

* Queues support pagination.
* Queues support filter/search.
* Default view does not load all records.
* Search fields are indexed where necessary.

---

## NFR-PERF-006 — File Upload Performance

### Priority

P1

### Requirement

File upload must provide clear progress or state.

### Acceptance Criteria

* User sees upload success/failure.
* Upload failure can be retried.
* Large files are rejected with clear reason.
* Evidence upload does not freeze the UI.

---

## NFR-PERF-007 — Asset Optimization

### Priority

P1

### Requirement

Frontend assets should be optimized through Vite/Tailwind build process.

### Acceptance Criteria

* Unused CSS is minimized through Tailwind build.
* Production assets are bundled/minified.
* Images/icons are optimized where possible.
* Page does not load unnecessary heavy libraries.

# 7. NFR-REL — Reliability & Availability Requirements

## NFR-REL-001 — Graceful Failure

### Priority

P0

### Requirement

UEConnect must handle failures gracefully.

### Applies To

```txt
Login
Verification submission
Evidence upload
Post creation
Comment creation
Greeting
Message send
Mentor request
Community join
Report submission
Admin action
```

### Acceptance Criteria

* User sees clear error state.
* Data is not silently lost when possible.
* Retry is available for recoverable errors.
* Failed state does not leave inconsistent data.

---

## NFR-REL-002 — Transactional Integrity for Critical Actions

### Priority

P0

### Requirement

Critical multi-step actions must be atomic where possible.

### Critical Actions

```txt
Approve verification
Reject verification
Accept greeting and create connection
Accept mentor request and unlock conversation
Suspend/ban user
Grant/revoke permission
Moderation action
Community approval
```

### Acceptance Criteria

* If part of action fails, system rolls back or marks safe failure.
* Audit log is created consistently for completed actions.
* User state and related records do not contradict each other.

---

## NFR-REL-003 — Idempotency for Sensitive Repeated Actions

### Priority

P1

### Requirement

Repeated submissions should not create duplicate critical records.

### Applies To

```txt
Verification submission
Greeting sent
Connection creation
Mentor request
Community join
Report submit
Message send if retried
```

### Acceptance Criteria

* Duplicate pending greeting is not created.
* Duplicate community membership is not created.
* Refresh/retry does not approve verification twice.
* Connection creation after accepted greeting is idempotent.

---

## NFR-REL-004 — Backup Direction

### Priority

P2

### Requirement

Production deployment should include database and file backup strategy.

### Applies To

```txt
Database
Evidence files
Media files
Audit logs
```

### Acceptance Criteria

* Backup strategy is documented before real deployment.
* Restore strategy is considered.
* Sensitive backups must be protected.

---

## NFR-REL-005 — Availability Direction

### Priority

P2

### Requirement

For MVP, UEConnect should be available enough for demo/pilot usage. For real deployment, availability target must be defined.

### Suggested Direction

```txt
MVP/demo: best effort
Pilot: monitored uptime
Production: define SLA/SLO with stakeholder
```

### Acceptance Criteria

* System should not depend on developer local machine.
* Deployment environment should restart app on crash if possible.
* Critical errors are logged.

# 10. NFR-PWA — PWA & Responsive Requirements

## NFR-PWA-001 — PWA-first Direction

### Priority

P1

### Requirement

UEConnect must be designed as a PWA-first web application.

### Acceptance Criteria

* App has responsive layout.
* App can include manifest.
* App has app-like navigation on mobile.
* Offline fallback is planned.
* Native app is not required for MVP.

---

## NFR-PWA-002 — Mobile-first Experience

### Priority

P0

### Requirement

Mobile UI must not be a compressed desktop UI.

### Acceptance Criteria

* Mobile layout uses single-column flow.
* Bottom navigation or app-like nav is available.
* Modals may become bottom sheets if suitable.
* Feed, Discovery, Messages, Mentor and Community are usable on mobile.

---

## NFR-PWA-003 — Desktop Experience

### Priority

P1

### Requirement

Desktop UI should use available space effectively.

### Acceptance Criteria

* Desktop can use sidebar/topbar.
* Feed/profile/messaging can use multi-column layout if useful.
* Admin UI is optimized primarily for desktop.

---

## NFR-PWA-004 — Responsive Breakpoints

### Priority

P1

### Requirement

Design and frontend must define consistent responsive breakpoints.

### Suggested Direction

Using TailwindCSS defaults unless project defines custom:

```txt
sm
md
lg
xl
2xl
```

### Acceptance Criteria

* Layout behavior is documented in page specs.
* Components do not randomly break between common viewports.

---

## NFR-PWA-005 — App Shell Consistency

### Priority

P1

### Requirement

Authenticated app should have consistent shell.

### Acceptance Criteria

* Navigation is consistent across main user pages.
* Admin shell is separated from normal user shell.
* Account status pages use focused layout.
* Loading/empty/error states fit shell.

---

## NFR-PWA-006 — Offline Fallback

### Priority

P2

### Requirement

UEConnect should show a friendly offline fallback when network is unavailable.

### Acceptance Criteria

* User does not see blank broken page.
* Offline page explains connection issue.
