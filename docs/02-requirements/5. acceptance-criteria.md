# Acceptance Criteria

## 1. Purpose

File này định nghĩa **acceptance criteria** cho UEConnect.

Mục tiêu:

- Biến functional requirements thành điều kiện nghiệm thu rõ ràng.
- Làm nền cho manual test, automated test, UAT và QA checklist.
- Giúp Product, Design, Engineering và QA hiểu giống nhau về “xong”.
- Đảm bảo các flow quan trọng có thể test theo pass/fail.
- Bảo vệ business rules: HCMUE-only, verified-first, no dating, permission-based, safety-first, audit-ready.

Acceptance criteria trả lời:

```txt
Requirement này được xem là đạt khi nào?
Điều kiện pass là gì?
Điều kiện fail là gì?
Actor nào thực hiện?
Precondition là gì?
Expected result là gì?
Có cần audit, notification, event, permission check không?
````

Không có acceptance criteria thì requirement chỉ là lời hứa. Và lời hứa trong software thì thường kết thúc bằng “để sprint sau sửa”.

---

## 2. Acceptance Criteria Principles

Tất cả acceptance criteria của UEConnect phải tuân thủ:

```txt
01. Testable
02. Observable
03. Specific
04. Role-aware
05. State-aware
06. Permission-aware
07. Privacy-aware
08. Error-aware
09. Audit-aware when needed
10. Traceable to requirement and business rule
```

## 2.1. Good Acceptance Criteria

Một acceptance criterion tốt phải có thể trả lời bằng:

```txt
Pass / Fail
```

Bad:

```txt
User thấy giao diện đẹp.
```

Good:

```txt
Given a verified user opens Discovery,
When profile suggestions are available,
Then each profile card displays display name, avatar, verified badge, and at least one shared context if available.
```

Đúng, QA không thể test “đẹp”. QA có thể test “có avatar, badge, shared context”. Cảm xúc thì để dành cho retrospective, nơi mọi người giả vờ bình tĩnh.

| Test Type   | Meaning                           |
| ----------- | --------------------------------- |

# 5. Authentication & Account Acceptance Criteria

## AC-AUTH-001 — Guest can register account

### Related Requirements

* REQ-AUTH-001

### Priority

P0

### Actor

Guest

### Preconditions

* Guest is not authenticated.
* Email is not already registered.

### Scenario

Given a guest opens the register page
When the guest enters valid registration information and submits
Then the system creates a new account with status `registered` and redirects the user to verification flow.

### Pass Conditions

* Account is created.
* Account status is `registered`.
* User is authenticated or directed to login depending implementation.
* User cannot access full product before verification.
* Event `user_registered` is recorded if analytics is enabled.

### Fail Conditions

* Account is automatically verified.
* User can access feed/discovery without verification.
* Duplicate email registration succeeds.
* Invalid fields create an account.

### Test Type

Feature / E2E

### Scenario

# 7. Role & Permission Acceptance Criteria

## AC-PERM-001 — User has one primary role

### Related Requirements

* REQ-PERM-001

### Priority

P0

### Actor

System

### Preconditions

* User verification is approved.

### Scenario

Given a user is approved through verification
When the system assigns role
Then the user receives exactly one primary role.

### Pass Conditions

* User has one of: `student`, `alumni`, `advisor`, `admin`.
* User does not have multiple primary roles.
* Additional capabilities are represented by permissions.

### Fail Conditions

* User has both Student and Alumni primary roles.
* Mentor or Club Manager is stored as primary role.

### Test Type

Unit / Database / Feature

---

## AC-PERM-002 — Permission is separate from role

### Related Requirements

* REQ-PERM-002

### Priority

P0

### Actor

System, Admin

### Preconditions

* User has primary role.

### Scenario

Given a Student needs to manage a club
When admin grants `manage_club` scoped permission
Then user remains Student but can manage assigned community.

### Pass Conditions

* Primary role remains `student`.
* Permission grant exists.
* Permission has scope `community_id`.
* User cannot manage other communities.

### Fail Conditions

* User primary role changes to Club Manager.
* User can manage all communities.
* Permission has no scope when scope is required.

### Test Type

Feature / Security

---

## AC-PERM-003 — Permission grant requires audit

### Related Requirements

* REQ-PERM-003
* REQ-AUDIT-001

### Priority

P0

### Actor

Permission Manager

### Preconditions

* Admin has `manage_permissions`.

### Scenario

Given admin grants sensitive permission
When permission is saved
Then system creates audit log with actor, target user, permission key, scope, reason and timestamp.

### Pass Conditions

* Permission is active.
* Audit log is created.
* Reason is stored.
* Permission takes effect immediately.

### Fail Conditions

* Permission is granted without audit.
* Permission is granted without reason.
* User can grant permission to self.

### Test Type

Feature / Security / Audit

---

## AC-PERM-004 — Revoked permission stops capability immediately

### Related Requirements

* REQ-PERM-004

### Priority

P0

### Actor

Permission Manager, Affected User

### Preconditions

* User has an active permission.

### Scenario

Given a user has `manage_club` for Community A
When admin revokes the permission
Then user immediately loses management access for Community A.

### Pass Conditions

* Management UI is hidden or disabled.
* Direct request to manage community fails.
* Audit log is created.
* Existing content remains but new management action is blocked.

### Fail Conditions

* User can still manage community through direct URL.
* Revocation affects unrelated permissions incorrectly.
* Audit log missing.

### Test Type

Security / Feature

---

## AC-PERM-005 — Server-side authorization blocks unauthorized action

### Related Requirements

* REQ-PERM-007

### Priority

P0

### Actor

Unauthorized User

### Preconditions

* User is authenticated but lacks required permission.

### Scenario

Given user lacks `review_verification`
When user sends direct request to approve verification
Then system rejects request with authorization failure.

### Pass Conditions

* Request fails with 403 or safe redirect.
* Verification state does not change.
* No audit action for approval is created.

### Fail Conditions

* Request succeeds because UI button was hidden only.
* Verification is modified by unauthorized user.

### Test Type

Security / Feature

# 26. Acceptance Criteria Review Checklist

Before accepting a requirement, check:

```txt
Does it have clear pass/fail conditions?
Does it define actor?
Does it define preconditions?
Does it define expected result?
Does it cover unauthorized/negative case if needed?
Does it mention permission check if relevant?
Does it mention privacy if sensitive data is involved?
Does it mention audit if admin action is involved?
Does it mention notification/event if required?
Can QA write a test case from it?
Can dev know what to implement from it?
Can product verify it in UAT?
```

Nếu câu trả lời là “chắc được”, thì chưa được. “Chắc” là nơi bug mở chi nhánh.
