# Edge Cases

## 1. Purpose

File này định nghĩa các **edge cases** quan trọng của UEConnect.

Mục tiêu:

- Dự đoán các tình huống bất thường có thể xảy ra trong sản phẩm.
- Làm rõ cách hệ thống nên phản hồi khi flow không đi theo happy path.
- Làm nền cho QA test cases, negative testing, security testing và exception handling.
- Bảo vệ các business rules quan trọng: HCMUE-only, verified-first, MSSV unique, permission-based, safety-first, privacy-first, audit-ready.
- Giúp backend, frontend, QA và product không “ngạc nhiên” khi người dùng làm chuyện người dùng hay làm: bấm sai, reload sai lúc, upload file kỳ quặc, spam, block nhau, report nhau, và nói chung là biến hệ thống thành bài kiểm tra đạo đức lẫn kỹ thuật.

Edge case không phải phần phụ.

Nếu chỉ test happy path, sản phẩm sẽ trông ổn trong demo và bắt đầu rơi từng mảnh khi có người dùng thật. Một truyền thống lâu đời của ngành phần mềm, nhưng ta không nhất thiết phải tham gia.

---

## 2. Edge Case Principles

Tất cả edge cases của UEConnect phải được xử lý theo nguyên tắc:

```txt
01. Fail safely
02. Do not expose sensitive data
03. Do not create duplicate critical records
04. Preserve audit trail for important actions
05. Keep user informed with clear state
06. Enforce permission server-side
07. Account status overrides permission
08. Privacy and safety override convenience
09. Roll back incomplete critical actions
10. Prefer explicit status over silent failure
````

---

## 3. Edge Case Severity

| Severity | Meaning                                                        | Example                                      |
| -------- | -------------------------------------------------------------- | -------------------------------------------- |
| Critical | Có thể gây sai trust, lộ dữ liệu, bypass permission, mất audit | Evidence public, duplicate MSSV approved     |
| High     | Làm hỏng core flow hoặc tạo trạng thái sai                     | Greeting accepted nhưng không tạo connection |
| Medium   | Gây lỗi UX hoặc operation nhưng không phá dữ liệu nghiêm trọng | Empty state sai, notification trễ            |
| Low      | Minor UI/edge polish                                           | Text dài làm vỡ card                         |

# 5. Authentication & Account Edge Cases

## EC-AUTH-001 — User registers with existing email

### Severity

High

### Related Requirements

- REQ-AUTH-001

### Scenario

Guest đăng ký bằng email đã tồn tại trong hệ thống.

### Expected Handling

System rejects registration and shows safe validation error.

### User-facing Behavior

User thấy thông báo email đã được sử dụng hoặc hướng dẫn login/reset password.

### System Behavior

- Không tạo account mới.
- Không thay đổi account cũ.
- Không expose thông tin nhạy cảm ngoài mức cần thiết.

### Test Notes

- Submit duplicate email.
- Verify only one account exists.

---

## EC-AUTH-002 — User logs in while verification is pending

### Severity

High

### Related Requirements

- REQ-AUTH-002
- REQ-AUTH-005
- REQ-VER-008

### Scenario

User đã submit verification, sau đó login lại.

### Expected Handling

System redirects to account status page, not full product.

### User-facing Behavior

User thấy trạng thái `pending_verification`.

### System Behavior

- Feed/discovery/messaging/community/mentor bị khóa.
- User chỉ xem status, logout, help/support.

### Test Notes

Try direct URL access to `/app/feed`, `/app/discovery`, `/app/messages`.

---

## EC-AUTH-003 — User is suspended while currently logged in

### Severity

Critical

### Related Requirements

- REQ-AUTH-006

### Scenario

Admin suspend user trong khi user vẫn đang có session active.

### Expected Handling

System must enforce new account status on next request/action/page reload.

### User-facing Behavior

User bị redirect sang restricted account status page hoặc thấy restriction khi action.

### System Behavior

- Existing session không được bypass status.
- User không thể post/comment/greet/message.
- Audit log đã được tạo từ action suspend.

### Test Notes

- User opens feed.
- Admin suspends user.
- User attempts to comment/send message.
- Expected: blocked.

---

## EC-AUTH-004 — User is banned while currently logged in

### Severity

Critical

### Related Requirements

- REQ-AUTH-007

### Scenario

Admin ban user trong khi user đang dùng app.

### Expected Handling

System blocks all product actions immediately on next request.

### User-facing Behavior

User thấy banned status page.

### System Behavior

- Active session không được tiếp tục dùng app.
- All write actions blocked.
- Direct route access blocked.

### Test Notes

Same as suspension but stricter.

---

## EC-AUTH-005 — Account has active permission but is not active

### Severity

Critical

### Related Requirements

- REQ-PERM-007
- REQ-AUTH-006
- REQ-AUTH-007

### Scenario

User có `manage_club` hoặc `mentor_access`, nhưng account bị suspended/banned.

### Expected Handling

Account status overrides permission.

### User-facing Behavior

User không thấy hoặc không dùng được feature tương ứng.

### System Behavior

- Permission check phải include account status.
- Direct request fails.

### Test Notes

Suspend a Club Manager and try editing community.

---

## EC-AUTH-006 — Deleted/deactivated user appears in old content

### Severity

Medium

### Related Requirements

- REQ-AUTH-008
- REQ-FEED-001
- REQ-PROF-007

### Scenario

User deactivated/deleted account nhưng trước đó có post/comment/message.

### Expected Handling

System displays historical content according to data retention policy.

### User-facing Behavior

Có thể hiển thị:

- `Tài khoản đã ngưng hoạt động`
- Avatar default
- Nội dung bị ẩn nếu policy yêu cầu

### System Behavior

- Không làm orphan dữ liệu.
- Không expose sensitive profile.
- Không xóa audit history.

### Test Notes

Deactivate user with existing posts/comments.

# 6. Verification Edge Cases

## EC-VER-001 — Duplicate MSSV submitted

### Severity

Critical

### Related Requirements

- REQ-VER-006

### Related Business Rules

- BR-C004

### Scenario

User B submit MSSV đã được User A verified.

### Expected Handling

System blocks submission or flags conflict before approval.

### User-facing Behavior

User thấy thông báo MSSV đã tồn tại hoặc cần liên hệ support.

### System Behavior

- Không approve duplicate MSSV.
- Admin thấy conflict warning.
- Duplicate detection event có thể được ghi.

### Test Notes

- Create approved user with MSSV X.
- Submit second verification with MSSV X.
- Try admin approve.

---

## EC-VER-002 — Admin approves verification while MSSV conflict appears after page loaded

### Severity

Critical

### Related Requirements

- REQ-VER-006
- REQ-VER-011

### Scenario

Admin mở request pending, không thấy duplicate. Trong lúc đó request khác được approve cùng MSSV. Admin đầu tiên bấm approve sau.

### Expected Handling

System re-checks MSSV uniqueness at approval time.

### User-facing Behavior

Admin thấy lỗi conflict và không approve được.

### System Behavior

- Approval fails safely.
- No duplicate approved MSSV.
- Transaction/constraint protects data.

### Test Notes

Simulate concurrent approvals.

---

## EC-VER-003 — Evidence file is corrupted or unreadable

### Severity

High

### Related Requirements

- REQ-VER-005
- REQ-VER-013

### Scenario

User upload file hợp lệ về type nhưng admin không đọc được hoặc file bị lỗi.

### Expected Handling

Admin marks `need_more_information`.

### User-facing Behavior

User thấy instruction: evidence bị mờ/hỏng/không đọc được, cần upload lại.

### System Behavior

- Existing request not approved.
- Admin note required.
- Resubmission allowed.

### Test Notes

Upload corrupted PDF/image if supported.

---

## EC-VER-004 — Evidence is valid file type but wrong content

### Severity

High

### Scenario

User upload ảnh không liên quan nhưng file type hợp lệ.

### Expected Handling

System cannot fully detect content automatically. Admin rejects or asks for more information.

### User-facing Behavior

User thấy reason rõ: minh chứng không phù hợp.

### System Behavior

- Verification remains unapproved.
- Admin reason saved.
- Audit log created.

### Test Notes

Upload random image as evidence.

---

## EC-VER-005 — User submits verification then deletes uploaded evidence before admin review

### Severity

High

### Scenario

Evidence record exists but file missing due to deletion/storage issue.

### Expected Handling

Admin cannot approve until evidence is available or user resubmits.

### User-facing Behavior

If user sees status, show need more information after admin action.

### System Behavior

- Evidence viewer shows missing file safe state.
- No raw storage error exposed.
- Admin can request re-upload.

### Test Notes

Manually simulate missing file path.

---

## EC-VER-006 — User tries to resubmit after already approved

### Severity

Medium

### Related Requirements

- REQ-VER-014

### Scenario

Approved user accesses old resubmit URL.

### Expected Handling

System denies or redirects to profile/app home.

### User-facing Behavior

User sees “Tài khoản đã được xác thực”.

### System Behavior

- No new verification request is created unless explicit re-verification flow exists.
- Approved status remains unchanged.

### Test Notes

Use stale resubmit link.

---

## EC-VER-007 — User changes MSSV after submission but before approval

### Severity

High

### Scenario

User in pending state wants to correct MSSV.

### Expected Handling

Only allowed through controlled edit/resubmit flow if policy supports it.

### User-facing Behavior

User can request update only if status allows.

### System Behavior

- Old MSSV change is logged in verification history.
- Duplicate check runs again.
- Admin sees updated data.

### Test Notes

Try editing MSSV in pending and need_more_information states.

---

## EC-VER-008 — User applies as wrong role

### Severity

Medium

### Scenario

User chọn Student nhưng evidence cho thấy Alumni hoặc Advisor.

### Expected Handling

Admin rejects or marks need more information / correction.

### User-facing Behavior

### System Behavior

- No automatic wrong role assignment.
- Admin reason required.
- Resubmit allowed if policy says.

### Test Notes

Submit alumni evidence in student form.

Critical

### Related Requirements

High

### Expected Handling

### Expected Handling

System blocks further interaction.

### User-facing Behavior

- Hidden status enforced server-side.

---

## EC-FEED-005 — User edits post after report submitted

Report post, edit post, review moderation detail.
