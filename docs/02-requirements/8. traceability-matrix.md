# Traceability Matrix

## 1. Purpose

File này định nghĩa **traceability matrix** cho UEConnect.

Mục tiêu:

- Khóa quan hệ giữa business rules, requirements, acceptance criteria, edge cases, test cases, page specs, data entities và metrics.
- Đảm bảo mỗi requirement đều có nguồn gốc rõ ràng.
- Đảm bảo mỗi feature được thiết kế/implement/test đều phục vụ business value.
- Giúp QA biết test case nào kiểm tra requirement nào.
- Giúp designer biết page spec nào cần support requirement nào.
- Giúp engineer biết requirement nào tác động đến database, permission, audit, notification và metrics.
- Giúp project tránh việc có feature “tự nhiên mọc lên”, thứ rất phổ biến trong phần mềm như nấm sau mưa và cũng gây đau đầu tương tự.

Traceability trả lời:

```txt
Business rule này sinh ra requirement nào?
Requirement này được nghiệm thu bằng acceptance criteria nào?
Acceptance criteria này có edge case nào?
Test case nào kiểm chứng nó?
Page nào cần thiết kế?
Entity/database nào bị ảnh hưởng?
Metric/event nào cần tracking?
````

Không có traceability matrix, docs có thể vẫn nhiều. Nhưng nhiều không đồng nghĩa với đúng. Một bài báo cáo 100 trang vẫn có thể lạc đề, chỉ là lạc đề có font chữ đẹp.

## 2. Traceability Model

UEConnect dùng mô hình trace như sau:

```txt
Business Foundation
→ Business Rules
→ Functional / Non-functional Requirements
→ Role / Permission Matrix
→ Acceptance Criteria
→ Edge Cases
→ Test Cases
→ User Flows
→ Page Specs
→ Data / Architecture
→ Metrics / Events
```

## 2.1. Trace Chain

```txt
Problem / Goal
→ Business Rule
→ Requirement
→ Acceptance Criteria
→ Edge Case
→ Test Case
→ Page Spec
→ Entity / Service
→ Metric Event
```

## 4. Traceability Coverage Rules

## 4.1. P0 Requirement Coverage

Every P0 requirement must have:

```txt
At least 1 business rule mapping
At least 1 acceptance criteria
At least 1 test case
Relevant edge case if risky
Relevant page spec if user-facing
Relevant data/entity mapping if persistent
Relevant permission mapping if protected
Relevant audit mapping if admin/critical action
Relevant event mapping if metric-related
```

## 4.2. P1 Requirement Coverage

Every P1 requirement should have:

```txt
Business rule mapping
Acceptance criteria
Test case
Page spec if user-facing
Data/entity mapping if persistent
```

## 4.3. P2/P3 Requirement Coverage

P2/P3 requirements may have lighter trace, but must still include:

```txt
Reason for future/backlog
Related business direction
Out-of-scope/future-ready note if not MVP
```

# 12. Traceability Review Checklist

Trước khi chốt một requirement hoặc page spec, kiểm tra:

```txt
Does this item trace to a business rule?
Does it support a user goal/persona?
Does it have acceptance criteria?
Does it have at least one test case direction?
Does it have edge case coverage if risky?
Does it need permission mapping?
Does it need audit?
Does it touch sensitive data?
Does it need notification?
Does it need analytics event?
Does it need database/entity support?
Does it need a page/component spec?
Does it conflict with out-of-scope?
```

Nếu một feature không trace được về business value, không test được, không có owner, không có rule, nhưng vẫn muốn thêm vào vì “nhìn hay”, thì đó không phải feature. Đó là scope creep đang mặc áo hoodie.

```txt

Requirement ambiguity
Acceptance criteria outdated
```
