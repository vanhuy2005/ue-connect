# Functional Requirements

## 1. Purpose

File này mô tả toàn bộ **functional requirements** của UEConnect.

Mục tiêu:

- Chuyển business rules thành yêu cầu chức năng cụ thể.
- Mô tả hệ thống phải làm gì cho từng domain.
- Làm rõ actor, precondition, expected behavior, edge behavior và acceptance criteria.
- Làm source of truth cho use cases, user flows, page specs, database design, backend services, frontend components và test cases.
- Giữ UEConnect đúng định hướng: HCMUE-only, verified-first, student-first, social nhưng không dating, permission-based, safety-first và enterprise-ready.

Functional requirements trả lời:

```txt
Hệ thống UEConnect phải làm gì?
Ai được làm?
Khi nào được làm?
Nếu không đủ điều kiện thì hệ thống phản hồi thế nào?
Action đó ảnh hưởng dữ liệu gì?
Action đó có cần audit, notification, moderation hoặc metric không?
````

# 4. Authentication & Account Requirements

## REQ-AUTH-001 — Register Account

### Priority

P0

### Actor

Guest

### User Story

As a guest, I want to create an account, so that I can start verification and use UEConnect.

### Description

System must allow a guest to register a new UEConnect account using required credentials.

### Preconditions

- Guest is not logged in.
- Email/username is not already registered.

### Functional Behavior

- System displays registration form.
- User enters required fields.
- System validates input.
- System creates account with initial status `registered`.
- System redirects user to verification entry flow.
- System records `user_registered` event.

### Acceptance Criteria

- User cannot register with an already-used email.
- User cannot register with invalid required fields.
- Newly registered account is not automatically verified.
- Newly registered user cannot access full product before verification.

### Business Rules

- BR-B001
- BR-B002
- BR-C001

### Related Metrics / Events

- `user_registered`
- `register_completed`

---

## REQ-AUTH-002 — Login Account

### Priority

P0

### Actor

Registered User

### User Story

As a registered user, I want to log in, so that I can continue verification or use the product if verified.

### Description

System must allow users to log in using valid credentials.

### Preconditions

- User has an existing account.
- Account is not deleted.

### Functional Behavior

- System validates credentials.
- System checks account status.
- System redirects user based on status:

  - `registered` → verification flow.
  - `pending_verification` → account status page.
  - `need_more_information` → account status/resubmit page.
  - `rejected_verification` → rejection reason/support page.
  - `active` → app home.
  - `suspended` → restricted status page.
  - `banned` → banned status page.

### Acceptance Criteria

- Invalid credentials show safe error message.
- Active verified users reach app home.
- Pending users do not access feed/discovery/messaging.
- Suspended/banned users cannot bypass status restriction.

### Business Rules

- BR-B002
- BR-C001

## REQ-AUTH-004 — Forgot Password

### Priority

P1

### Actor

Registered User

### User Story

As a registered user, I want to reset my password if I forget it.

### Functional Behavior

- User requests password reset.
- System sends reset instruction through configured email channel.
- Reset token expires after a defined period.

### Acceptance Criteria

- New password must pass validation.

## REQ-AUTH-005 — Account Status Routing

### Priority

P0

### Actor

Authenticated User

### Description

System must route users to the correct page based on account and verification status.

### Functional Behavior

- On login, page refresh, or protected route access, system checks:

  - authentication state.
  - account status.
  - verification status.
  - required permission.
- User is redirected to allowed page.

### Acceptance Criteria

- Pending verification user cannot access `/feed`.
- Rejected user can view reason but cannot post.
- Active verified user can access app areas based on permission.
- Banned user cannot access app functions.

### Business Rules

- BR-B002
- BR-C001
- BR-D007

- Suspension action creates audit log.

- BR-O001

# 6. Role & Permission Requirements

## REQ-PERM-001 — Primary Role Assignment

### Priority

P0

### Actor

System, Admin

### Functional Behavior

- System assigns exactly one primary role after verification approval.
- Primary role can be:

  - Student.
  - Alumni.
  - Advisor.
  - Admin.

### Acceptance Criteria

- User cannot have multiple primary roles.
- Role is persisted and used for routing/permissions.
- Role assignment is auditable if admin-driven.

### Business Rules

- BR-D001
- BR-C002

---

## REQ-PERM-002 — Permission Model

### Priority

P0

### Actor

System, Admin

### Functional Behavior

- System supports additional permissions separate from primary role.
- Permissions include:

  - `mentor_access`.
  - `manage_club`.
  - `moderate_content`.
  - `review_verification`.
  - `manage_permissions`.
  - `view_audit_log`.
  - `approve_community`.
  - `manage_users`.

### Acceptance Criteria

- Permission checks are separate from primary role checks.
- Backend enforces permission on protected actions.
- UI visibility follows permission state but does not replace backend enforcement.

### Business Rules

- BR-D002
- BR-D007

---

## REQ-PERM-003 — Grant Permission

### Priority

P0

### Actor

System Admin

### Functional Behavior

- Authorized admin can grant permission to user.
- Admin must select permission type.
- If permission is scoped, admin selects scope:

  - community_id.
  - moderation scope.
- Admin must enter reason for sensitive permission.
- System writes audit log.

### Acceptance Criteria

- User cannot grant permission to self unless explicitly allowed by higher policy.
- Grant action is audited.
- Scoped permission applies only to defined scope.

### Business Rules

- BR-D003
- BR-D006
- BR-O001

---

## REQ-PERM-004 — Revoke Permission

### Priority

P0

### Actor

System Admin

### Functional Behavior

- Authorized admin can revoke permission.
- System disables related capabilities.
- System writes audit log.

### Acceptance Criteria

- Revoked Club Manager loses community management tools.
- Revoked mentor loses active mentor request capability.
- Revocation is audited.

### Business Rules

- BR-D006

---

### Priority

P1

### Actor

Admin

### Functional Behavior

- Admin assigns `manage_club` permission to a user for a specific community.
- Assigned user becomes Club Manager for that community only.

### Acceptance Criteria

- Club Manager cannot manage unrelated community.
- Assignment is visible in community admin tools.
- Assignment is audited.

### Business Rules

- BR-D004
- BR-J003

---

## REQ-PERM-006 — Mentor Permission

### Priority

P1

### Actor

Admin

### Functional Behavior

- Admin grants `mentor_access` permission to eligible users.
- User can create mentor profile after permission is granted.
- User can receive mentor requests only when mentor profile is active and availability allows.

### Acceptance Criteria

- User without mentor permission cannot receive mentor requests.
- Mentor permission can be revoked.
- Grant/revoke is audited.

### Business Rules

- BR-D005
- BR-I001

---

## REQ-PERM-007 — Server-side Authorization

### Priority

P0

### Actor

System

### Functional Behavior

- Every protected action checks permission server-side.
- Unauthorized request returns safe error or redirects.

### Acceptance Criteria

- Hidden UI button is not the only protection.
- Direct API/form submission without permission fails.
- QA can test unauthorized access.

### Business Rules

- BR-D007
