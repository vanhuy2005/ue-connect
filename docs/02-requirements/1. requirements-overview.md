# Requirements Overview

## 1. Purpose

File này định nghĩa tổng quan requirements cho UEConnect.

Mục tiêu:

- Chốt cách viết requirements cho toàn bộ sản phẩm.
- Xác định requirement types, priority, format, ID convention và traceability.
- Làm rõ requirements phải trace ngược về business rules, user journeys, personas, domains và roadmap.
- Làm nền cho functional requirements, non-functional requirements, role-permission matrix, acceptance criteria, edge cases và test cases.
- Giúp team chuyển từ business planning sang implementation planning một cách có kiểm soát.

Từ folder `02-requirements`, UEConnect không còn chỉ mô tả “muốn làm gì” nữa.

UEConnect bắt đầu định nghĩa:

```txt
Hệ thống phải làm gì?
Ai được làm?
Trong điều kiện nào?
Kết quả mong đợi là gì?
Nếu lỗi thì xử lý thế nào?
Làm sao biết requirement đã đạt?
Requirement này trace về business rule nào?
````

Nếu business docs là bản đồ, requirements là hợp đồng. Và hợp đồng mà viết kiểu “làm app cho thân thiện” thì xin chia buồn với QA, họ vừa được tặng một cục sương mù.

## 4. Requirement Priority

UEConnect dùng 4 mức priority.

| Priority | Meaning        | Rule                                         |
| -------- | -------------- | -------------------------------------------- |
| P0       | Must Have      | Bắt buộc cho MVP hoặc trust/safety/core flow |
| P1       | Should Have    | Rất quan trọng, nên có sớm                   |
| P2       | Could Have     | Có giá trị nhưng có thể làm sau              |
| P3       | Future / Later | Không làm trong MVP, chỉ chuẩn bị hướng      |

## 4.1. P0 Definition

Một requirement được xem là P0 nếu nó thuộc ít nhất một nhóm:

```txt
Bắt buộc để user verified
Bắt buộc để user active core flow
Bắt buộc để đảm bảo trust
Bắt buộc để đảm bảo safety
Bắt buộc để admin vận hành verification/moderation
Bắt buộc để tránh dating positioning
Bắt buộc để tránh sai permission/security
Bắt buộc để demo MVP có ý nghĩa
```

## 4.2. P1 Definition

Một requirement được xem là P1 nếu:

```txt
Tăng đáng kể chất lượng UX
Tăng khả năng retention
Tăng hiệu quả admin
Tăng độ rõ của mentor/community
Cần sớm sau MVP core
```

## 4.3. P2 Definition

Một requirement được xem là P2 nếu:

```txt
Có ích nhưng chưa cần cho MVP
Có thể làm sau khi P0/P1 ổn định
Không ảnh hưởng core value nếu thiếu
```

## 4.4. P3 Definition

Một requirement được xem là P3 nếu:

```txt
Là future expansion
Cần architecture awareness nhưng chưa implement
Không đưa vào MVP delivery
```

## 6. Requirement Format Standard

Mỗi functional requirement nên dùng format sau:

```md
## REQ-[DOMAIN]-[NUMBER] — Requirement Name

### Priority
P0 / P1 / P2 / P3

### Status
Draft / Approved / Changed / Deprecated

### Actor
Primary actor

### User Story
As a [actor], I want to [action], so that [benefit].

### Description
Detailed requirement description.

### Preconditions
- Condition 1
- Condition 2

### Trigger
What starts this requirement flow.

### Main Flow
1. Step 1
2. Step 2
3. Step 3

### Alternate / Edge Flows
- Case 1
- Case 2

### Business Rules
- BR-XXX

### Acceptance Criteria
- AC-001
- AC-002

### Data / Entity Impact
- Entity 1
- Entity 2

### Security / Privacy Notes
- Note 1

### Metrics / Events
- event_name

### Related Pages
- Page name

### Related Use Cases
- UC-XXX
```

Không phải requirement nào cũng cần dài như tiểu thuyết Nga, nhưng P0 nên đủ rõ để dev, designer và QA hiểu giống nhau. Đó là tham vọng nhỏ bé nhưng hiếm gặp.

## 7. Requirement Quality Criteria

Một requirement tốt phải đạt các tiêu chí:

```txt
Clear
Testable
Traceable
Scoped
Unambiguous
Role-aware
State-aware
Privacy-aware
Error-aware
Mobile-aware when user-facing
```

## 7.1. Clear

Requirement phải nói rõ hệ thống cần làm gì.

Bad:

```txt
App nên có verification tốt.
```

Good:

```txt
System must allow registered users to submit a verification request with role type, MSSV if applicable, and at least one evidence file.
```

## 7.2. Testable

Requirement phải test được.

Bad:

```txt
Discovery phải thân thiện.
```

Good:

```txt
Discovery card must display at least one shared context when available, such as same faculty, same major, same cohort, shared interest, or shared community.
```

## 7.3. Traceable

Requirement phải trace về business rule hoặc user journey.

Ví dụ:

```txt
REQ-DISC-001 trace về BR-G002, Journey 05, Persona ST-01/ST-02/ST-05.
```

## 7.4. Scoped

Requirement không được mở rộng quá scope.

Bad:

```txt
System must allow students to create marketplace listings.
```

Good:

```txt
System must allow verified users to create posts in the feed.
```

Marketplace không thuộc MVP. Xin đừng đem xe tải scope creep đâm vào docs vừa xây.

## 9. Traceability Model

UEConnect dùng traceability từ business xuống implementation.

```txt
Business Problem
→ Value Proposition
→ Business Rule
→ Requirement
→ Use Case
→ User Flow
→ Page Spec
→ Data Model
→ Test Case
→ Metric/Event
```

## 9.1. Example Trace

```txt
Problem:
Sinh viên khó làm quen bạn mới trong trường.

Business Rule:
BR-G003 — Main discovery action is Greeting.
BR-H001 — Messaging is permission-based.

Requirement:
REQ-DISC-001 — System must allow verified users to send greeting.
REQ-MSG-001 — System must unlock messaging after accepted greeting.

Use Case:
UC-STUDENT-DISC-001 — Send greeting to UEer.

User Flow:
student-flow.md → Peer Discovery & Greeting Flow.

Page Spec:
discovery-page.md
profile-card.md
greeting-modal.md
conversation-page.md

Test Case:
TC-DISC-001 — Verified user can send greeting.
TC-MSG-001 — User cannot message before accepted greeting.

Metric:
greeting_sent
greeting_accepted
conversation_started_after_greeting
```

## 10. Requirement Governance

## 10.1. Requirement Status

| Status     | Meaning                       |
| ---------- | ----------------------------- |
| Draft      | Đang viết, chưa chốt          |
| Review     | Đang chờ review               |
| Approved   | Đã chốt để thiết kế/implement |
| Changed    | Đã thay đổi so với bản trước  |
| Deprecated | Không còn dùng                |
| Future     | Chỉ để định hướng sau MVP     |

## 10.2. Change Rule

Một requirement P0 đã approved chỉ được sửa nếu:

```txt
Business rule thay đổi
Security/privacy issue phát hiện
User journey thay đổi rõ
Technical constraint bắt buộc
Scope được cập nhật chính thức
```

## 10.3. Change Log Requirement

Khi sửa requirement quan trọng, cần ghi:

```txt
What changed
Why changed
Affected business rules
Affected use cases
Affected pages
Affected database/entities
Affected tests
```

## 10.4. Conflict Resolution

Nếu requirement conflict, ưu tiên theo thứ tự:

```txt
01. Safety / Privacy
02. Verification / Trust
03. Role / Permission
04. Product Scope
05. Business Rules
06. User Experience
07. Engagement Growth
08. Convenience
```

Ví dụ: user muốn nhắn tin tự do nhưng business rule yêu cầu messaging permission-based.

Kết luận:

```txt
Business rule thắng. Không mở inbox tự do.
```

Một chiến thắng nhỏ cho văn minh.

# 13. Requirement Dependency Map

## 13.1. Core Dependency Flow

```txt
Authentication
→ Verification
→ Role / Permission
→ Profile
→ Feed / Discovery / Mentor / Community
→ Messaging / Notification
→ Report / Moderation / Audit
```

## 13.2. Critical Dependencies

| Requirement Area     | Depends On                                   |
| -------------------- | -------------------------------------------- |
| Verification         | Authentication                               |
| Profile              | Verification                                 |
| Feed                 | Verified active account                      |
| Discovery            | Verified active account + profile            |
| Greeting             | Discovery + profile                          |
| Messaging            | Connection or accepted mentor request        |
| Mentor Request       | Verified account + mentor profile            |
| Community Management | Community approval + Club Manager permission |
| Moderation           | Report/flag/auto-hide                        |
| Audit                | Admin/moderation/permission actions          |
| Metrics              | Events from all major flows                  |

## 13.3. Dependency Rules

- Không implement full Discovery trước Profile metadata.
- Không mở Messaging trước Connection/Mentor permission.
- Không public Community trước Community approval.
- Không cho Admin action nếu chưa có Audit baseline.
- Không cho Evidence upload nếu chưa có privacy/storage direction.

# 15. Requirement Anti-patterns

Không viết requirements kiểu này:

```txt
REQ-XXX — App should be beautiful.
REQ-XXX — User can interact with everything.
REQ-XXX — Admin can manage all things.
REQ-XXX — System should be fast.
REQ-XXX — Discovery should feel like Tinder but not dating.
REQ-XXX — Add AI recommendation.
REQ-XXX — Make it enterprise.
```

Viết lại thành:

```txt
REQ-DISC-002 — Discovery card must display shared context.
REQ-ADMIN-003 — Admin verification queue must support filter by status, role and submitted date.
REQ-PERF-001 — Feed page should load initial content within defined performance budget.
REQ-SAFE-001 — User must be able to report post, comment, profile, message and community content.
```

“Make it enterprise” không phải requirement. Đó là câu thần chú trong họp, và thần chú thì không pass test case.

# 16. Requirement Review Checklist

Trước khi approve requirement, kiểm tra:

```txt
Does it have an ID?
Does it have priority?
Does it define actor?
Does it define system behavior?
Does it have preconditions?
Does it have acceptance criteria?
Does it map to business rule?
Does it respect product scope?
Does it consider role/permission?
Does it consider error/edge cases?
Does it consider privacy/security if relevant?
Does it consider mobile/PWA if user-facing?
Does it define related metric/event if relevant?
Can QA test it?
Can engineering implement it?
Can design produce page/component specs from it?
```

Nếu câu trả lời là “không biết”, requirement chưa đủ chín. Đem requirement sống đi implement là cách nuôi bug bằng thức ăn hữu cơ.
