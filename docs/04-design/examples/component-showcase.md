# component-showcase.md

## 1. Purpose
Showcase the key component set in one place for review and comparison.

## 2. Design Decision
Present primitives, variants, and states together so the system can be inspected quickly.

## 3. Rationale
A component showcase is useful for QA, handoff, and system consistency checks.

## 4. Rules
- Group by component family.
- Show default and state variants.
- Use the actual tokens from the foundation docs.

## 5. Do / Don't
Do: make visual comparisons easy.

Don't: turn the showcase into a marketing page.

## 6. Tokens / Specs
Include button, input, badge, avatar, card, modal, and navigation samples.

## 7. Component / Screen Impact
Impacts component review, QA, and implementation handoff.

## 8. QA Checklist
- The showcase covers major primitives.
- State coverage is visible.
- Tokens match the canonical docs.

## 9. AI Prompt Notes
Ask AI to render a compact component board with clear labels and states.
