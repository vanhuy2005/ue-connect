# Database Overview

## 1. Purpose

File này định nghĩa tổng quan kiến trúc database cho UEConnect.

Database của UEConnect phải hỗ trợ một verified campus social platform cho cộng đồng HCMUE, bao gồm:

- Authentication.
- Account status.
- Identity verification.
- Profile management.
- Privacy controls.
- Home feed.
- Post/comment/media.
- Discovery.
- Greeting/connection.
- Messaging.
- Notification.
- Mentor matching.
- Career pathway.
- Community/club.
- Resource library.
- Safety reporting.
- Moderation.
- Admin operations.
- Scoped permission.
- Audit log.
- Analytics events.
- PWA/browser push.
- Future AI-assisted evidence review.

Database không chỉ là nơi “lưu dữ liệu”. Nó là nơi giữ business truth của hệ thống.

Nếu database thiết kế sai, frontend có đẹp như studio Hàn Quốc cũng chỉ là cái mặt nạ cho một bãi lầy quan hệ bảng. Một loại bi kịch có foreign key.

# 2. Database Decision

## 2.1. Selected Database

UEConnect sử dụng:

```txt
SQL Server
````

## 2.2. ORM

Backend sử dụng:

```txt
Laravel Eloquent ORM
```

## 2.3. Architecture Style

Database phục vụ kiến trúc:

```txt
Modular Monolith
+ Laravel
+ SQL Server
+ Eloquent
+ Queue
+ Realtime
+ Audit-first admin operations
```

## 2.4. Source of Truth Rule

Database là source of truth cho:

```txt
account status
verification status
profile data
privacy settings
feed content
connection state
conversation/message state
community membership
mentor request state
report/moderation state
notification read state
permission grants
audit logs
analytics events
```

Realtime/WebSocket không phải source of truth.

```txt
Database = truth
Realtime = delivery
Cache = acceleration
Frontend state = temporary view
```

WebSocket không được tự phong làm sự thật. Nó chỉ là đường ống, và đường ống thì thỉnh thoảng rò.

# 4. Core Data Principles

## 4.1. Relational-first

UEConnect có nhiều quan hệ rõ ràng:

```txt
user has profile
user has verification request
user creates posts
post has comments
user sends greetings
accepted greeting creates connection
connection creates conversation
conversation has messages
user joins communities
community has members/posts/resources/chat
student sends mentor request
mentor accepts/declines request
report creates moderation case
admin action creates audit log
```

Vì vậy, relational schema là lựa chọn chính xác.

## 4.2. Explicit State Over Boolean Chaos

Không dùng nhiều boolean để biểu diễn state phức tạp.

Bad:

```txt
is_verified
is_rejected
is_pending
is_suspended
is_active
is_banned
```

Good:

```txt
verification_status = pending_review | approved | rejected | needs_more_information
account_status = active | restricted | suspended | banned
```

Một bảng có 9 boolean state là cách database bắt đầu nói dối.

## 4.3. String Enum for Business State

Dùng string enum cho business state.

Examples:

```txt
account_status
verification_status
profile_status
post_status
comment_status
connection_status
greeting_status
conversation_status
message_status
mentor_request_status
community_status
membership_status
report_status
moderation_case_status
notification_type
```

Lý do:

```txt
readable in DB
easy debugging
audit-friendly
stable across code and docs
clear for admin tools
```

## 4.4. Audit-first for Trust Operations

Mọi hành động nhạy cảm phải có audit.

Audit bắt buộc cho:

```txt
verification approve/reject/need_more_information
admin edit verification data
account suspend/ban/restrict
permission grant/revoke
mentor access approve/reject/grant
community suspend/archive
moderation hide/delete/restore/warn/suspend/ban
report resolve/dismiss/escalate
admin system announcement
```

## 4.5. Privacy by Design

Database phải hỗ trợ privacy ngay từ schema.

Không được chỉ “ẩn ở frontend”.

Privacy cần áp dụng cho:

```txt
profile fields
discovery visibility
message access
blocked users
community membership
private resources
verification evidence
reports
moderation notes
notifications
search results
```

## 4.6. Soft Delete with Intent

Dùng soft delete cho user-generated content và operational records cần khôi phục/truy vết.

Soft delete nên dùng cho:

```txt
posts
comments
messages
communities
community_resources
media_files
mentor_profiles
notifications
```

Không nên soft delete cho mọi thứ một cách máy móc.

Append-only hoặc immutable tốt hơn cho:

```txt
audit_logs
verification_review_actions
moderation_actions
permission_grant_history
analytics_events
```

Audit log mà sửa được như bài nháp thì thôi khỏi gọi là audit, gọi là nhật ký tưởng tượng.

# 6. Global Column Standards

## 6.1. Primary Key

Default:

```txt
id
```

Recommended type:

```txt
bigint identity / bigIncrements
```

UUID/ULID optional for public identifiers.

## 6.2. Public Identifier

For public-facing resources, consider:

```txt
public_id
slug
```

Use for:

```txt
public profile
community
post
resource
conversation if exposed carefully
```

Do not expose raw sequential IDs in sensitive contexts if avoidable.

## 6.3. Status Field

Use:

```txt
status
```

or domain-specific:

```txt
account_status
verification_status
membership_status
message_status
```

## 6.4. Reason Field

For decisions:

```txt
reason
admin_note
user_note
instruction
```

Rules:

```txt
reason = explanation for decision
instruction = what user should do next
admin_note = internal note, not shown to user unless explicitly designed
user_note = user-provided note
```

## 6.5. Metadata Field

Use JSON only when structure is optional or flexible.

```txt
metadata_json
properties_json
settings_json
```

Do not dump core relational data into JSON just to avoid designing schema. That is not flexibility, that is procrastination with braces.

# 8. Account & Verification State Design

## 8.1. Account Status

`users.account_status` allowed values:

```txt
active
restricted
suspended
banned
deleted
```

Meaning:

| Status       | Meaning                       |
| ------------ | ----------------------------- |
| `active`     | User can use allowed features |
| `restricted` | Some features limited         |
| `suspended`  | Temporarily blocked           |
| `banned`     | Permanently blocked           |
| `deleted`    | Account deleted/deactivated   |

## 8.2. Verification Status

`verification_requests.status` allowed values:

```txt
draft
submitted
pending_review
under_review
needs_more_information
approved
rejected
conflict
suspicious
expired
cancelled
```

## 8.3. Profile Completion Status

`profiles.profile_status` allowed values:

```txt
incomplete
ready
restricted
hidden
```

## 8.4. Gate Logic

Login gate uses:

```txt
users.account_status
users.email_verified_at
latest verification request status
profiles.profile_status
role/permission
```

Flow:

```txt
banned/suspended → restricted screen
not verified → verification
verified but profile incomplete → profile setup/onboarding
ready → /app/home
admin route → check permission
```

# 17. Data Consistency Rules

## 17.1. Transactions

Use DB transactions for:

```txt
approve verification + update user/profile + audit + notification event
accept greeting + create connection + create conversation + notification
send message + attachment relation + broadcast event
accept mentor request + create conversation + notification
moderation action + content status update + audit + notification
grant permission + audit
```

## 17.2. After Commit Events

Dispatch side effects after commit:

```txt
notification job
broadcast event
email job
analytics event
```

Do not broadcast success before DB commit. That is how UI celebrates lies.

## 17.3. Unique Constraints

Need unique constraints for:

```txt
users.email
student_profiles.student_code where applicable
active connection pair
pending greeting pair
blocked pair
community membership pair
pending report per reporter + target
saved profile pair
saved pathway pair
mentor topic mapping
```

# 19. Laravel Migration Rules

Migration details are defined in:

```txt
migration-strategy.md
```

But global rules:

```txt
use explicit indexes
use foreign keys where safe
use constrained deletes intentionally
avoid cascade delete on audit-sensitive records
use softDeletes for restorable user content
use timestamps consistently
use nullable only when business-valid
```

Do not make everything nullable “cho dễ seed”. That convenience will return later holding a knife.

# 24. Final Rule

Database của UEConnect phải bảo vệ trust.

Trust trong database nghĩa là:

```txt
state rõ ràng
permission rõ ràng
privacy rõ ràng
audit rõ ràng
file riêng tư rõ ràng
moderation rõ ràng
admin action rõ ràng
relationship giữa người dùng rõ ràng
```

Nếu một business rule quan trọng chỉ tồn tại trong đầu dev hoặc trong một đoạn if ở frontend, rule đó chưa tồn tại thật.

Database không cần thông minh quá mức.
