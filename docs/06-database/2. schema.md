# Database Schema

## 1. Purpose

File này định nghĩa schema tổng thể cho database UEConnect.

Mục tiêu:

- Chia database theo domain module rõ ràng.
- Xác định bảng chính trong từng module.
- Xác định quan hệ chính giữa các bảng.
- Xác định state field quan trọng.
- Xác định bảng nào thuộc MVP, bảng nào dành cho P1/P2.
- Làm nền cho `table-specifications.md`, `erd.md` và migration thật trong Laravel.

File này không mô tả đầy đủ từng column chi tiết. Phần đó thuộc về:

```txt
table-specifications.md
````

File này trả lời:

```txt
Database gồm những nhóm bảng nào?
Bảng nào phục vụ feature nào?
Bảng nào liên kết với bảng nào?
Business state nằm ở đâu?
MVP cần bảng nào trước?
```

Nếu `database-overview.md` là triết lý database, thì `schema.md` là bản đồ thành phố. Còn `table-specifications.md` là địa chỉ từng nhà. Làm đúng thứ tự để không xây đường xuyên qua phòng khách.

# 2. Schema Design Principles

## 2.1. Domain-first Schema

Schema được chia theo domain:

```txt
Identity & Account
Verification
Profile & Privacy
Media
Feed / Post / Comment
Discovery / Greeting / Connection
Messaging
Notification
Mentor
Career Pathway
Community / Club
Safety Reporting
Moderation
Admin / Permission / Audit
Analytics
System Reference Data
AI-assisted Evidence Intelligence
```

## 2.2. Database Is Source of Truth

Database là source of truth cho:

```txt
account status
verification status
profile completion
privacy settings
connection status
message status
community membership
mentor request status
report/moderation status
permission grants
audit logs
notification read state
```

Realtime, cache, browser state, Livewire state không được thay thế database truth.

## 2.3. Explicit State Fields

Mọi workflow quan trọng phải có status enum rõ.

Examples:

```txt
users.account_status
verification_requests.status
profiles.profile_status
posts.status
comments.status
greetings.status
connections.status
conversations.status
messages.status
mentor_requests.status
communities.status
community_members.status
reports.status
moderation_cases.status
```

Không dùng nhiều boolean tùy tiện để mô tả state. Boolean sinh sản nhanh hơn bug, và cả hai đều khó dọn.

## 2.4. Audit-first Trust Operations

Các domain sau phải có bảng action/history/audit riêng:

```txt
verification
moderation
permission grants
account status changes
mentor access
admin operations
```

## 2.5. Privacy-aware Schema

Schema phải hỗ trợ:

```txt
field-level profile privacy
blocked users
private evidence files
private message attachments
community-scoped resources
privacy-safe notification preview
moderation redaction
```

Không được chỉ ẩn bằng CSS hay frontend condition.

# 25. Schema Risks

## 25.1. Risk: Too Many Tables Too Early

Mitigation:

```txt
mark MVP vs P1/P2 clearly
implement domain-by-domain
avoid future tables unless needed
```

## 25.2. Risk: Polymorphic Targets Become Messy

Mitigation:

```txt
use target_type values from controlled enum
document all target types
add indexes target_type + target_id
avoid using polymorphism for core relations where FK is better
```

## 25.3. Risk: Privacy Logic Only in Frontend

Mitigation:

```txt
privacy fields in DB
policy checks in backend
search filters enforce privacy
private file access through controller
```

## 25.4. Risk: Admin Actions Not Audited

Mitigation:

```txt
all admin actions go through action classes
action class writes audit log
no direct update from controller
```

## 25.5. Risk: Realtime State Drift

Mitigation:

```txt
database first
broadcast after commit
client refreshes from DB
message status stored
```
