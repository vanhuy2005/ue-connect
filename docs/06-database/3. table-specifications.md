# Table Specifications

## 1. Purpose

File này mô tả chi tiết các bảng database chính của UEConnect.

Mỗi bảng được mô tả theo:

- Purpose.
- Columns.
- Required / nullable rules.
- Status enum nếu có.
- Indexes.
- Unique constraints.
- Foreign keys.
- Soft delete policy.
- Audit / privacy notes.
- MVP priority.

File này là tài liệu để viết Laravel migrations, Eloquent models, policies, seeders và test cases.

Nếu `schema.md` trả lời “có bảng gì”, thì file này trả lời “bảng đó sống ra sao, có cột gì, cột nào bắt buộc, cột nào bị cấm dùng bừa”. Database mà không có table spec thì chẳng khác gì thuê người xây nhà bằng câu “làm sao cho đẹp”.

# 2. Global Table Rules

## 2.1. ID Strategy

Default primary key:

```txt
id BIGINT IDENTITY PRIMARY KEY
````

Laravel equivalent:

```php
$table->id();
```

Use `public_id`, `slug`, or UUID only when needed for public-facing URLs.

## 2.2. Timestamp Standard

Default timestamps:

```txt
created_at
updated_at
```

Use soft delete only when business needs restoration or moderation trace:

```txt
deleted_at
```

## 2.3. Status Fields

Status fields must use string enum values documented in this file.

Recommended SQL type:

```txt
nvarchar(50)
```

Laravel:

```php
$table->string('status', 50);
```

## 2.4. JSON Fields

Use JSON only for flexible metadata, never for core relational fields.

Recommended name:

```txt
metadata_json
properties_json
settings_json
before_snapshot_json
after_snapshot_json
```

## 2.5. Polymorphic Fields

For generic target references:

```txt
target_type
target_id
context_type
context_id
source_type
source_id
```

Rules:

* `target_type` must be controlled by enum in code.
* Always index `target_type + target_id`.
* Do not use polymorphic fields when a normal FK is better.

## 2.6. Soft Delete Policy

Use soft delete for:

```txt
posts
comments
messages
communities
community_resources
media_files
profiles
mentor_profiles
```

Do not soft delete append-only records:

```txt
audit_logs
analytics_events
moderation_actions
verification_review_actions
permission_grants history
```

# 3. Identity & Account Tables

## 3.1. `users`

### Purpose

Stores core authentication identity and account-level status.

### Priority

```txt
P0
```

### Columns

| Column                     | Type        | Required | Notes                                 |
| -------------------------- | ----------- | -------: | ------------------------------------- |
| `id`                       | bigint      |      Yes | Primary key                           |
| `email`                    | string(255) |      Yes | Unique, must end with `hcmue.edu.vn`  |
| `password`                 | string(255) |      Yes | Laravel hashed password               |
| `remember_token`           | string(100) |       No | Laravel remember me                   |
| `email_verified_at`        | datetime    |       No | Email verification timestamp          |
| `account_status`           | string(50)  |      Yes | Account gate status                   |
| `account_status_reason`    | text        |       No | Human-readable reason for restriction |
| `account_restricted_until` | datetime    |       No | Temporary restriction end             |
| `last_login_at`            | datetime    |       No | Last successful login                 |
| `created_at`               | datetime    |      Yes | Standard timestamp                    |
| `updated_at`               | datetime    |      Yes | Standard timestamp                    |
| `deleted_at`               | datetime    |       No | Soft delete                           |

### Allowed `account_status`

```txt
active
restricted
suspended
banned
deleted
```

### Indexes

```txt
users_email_unique
users_account_status_index
users_deleted_at_index
```

### Unique Constraints

```txt
email unique
```

### Foreign Keys

None.

### Soft Delete

Yes.

### Privacy

High sensitivity.

Never expose:

```txt
password
remember_token
account_status_reason to other users
```

### Business Rules

* Only `hcmue.edu.vn` email is allowed.
* No username required.
* `account_status` controls access after login.
* Suspended/banned user must not access main app.

# 21. Table QA Checklist

Before implementing migrations:

```txt
[ ] Every table has a clear purpose.
[ ] Every workflow has explicit status.
[ ] Required fields are not nullable.
[ ] Nullable fields have business reason.
[ ] Foreign keys are defined.
[ ] Unique constraints are defined.
[ ] Indexes support common queries.
[ ] Soft delete is intentional.
[ ] Audit-sensitive tables are append-only.
[ ] Private data is identified.
[ ] Polymorphic targets are controlled by enum.
[ ] MVP and future tables are separated.
[ ] Naming follows snake_case.
[ ] Status enum values match state machines.
[ ] No sensitive payload in analytics.
[ ] Private file access does not expose raw paths.
```

# 22. Final Rule

A table is approved only when it answers:

```txt
Nó lưu cái gì?
Ai tạo nó?
Ai được xem nó?
Ai được sửa nó?
Nó có state gì?
Nó liên kết với bảng nào?
Nó có cần audit không?
Nó có chứa dữ liệu nhạy cảm không?
Nó có cần soft delete không?
Nó cần index gì?
```

Nếu không trả lời được, đừng migration. Migration dễ chạy, nhưng xóa nợ thiết kế database thì khó hơn nhiều.
