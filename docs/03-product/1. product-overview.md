## 2. Product Summary

UEConnect là một **verified student social platform** dành riêng cho sinh viên, cựu sinh viên, mentor/advisor và cộng đồng nội bộ HCMUE.

UEConnect giúp sinh viên HCMUE:

* Xác thực danh tính bằng mã sinh viên, minh chứng sinh viên hoặc nguồn xác thực phù hợp.
* Tạo profile đáng tin, có context học tập/cộng đồng.
* Đọc, đăng bài và thảo luận trong feed nội bộ.
* Khám phá UEers phù hợp dựa trên khoa, ngành, khóa, interest, mục tiêu học tập và community context.
* Gửi lời chào để bắt đầu kết nối.
* Nhắn tin khi đã có connection hoặc accepted mentor request.
* Tìm mentor, alumni hoặc advisor để được hỗ trợ học tập/career.
* Tham gia community/CLB có governance.
* Report/block khi gặp nội dung hoặc hành vi không phù hợp.
* Hoạt động trong một môi trường có admin moderation, permission, audit và privacy direction rõ ràng.

Core product statement:

```txt
UEConnect giúp sinh viên HCMUE kết nối đúng người, đúng cộng đồng và đúng sự hỗ trợ trong một môi trường social đã xác thực, an toàn và dành riêng cho HCMUE.
```

Short positioning:

```txt
Kết nối và làm quen chuẩn HCMUEr.
```

## 7. Product Pillars

## 7.1. Pillar 1 — Verified Identity

### Purpose

Đảm bảo user trong UEConnect là người thuộc HCMUE hoặc có quan hệ hợp lệ với HCMUE.

### Core Capabilities

```txt
register
login
student verification
alumni verification
advisor verification
upload evidence
manual admin review
MSSV uniqueness
verification status
account status routing
```

### Product Rules

* Full app access requires verification.
* Mỗi MSSV chỉ được gắn với một verified account.
* Evidence là sensitive data.
* Admin review action phải có audit.
* User pending không được vào feed/discovery/messaging.

### Priority

P0.

Đây là trust core. Không có trust core thì UEConnect chỉ là một nhóm Facebook có giao diện tự viết, mà đời đã có đủ nhóm Facebook rồi.

## 7.2. Pillar 2 — Student Profile

### Purpose

Tạo identity đáng tin, có context học tập/cộng đồng, giúp user xuất hiện đúng cách trong feed, discovery, mentor và community.

### Core Capabilities

```txt
profile setup
avatar
display name
faculty
major
cohort
class
bio
interests
learning goals
career interests
profile visibility
verified badge
public profile view
```

### Product Rules

* Full MSSV không xuất hiện public.
* Profile không được dating-first.
* Profile không nên LinkedIn hóa quá mức.
* Profile phải đủ context để user biết “người này liên quan gì đến mình”.

### Priority

P0.

## 7.3. Pillar 3 — Social Feed

### Purpose

Tạo không gian chia sẻ, hỏi đáp, thảo luận và tương tác social nội bộ HCMUE.

### Core Capabilities

```txt
home feed
create post
post detail
comment
reply
like/reaction
save post
report post/comment
moderation state
empty/loading/error states
```

### Product Direction

* Inspired by Threads for blogging/social feed.
* Comment interaction có thể tham khảo Facebook vì rõ và quen.
* Không public viral feed.
* Không biến thành confession ẩn danh.
* Không để career/opportunity spam chiếm feed.

### Priority

P0.

## 7.4. Pillar 4 — Discovery & Greeting

### Purpose

Giúp sinh viên khám phá UEers phù hợp trong trường để làm quen, học cùng, tìm bạn cùng khoa/môn/interest/community.

### Core Capabilities

```txt
discovery profile cards
shared context
filter
save profile
send greeting
accept greeting
decline greeting
connection creation
report profile
block profile
```

### Product Direction

* Interaction có thể nhanh và hấp dẫn như Tinder.
* Nhưng framing phải bỏ hoàn toàn dating vibe.
* CTA chính: `Gửi lời chào`.
* Không dùng `match`, `swipe`, `crush`, `hot`, `dating`.

### Priority

P0.

---

## 7.5. Pillar 5 — Messaging

### Purpose

Cho phép user giao tiếp riêng tư trong phạm vi có trust và permission.

### Core Capabilities

```txt
inbox
conversation
send message
message status
conversation participants
message report
block user
mute conversation
conversation unlocked by accepted greeting
conversation unlocked by accepted mentor request
```

### Product Rules

* Không mở inbox tự do cho toàn bộ users.
* Non-connected users không được nhắn tin trực tiếp.
* Blocked users không thể tiếp tục direct interaction.
* Private message body không đưa vào analytics.
* Non-participant không được xem conversation.

### Priority

P0.

Realtime nâng cao là future-ready, không phải điều kiện MVP. Nhắn tin permission sai nhưng realtime mượt thì vẫn chỉ là spam tốc độ cao.

## 7.6. Pillar 6 — Mentor & Growth

### Purpose

Giúp sinh viên tìm hỗ trợ học tập, career direction, kỹ năng và định hướng từ mentor/advisor/alumni/senior student.

### Core Capabilities

```txt
mentor profile
mentor availability
mentor topics
mentor list
mentor filter
mentor request
request detail
accept request
decline request
need more information
mentor conversation
complete request
report mentor interaction
```

### Product Rules

* Mentor là permission/capability, không phải primary role.
* Alumni/advisor/student có thể được cấp `mentor_access`.
* Mentor không bắt buộc nhận request vô hạn.
* Mentor flow không được LinkedIn hóa.
* Request phải có topic/question/context rõ.

### Priority

P1, nhưng là pillar chiến lược quan trọng.

### Purpose

Giúp sinh viên khám phá, tham gia và tương tác với CLB, lớp, nhóm học tập hoặc cộng đồng có governance.

```txt
community listing

join/follow community
community posts
community manager
community moderation
community report
community approval
```

### Product Direction

* Tham khảo Discord cho community structure.
* Không copy full Discord trong MVP.

### Priority

P1/P2 tùy MVP scope.

## 7.8. Pillar 8 — Safety, Moderation & Admin Governance

### Purpose

Giữ UEConnect an toàn, đáng tin và vận hành được trong môi trường trường học.

### Core Capabilities

```txt
report
block
moderation queue
report detail
hide content
delete content by moderation
warn user
suspend user
ban user
verification queue
user management
permission management
community approval
audit log
```

### Product Rules

* Report cần reason.
* Report là signal, không phải final violation.
* Serious admin action cần reason + confirmation + audit.
* Permission grant/revoke cần audit.
* Admin không mặc định có mọi quyền.
* Evidence, MSSV, message, report details là sensitive data.

### Priority

P0.

Không có safety/admin/audit thì social platform không phải “tối giản”, nó chỉ là thiếu phanh.

## 9. Role Model in Product

## 9.1. Primary System Roles

| Role      | Description                             |
| --------- | --------------------------------------- |
| `student` | Sinh viên HCMUE hiện tại                |
| `alumni`  | Cựu sinh viên HCMUE                     |
| `advisor` | Cố vấn học tập / người hỗ trợ học thuật |
| `admin`   | Người vận hành hệ thống                 |

## 9.2. Operational Capabilities

Các capability dưới đây không phải primary role:

| Capability            | Implemented As                         |
| --------------------- | -------------------------------------- |
| Mentor                | `mentor_access` permission             |
| Club Manager          | `manage_club` scoped permission        |
| Community Moderator   | `moderate_community` scoped permission |
| Content Moderator     | `moderate_content` permission          |
| Verification Reviewer | `review_verification` permission       |
| Permission Manager    | `manage_permissions` permission        |
| Audit Viewer          | `view_audit_log` permission            |

Rule:

```txt
Role trả lời “user là ai”.
Permission trả lời “user được làm gì”.
Scope trả lời “user được làm ở đâu”.
Audit trả lời “ai đã làm gì, khi nào, vì sao”.
```

Một câu ngắn thôi, nhưng nếu không có nó thì backend sẽ thành một lễ hội `if role == ...`.

| ----------------------- | ----------------------------------- |

## 14.2. Frontend Direction

Recommended frontend approach:

```txt
Blade components
TailwindCSS design tokens
Vite asset pipeline
Small JavaScript modules
Progressive enhancement
Mobile-first responsive layout
PWA manifest/service worker
```

Không cần heavy SPA trong MVP.

## 14.3. Backend Direction

Recommended backend approach:

```txt
Laravel modular monolith
Controller + Form Request + Policy + Service/Action layer
Laravel Gate/Policy for authorization
Queue-ready architecture
Private storage for evidence
Application audit log
Analytics event table or event pipeline
```

MS SQL Server should support:

```txt
verified identity constraints
MSSV uniqueness
role/permission grants
scoped permission
audit logs
reports
moderation actions
messages
notifications
analytics events
```

## 14.5. UI Library / Icon Direction

Recommended:

```txt
TailwindCSS
Blade components
Heroicons or Lucide icons
Alpine.js if lightweight interactivity is needed
Laravel Livewire only if server-driven reactive UI is beneficial
```

Avoid:

```txt
heavy UI framework without reason
random icon packs mixed together
gradient-heavy components
overanimated UI
```

## 18. Product Risk Overview

| Risk                             | Product Response                                          |
| -------------------------------- | --------------------------------------------------------- |
| Discovery bị hiểu thành dating   | No dating terms, context-first cards, Greeting CTA        |
| User giả mạo sinh viên           | Verification, evidence, MSSV uniqueness, admin review     |
| Duplicate MSSV                   | Unique constraint + admin conflict warning                |
| User bị spam message             | Permission-based messaging                                |
| Mentor bị quá tải                | Availability, request limits, pause state                 |
| Alumni biến feed thành job board | Opportunity moderation, no recruiter portal               |
| Community thiếu kiểm soát        | Approval, scoped manager, report/moderation               |
| Admin abuse                      | Permission scope + audit log                              |
| Sensitive data leak              | Private storage, restricted views, sanitized analytics    |
| UI nhìn AI-generated             | Neutral UI, restrained brand usage, real product patterns |
| MVP thành super app              | Out-of-scope rules and feature priority                   |
