# Verification Identity

## 1. Feature Summary

### 1.1. One-line Description

Verification Identity cho phép UEConnect xác thực user thuộc cộng đồng HCMUE trước khi mở quyền sử dụng đầy đủ các tính năng social, discovery, messaging, mentor và community.

### 1.2. Feature Goal

Feature này tồn tại để:

- Đảm bảo UEConnect là nền tảng dành riêng cho HCMUE.
- Chặn tài khoản giả mạo, spam, duplicate identity và user ngoài phạm vi trường.
- Xác thực Student, Alumni và Advisor/Mentor ngay từ MVP.
- Bảo vệ rule: mỗi MSSV chỉ gắn với một verified account.
- Cho phép admin duyệt, yêu cầu bổ sung, từ chối, đánh dấu conflict hoặc suspend suspicious account.
- Tạo trust foundation cho feed, discovery, greeting, messaging, mentor, community và moderation.

### 1.3. Product Value

| Value | Description |
|---|---|
| User value | User biết người mình tương tác là UEer thật hoặc người có quan hệ hợp lệ với HCMUE |
| Product value | Tạo khác biệt so với Facebook group/Zalo/Messenger bằng verified identity |
| Business value | Giúp sản phẩm có tính triển khai thật trong môi trường trường học |
| Trust value | Giảm fake account, impersonation, spam, dating abuse và scam |
| Safety value | Admin có cơ sở xử lý identity conflict và account abuse |
| Portfolio value | Thể hiện enterprise planning: verification workflow, evidence privacy, audit, permission, state machine |

### 1.4. Non-goals

Feature này không làm:

- Không thay thế hệ thống quản lý sinh viên chính thức của trường.
- Không đồng bộ trực tiếp với database học vụ trong MVP.
- Không mở public registration cho user ngoài HCMUE.
- Không cho user tự gán role Student/Alumni/Advisor mà không qua verification.
- Không public MSSV, evidence, email, admin note hoặc verification history.
- Không dùng blockchain/NFT/AI màu mè để xác thực danh tính, vì đời đã đủ trò khó hiểu rồi.
- Không tự động approve hoàn toàn chỉ vì user upload file, admin review vẫn là core.

## 2. Scope

### 2.1. In Scope

```txt
- Register bằng email HCMUE hợp lệ
- Google OAuth / institutional email quick login direction
- Student verification
- Alumni verification
- Advisor verification
- Teacher/advisor domain support through configurable institutional domains
- Evidence upload
- Evidence note per file
- External evidence URL support
- MSSV uniqueness check
- Verification request status tracking
- Admin verification queue
- Admin verification detail
- Admin evidence preview
- Approve with reason
- Reject with reason
- Need more information with instruction
- Mark identity conflict
- Suspend suspicious account
- Admin correction before approval with audit
- Resubmission with rate limit
- Verification request expiry
- Notification for verification result
- Audit log for all admin verification actions
- Privacy protection for sensitive verification data
````

### 2.2. Out of Scope

```txt
- Full HCMUE official SSO integration
- Direct student database sync
- Course registration
- Academic record management
- Public evidence URL
- Automatic verification without admin review
- Multi-university verification
- Payment-based verification
- Recruiter/employer verification
```

```txt
- Official HCMUE SSO
- Microsoft/Google institutional OAuth hardening
- Automatic email-domain based pre-verification
- OCR-assisted evidence review

- Data retention automation
- Verification appeal workflow
- Advanced fraud detection
```

### 4.3. Access Rules

```txt
- Guest cannot access verification submit pages.
- Authenticated user without approved verification cannot access /app/*.
- Pending user can only access verification status and correction pages.
- Rejected user can view reason and resubmit if allowed by rate limit.
- Suspended/banned user cannot submit new verification unless admin explicitly reopens.
- Admin route must always check permission server-side.
- Evidence file must be served only through protected controller.
- Account status overrides role and permission.
```

## 8. Email Policy

### 8.1. Registration Email Rule

User can register only with valid HCMUE email.

```txt
Allowed primary domain:
- hcmue.edu.vn
```

### 8.2. Google OAuth / Institutional Email Quick Login

UEConnect should support a fast login/register method using Google OAuth restricted to institutional domain.

Implementation direction:

```txt
- User clicks "Tiếp tục với email HCMUE"
- System redirects to Google OAuth
- OAuth response returns email
- System validates email domain
- If domain is allowed, account can be created or linked
- User still goes through verification flow if role/evidence is not approved
```

### 8.3. Important Security Rule

OAuth email domain validation improves trust but does not fully replace verification in MVP.

```txt
Google OAuth proves access to institutional email.
Verification proves correct role and identity data.
```

Vì nếu chỉ có email mà tự động cho vào hết, hệ thống sẽ nhanh chóng học được một bài học buồn: “có email trường” không đồng nghĩa “đúng role, đúng MSSV, đúng người”.

## 9. Main User Flow

### 9.1. Happy Path: Student Verification

```txt
User registers with hcmue.edu.vn email
→ User logs in
→ System detects user is not verified
→ Redirect to /verification/start
→ User chooses Student
→ User enters student information
→ User uploads evidence files or evidence links
→ User adds note per evidence
→ User reviews submission
→ User submits verification
→ Status becomes pending
→ Admin reviews request
→ Admin approves with reason
→ Account status becomes active
→ Primary role becomes student
→ User receives notification
→ User goes to Welcome screen
→ User continues to Profile Setup
```

### 9.2. Happy Path: Alumni Verification

```txt
User registers with HCMUE email if available
→ User chooses Alumni
→ User enters HCMUE background
→ User enters graduation/cohort information
→ User uploads alumni evidence
→ User submits request
→ Admin reviews
→ Admin approves with reason
→ Primary role becomes alumni
→ User receives notification
→ User goes to role-based onboarding
```

### 9.3. Happy Path: Advisor Verification

```txt
User registers with allowed institutional/advisor email
→ User chooses Advisor
→ User enters advisor identity and department information
→ User uploads or links proof if required
→ User submits request
→ Admin reviews
→ Admin approves with reason
→ Primary role becomes advisor
→ User receives notification
→ User goes to advisor onboarding / mentor setup if mentor enabled
```

### 9.4. Need More Information Flow

```txt
Admin reviews request
→ Evidence or information is insufficient
→ Admin chooses Need More Information
→ Admin enters instruction
→ System updates status = need_more_information
→ User receives notification
→ User opens correction page
→ User updates information/evidence/note
→ User resubmits
→ Status returns to pending
```

### 9.5. Reject Flow

```txt
Admin reviews request
→ Admin chooses Reject
→ Admin enters reason
→ System updates status = rejected
→ User receives notification
→ User sees rejection reason
→ User may resubmit if policy and rate limit allow
```

### 9.6. Conflict Flow

```txt
Admin detects duplicate MSSV / suspicious evidence / identity conflict
→ Admin marks request as conflict
→ System status = conflict
→ Request is highlighted in admin queue
→ Admin may request more info, reject, or suspend suspicious account
→ All actions are audited
```

## 13. MSSV Uniqueness & Conflict Handling

### 13.1. Rule

```txt
A verified MSSV can belong to only one active verified account.
```

### 13.2. Recommended Database Protection

Use a unique filtered constraint or application-enforced unique rule:

```txt
unique approved student_id among verified student/alumni identities
```

Because MSSV may appear in pending/rejected/conflict requests, hard unique on all requests can block legitimate resubmission. The final protection must happen at approval.

### 13.3. Submit-time Behavior

When user submits MSSV that already appears in an approved account:

```txt
- Do not expose the owner identity.
- Show safe message or mark request as conflict.
- Allow support/admin review if policy allows.
```

Recommended user-facing copy:

```txt
Mã sinh viên này cần được kiểm tra thêm. Vui lòng gửi minh chứng rõ ràng để admin xác minh.
```

### 13.4. Approval-time Behavior

Before approval:

```txt
- Recheck MSSV uniqueness inside transaction.
- If duplicate exists, approval fails.
- Request becomes conflict or remains pending with warning.
- Admin sees conflict message.
- Audit logs attempted approval failure if meaningful.
```

### 13.5. Edge Case Solutions

| Edge Case                                         | Handling                                                                                               |
| ------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| Two admins approve same MSSV concurrently         | Use transaction + unique constraint / lock                                                             |
| User enters typo MSSV and gets approved           | Admin correction before approval with audit; after approval requires admin correction workflow         |
| User claims another account used their MSSV       | Mark conflict, suspend suspicious if needed, request stronger evidence                                 |
| Alumni and Student use same MSSV                  | Only one account can own MSSV; if same person transitioning role, update role, do not create duplicate |
| User resubmits multiple times with different MSSV | Keep verification history and audit                                                                    |
| Duplicate pending requests                        | Allow pending but flag; block final approval                                                           |

## 14. Verification Status Model

### 14.1. Verification Request Status

| Status                  | Meaning                                |
| ----------------------- | -------------------------------------- |
| `draft`                 | User started but not submitted         |
| `pending`               | Submitted and waiting for review       |
| `need_more_information` | Admin asks user to correct/add info    |
| `resubmitted`           | User resubmitted after correction      |
| `approved`              | Verified                               |
| `rejected`              | Rejected with reason                   |
| `conflict`              | Duplicate/suspicious identity issue    |
| `expired`               | Request not completed/reviewed in time |
| `cancelled`             | User/admin cancelled if supported      |

### 14.2. Account Status After Verification

| Verification Result  | Account Status                          |
| -------------------- | --------------------------------------- |
| Draft                | `registered`                            |
| Pending              | `pending_verification`                  |
| Need more info       | `pending_verification`                  |
| Rejected             | `registered` or `rejected_verification` |
| Approved             | `active`                                |
| Conflict             | `pending_verification` or `restricted`  |
| Suspicious suspended | `suspended`                             |

### 14.3. State Transitions

```txt
draft
→ pending
→ approved

draft
→ pending
→ rejected
→ resubmitted
→ pending

draft
→ pending
→ need_more_information
→ resubmitted
→ pending

draft
→ pending
→ conflict
→ need_more_information / rejected / suspended_suspicious / approved after resolution

pending
→ expired
```

### 14.4. Invalid Transitions

```txt
approved → pending without admin action
approved → draft
rejected → approved without resubmission or admin review
conflict → approved without reason/audit
expired → approved without reopening
```

## 15. Expiry & Resubmission

### 15.1. Verification Expiry

Recommended policy:

```txt
Pending verification expires after 30 days if no admin/user action.
Draft verification expires after 14 days if never submitted.
Need more information expires after 14 days if user does not respond.
```

### 15.2. Resubmission Rate Limit

User can resubmit multiple times, but must be rate-limited.

Recommended:

```txt
Max 3 resubmissions per 24 hours.
Max 10 resubmissions per verification request lifecycle.
Admin can manually reopen/reset if needed.
```

### 15.3. Why Rate Limit

To prevent:

```txt
- spam submissions
- evidence flooding
- admin queue abuse
- brute-force MSSV attempts
```

Loài người khi thấy form sẽ submit thử 19 lần để “coi sao”. Hệ thống nên thông cảm, nhưng không nên đầu hàng.

## 16. Admin Actions

### 16.1. Supported Actions

| Action                 | Status Result                        | Reason Required | User Notification | Audit Required |
| ---------------------- | ------------------------------------ | --------------: | ----------------: | -------------: |
| Approve                | `approved` + account `active`        |             Yes |               Yes |            Yes |
| Reject                 | `rejected`                           |             Yes |               Yes |            Yes |
| Need more information  | `need_more_information`              |             Yes |               Yes |            Yes |
| Mark conflict          | `conflict`                           |             Yes | Optional/internal |            Yes |
| Suspend suspicious     | account `suspended`                  |             Yes |               Yes |            Yes |
| Correct submitted info | stays current or proceeds            |             Yes |          Optional |            Yes |
| Reopen expired request | `pending` or `need_more_information` |             Yes |               Yes |            Yes |

### 16.2. Approve Rules

Admin can approve only if:

```txt
- Required fields are present.
- Evidence is sufficient.
- MSSV uniqueness is valid.
- Email domain is allowed.
- Request is not expired unless reopened.
- Admin provides approval reason.
- Audit log can be created.
```

Approve result:

```txt
- verification_request.status = approved
- user.account_status = active
- user.primary_role = selected verification role
- profile seed data is prepared
- notification is created
- audit log is created
- user redirected to welcome/onboarding on next login
```

### 16.3. Reject Rules

Reject requires:

```txt
- reason
- optional guideline/category
- audit log
- user notification
```

Reject should not delete evidence.

### 16.4. Need More Information Rules

Need more information requires:

```txt
- clear instruction
- missing/incorrect fields if possible
- audit log
- user notification
```

Example instruction:

```txt
Minh chứng hiện tại chưa thể hiện rõ mã sinh viên. Vui lòng upload thẻ sinh viên hoặc giấy xác nhận sinh viên có MSSV rõ ràng.
```

### 16.5. Admin Correction Before Approval

Admin may correct submitted info before approval, but only with strict controls.

Allowed corrections:

```txt
- Trim typo/spaces
- Normalize faculty/major/cohort
- Correct obvious formatting
- Correct MSSV only if evidence clearly supports it
- Update role only if evidence clearly shows wrong role
```

Rules:

```txt
- Admin must enter reason.
- Previous value and new value must be audited.
- User should be notified if correction affects visible/profile identity.
- Sensitive corrections should not silently happen.
```

## 23. Security & Privacy

### 23.1. Security Rules

```txt
- All verification submit/update routes require auth.
- All admin verification routes require review_verification permission.
- Evidence files are stored in private storage.
- Evidence preview must go through controller authorization.
- Do not expose storage path to frontend.
- Validate MIME and size server-side.
- Validate email domain server-side.
- Never rely only on frontend route guard.
- Approval must happen inside transaction.
- MSSV uniqueness must be checked at approval time.
```

### 23.2. Privacy Rules

```txt
- Full MSSV is admin-only after verification.
- Public profile must not expose full MSSV.
- Evidence file is never public.
- Evidence URL is admin-only.
- Admin notes are admin-only.
- Verification reason sent to user should be safe and clear.
- Analytics must not store evidence URL, file path, full MSSV or admin private notes.
```

| `verification_conflict_marked`       | Admin/system marks conflict | request_id, conflict_type                  | No matched identity |
| `verification_expired`               | Scheduler expires request   | request_id, type                           | No sensitive fields |

### 26.2. Payload Rules

```txt
- Do not store full MSSV.
- Do not store evidence file path.
- Do not store evidence URL.
- Do not store admin private reason.
- Do not store uploaded file name if sensitive.
```

## 27. Edge Cases & Handling

| Edge Case                                    | Expected Handling                           |
| -------------------------------------------- | ------------------------------------------- |
| Duplicate MSSV submitted                     | Flag conflict / block approval safely       |
| Two admins approve same MSSV concurrently    | Transaction + final uniqueness check        |
| Evidence corrupted                           | Admin requests more info                    |
| Evidence wrong content                       | Admin rejects or asks more info             |
| Evidence missing                             | Show evidence unavailable, cannot approve   |
| User applies wrong role                      | Admin corrects with audit or asks more info |
| User changes MSSV before approval            | Store correction history                    |
| User resubmits too many times                | Rate limit and show retry time              |
| User approved then tries to resubmit         | Redirect to app/status                      |
| Admin rejects without reason                 | Server rejects action                       |
| Admin correction without reason              | Server rejects action                       |
| Evidence stored public accidentally          | Security failure, block release             |
| User pending opens `/app/home`               | Redirect verification status                |
| Suspended user tries resubmit                | Block unless admin reopens                  |
| Verification expires while admin detail open | Admin sees stale/expired state              |
| Admin action while request already reviewed  | Reject stale action, ask refresh            |
| OAuth email domain mismatch                  | Block registration/login                    |
| External evidence URL inaccessible           | Admin requests more info                    |
