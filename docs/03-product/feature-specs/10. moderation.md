# Moderation

## 1. Feature Summary

### 1.1. One-line Description

Moderation cho phép admin/moderator xem xét report, keyword flag, auto-hidden content và thực hiện hành động xử lý nội dung hoặc tài khoản theo quy trình có reason, audit, notification và permission rõ ràng.

### 1.2. Feature Goal

Feature này tồn tại để:

- Cho moderator/admin xử lý report từ user.
- Cho hệ thống xử lý content bị auto-hide do nhiều report.
- Cho hệ thống đưa keyword-flagged content vào review queue.
- Hỗ trợ xử lý nhiều target type: profile, post, comment, message, community, community resource, mentor request.
- Hỗ trợ moderator action: dismiss, hide, delete, restore, warn, suspend, ban.
- Bắt buộc action reason cho mọi moderator/admin decision.
- Ghi audit log cho toàn bộ moderation action.
- Bảo vệ reporter identity.
- Bảo vệ target user khỏi việc bị lộ report detail không cần thiết.
- Cho content bị hidden hiển thị placeholder phù hợp.
- Chuẩn bị appeal/kháng nghị để user có thể yêu cầu xem xét lại.
- Tạo nền governance đủ nghiêm túc cho môi trường HCMUE.

### 1.3. Product Value

| Value | Description |
|---|---|
| User value | User thấy nền tảng có kiểm soát, report được xử lý, nội dung độc hại không bị bỏ mặc |
| Product value | UEConnect có safety governance thay vì chỉ có nút report cho đẹp |
| Business value | Sản phẩm đủ điều kiện trình bày như một platform có thể triển khai thật trong trường |
| Trust value | Quyết định moderation có reason, audit, permission, không xử lý tùy hứng |
| Safety value | Nội dung rủi ro được hide/delete/warn/suspend/ban đúng luồng |
| UX value | User thấy trạng thái nội dung rõ ràng: unavailable, hidden, restored, restricted |

### 1.4. Non-goals

Feature này không làm:

- Không tự động ban user chỉ vì một report đơn lẻ.
- Không public danh tính reporter.
- Không public report detail.
- Không cho moderator xử lý không reason.
- Không cho moderator tự ý vượt quyền permission.
- Không cho admin xóa audit log để che hành động.
- Không dùng AI/keyword flag làm quyết định cuối cùng trong MVP.
- Không biến moderation thành nơi “xử án công khai”.
- Không cho community owner tự xử lý report liên quan chính họ.
- Không thay thế policy/rule document của toàn bộ platform.

### 4.3. Permission Rules

```txt
- All moderation routes require active account.
- All moderation routes require admin/moderator permission.
- Scoped permission must be enforced for community moderation.
- Club Manager cannot moderate reports where they are the reported target.
- Community Owner cannot dismiss report against their own community without escalation.
- Sensitive message/evidence report requires stronger permission.
- Moderator action requires reason.
- Account-level action requires stronger permission than content-level action.
- UI hiding is not authorization.
```

## 9. Moderation Case Grouping

### 9.1. Why Case Grouping

Multiple reports can point to the same content or user. Grouping prevents moderators from reviewing the same drama five times like a cursed slideshow.

### 9.2. Case Grouping Rule

Create or update a moderation case by:

```txt
target_type
target_id
active unresolved status
```

### 9.3. Case Status

| Status         | Meaning                          |
| -------------- | -------------------------------- |
| `open`         | Case created                     |
| `queued`       | Waiting for review               |
| `under_review` | Moderator is reviewing           |
| `action_taken` | Action applied                   |
| `dismissed`    | No action needed                 |
| `escalated`    | Higher admin needed              |
| `resolved`     | Closed                           |
| `reopened`     | Reopened after new report/appeal |

### 9.4. Case Priority

Priority is calculated from:

```txt
- highest report severity
- number of distinct reporters
- target type
- keyword flags
- target owner violation history
- content reach
- auto-hide state
```

### 9.5. Case Assignment

P1/P2:

```txt
- Admin can assign case to moderator.
- Moderator can claim case if allowed.
- Case cannot be edited by multiple moderators at same time without stale state warning.
```

## 10. Moderation Actions

### 10.1. Supported Actions

| Action           | Target                             | Effect                        | Reason Required | Audit Required |
| ---------------- | ---------------------------------- | ----------------------------- | --------------: | -------------: |
| `dismiss`        | report/case                        | Close with no violation       |             Yes |            Yes |
| `hide`           | content/profile/community/resource | Hide from normal users        |             Yes |            Yes |
| `delete`         | content/resource                   | Remove from normal users      |             Yes |            Yes |
| `restore`        | hidden/deleted content             | Make visible again if allowed |             Yes |            Yes |
| `warn`           | user                               | Send warning                  |             Yes |            Yes |
| `suspend`        | user                               | Temporarily restrict account  |             Yes |            Yes |
| `ban`            | user                               | Strong account restriction    |             Yes |            Yes |
| `escalate`       | case                               | Send to higher admin          |             Yes |            Yes |
| `mark_duplicate` | report                             | Merge/close duplicate         |             Yes |            Yes |
| `lock_content`   | post/comment/community             | Prevent further interaction   |             Yes |            Yes |

### 10.2. Dismiss

Use when:

```txt
- no violation found
- report lacks evidence
- target content already resolved
- false report
```

Effect:

```txt
- report/case status = dismissed or resolved
- target content remains active unless previously hidden by another action
- reporter may receive safe update
- target usually not notified
```

### 10.3. Hide

Use when:

```txt
- content may violate policy
- harm should be reduced while retaining review ability
- auto-hide was triggered and moderator confirms temporary hiding
```

Effect:

```txt
- target moderation_status = hidden_by_moderation
- normal users see placeholder or content unavailable
- owner receives safe notification if policy says yes
- audit log created
```

### 10.4. Delete / Remove

Use when:

```txt
- content clearly violates policy
- content contains personal info leak
- scam/fraud
- sexual/dating content
- serious harassment
- repeated copyright violation
```

Effect:

```txt
- target status = removed_by_moderation
- content unavailable to normal users
- admin/moderator can still view audit/snapshot if permitted
- owner receives safe notification
```

### 10.5. Restore

Use when:

```txt
- report was wrong
- keyword flag false positive
- auto-hide false positive
- moderator/admin reverses decision
```

Effect:

```txt
- target status returns to active if parent/account allows
- report/case updated
- audit log created
- owner may receive safe notification
```

### 10.6. Warn User

Use when:

```txt
- minor or first-time violation
- behavior needs correction
- content removed but account restriction not yet needed
```

Effect:

```txt
- warning record created
- user receives warning notification
- warning_count increments
- future moderation priority may increase
```

### 10.7. Suspend User

Use when:

```txt
- repeated violation
- harassment
- impersonation
- scam
- serious safety issue
- investigation needed
```

Effect:

```txt
- user.account_status = suspended
- user cannot create/interact/message/greet
- profile/feed/discovery visibility restricted
- active sessions should be restricted on next request
- notification sent with safe reason
- audit log created
```

### 10.8. Ban User

Use when:

```txt
- severe violation
- repeated abuse after warnings/suspensions
- scam/fraud
- serious impersonation
- severe personal info leak
```

Effect:

```txt
- user.account_status = banned
- user cannot access app
- user removed from Discovery/feed interactions
- existing content may be hidden/retained depending policy
- audit log created
```

### 10.9. Escalate

Use when:

```txt
- target is admin/moderator/community owner
- high-risk political/sensitive issue
- legal/copyright uncertainty
- evidence-related abuse
- conflict of interest
```

Effect:

```txt
- case status = escalated
- assigned_to higher admin or null
- normal moderator cannot close without permission
```

## 11. Content Visibility After Moderation

### 11.1. Placeholder Rules

| Target         | Hidden State UI                      |
| -------------- | ------------------------------------ |
| Post           | `Nội dung này hiện không khả dụng.`  |
| Comment        | `Bình luận này hiện không khả dụng.` |
| Message        | `Tin nhắn này hiện không khả dụng.`  |
| Profile        | Limited unavailable/restricted state |
| Community      | Community locked/suspended page      |
| Resource       | Resource unavailable                 |
| Mentor request | Request restricted/under review      |

### 11.2. Placeholder Principles

```txt
- Do not expose original content.
- Do not expose report reason to other users.
- Do not expose reporter identity.
- Keep thread layout stable where necessary.
- For comments with replies, placeholder may remain.
- For content without dependents, item may disappear.
```

### 11.3. Content Removal vs Hide

| Action                | Normal User Sees                        | Admin/Moderator Sees         |
| --------------------- | --------------------------------------- | ---------------------------- |
| Hide                  | Placeholder/unavailable                 | Content + reason + audit     |
| Delete/remove         | Unavailable or absent                   | Snapshot/audit if permitted  |
| Soft delete by author | Unavailable/absent                      | Owner/admin depending policy |
| Auto-hide             | Placeholder + pending review internally | Auto-hide badge in queue     |

## 12. Account-level Moderation

### 12.1. Account Actions

| Action            | Effect                                 |
| ----------------- | -------------------------------------- |
| `warn`            | User can continue but receives warning |
| `suspend`         | Temporary restricted access            |
| `ban`             | Strong restriction/no access           |
| `reactivate`      | Restore suspended user if allowed      |
| `mark_suspicious` | Internal flag for review               |

### 12.2. Account Status Mapping

| Moderation Action | User Account Status                            |
| ----------------- | ---------------------------------------------- |
| Warn              | unchanged                                      |
| Suspend           | `suspended`                                    |
| Ban               | `banned`                                       |
| Reactivate        | `active` if verified/profile complete          |
| Mark suspicious   | `restricted` or internal flag depending policy |

### 12.3. Effects of Suspension

```txt
- Cannot create post.
- Cannot comment.
- Cannot react/save.
- Cannot send greeting.
- Cannot send message.
- Hidden from Discovery.
- Profile may show restricted/unavailable.
- Existing sessions restricted on next request.
```

### 12.4. Effects of Ban

```txt
- Cannot access app.
- Cannot appear in Discovery.
- Cannot create/interact.
- Conversations restricted.
- Content may be hidden or retained for audit depending policy.
```

### 13.1. MVP Decision

Appeal can be P1/P2, but data model and notifications should be appeal-ready.

### 13.2. Appeal Meaning

User can request review of:

```txt
- hidden content
- removed content
- warning
- suspension
- ban
```

### 13.3. Appeal Status

| Status          | Meaning                              |
| --------------- | ------------------------------------ |
| `not_available` | Appeal not supported for this action |
| `eligible`      | User can appeal                      |
| `submitted`     | Appeal submitted                     |
| `under_review`  | Admin reviewing                      |
| `accepted`      | Action reversed/modified             |
| `rejected`      | Action upheld                        |

### 13.4. Appeal Rules Future

```txt
- Appeal has deadline.
- Appeal requires reason.
- Appeal reviewer should not be same moderator if possible.
- Appeal decision requires reason and audit.
- Appeal cannot expose reporter identity.
```

## 14. Main Moderator Flow

### 14.1. Review Report Flow

```txt
Moderator opens /admin/moderation/reports
→ Filters by priority/status
→ Opens report/case detail
→ Reviews target snapshot, report reason, severity, history
→ Chooses action
→ Enters reason
→ Confirms action
→ System applies action transactionally
→ Audit log created
→ Notification sent if needed
→ Case/report status updated
```

### 14.2. Auto-hidden Review Flow

```txt
System auto-hides content after threshold
→ Case appears in Auto-hidden Queue
→ Moderator opens case
→ Reviews reports and target snapshot
→ Chooses restore / keep hidden / delete / warn / suspend
→ Reason required
→ Audit created
```

### 14.3. Keyword Flag Review Flow

```txt
System flags content by keyword
→ Keyword flag appears in queue
→ Moderator reviews content
→ Dismiss flag or take moderation action
→ If action taken, reason required
→ Audit created
```

### 14.4. Account Sanction Flow

```txt
Moderator/admin reviews repeated/severe violation
→ Opens user moderation history
→ Chooses warn / suspend / ban
→ Enters reason and optional duration
→ System updates account status
→ Active sessions restricted
→ User notified with safe copy
→ Audit created
```

## 15. Detailed Flow

|    1 | Moderator opens queue     | Load reports/cases                   | loading          |

|    3 | Moderator opens detail    | Load target snapshot/context         | reviewing        |
|    4 | Moderator selects action  | Show action form                     | action_selected  |
|    5 | Moderator enters reason   | Validate reason                      | ready_to_submit  |
|    6 | Moderator confirms        | Apply transaction                    | processing       |
|    7 | System updates target     | Hide/delete/restore/warn/suspend/ban | action_applied   |
|    8 | System creates audit      | Audit saved                          | audited          |
|    9 | System sends notification | Safe notification queued             | notified         |
|   10 | Queue updates             | Case status updated                  | resolved/updated |

## 16. Failure & Alternative Flows

| Flow                                       | Condition                                           | Expected Behavior |
| ------------------------------------------ | --------------------------------------------------- | ----------------- |
| Moderator opens already resolved case      | Show stale state and latest action                  |                   |
| Two moderators act same case               | Only first succeeds; second sees stale warning      |                   |
| Action submitted without reason            | Server rejects                                      |                   |
| Moderator lacks permission                 | 403                                                 |                   |
| Audit creation fails                       | Rollback moderation action                          |                   |
| Notification fails                         | Action succeeds; notification retry                 |                   |
| Target content deleted by author           | Moderator sees snapshot and can close/escalate      |                   |
| Target account already banned              | Show current status, prevent duplicate action       |                   |
| Restore content but parent deleted         | Reject restore or restore parent chain if permitted |                   |
| Restore content but author banned          | Reject or keep hidden due to account status         |                   |
| Community suspended                        | Community content remains unavailable               |                   |
| Reporter is target owner                   | Reject/self-report invalid                          |                   |
| Moderator has conflict of interest         | Escalate                                            |                   |
| Sensitive target needs stronger permission | Hide content detail and show permission required    |                   |

## 23. Security & Privacy

### 23.1. Security Rules

```txt
- Every moderation route requires permission.
- Every moderation action requires server-side authorization.
- Every moderation action requires reason.
- Every moderation action must create audit log.
- Reporter identity is hidden unless permission allows.
- Sensitive report targets require stronger permission.
- Moderator cannot act on case with conflict of interest.
- Account-level actions require stronger permission than content-level actions.
- Hidden/deleted content must not leak through normal APIs.
- Restore must recheck parent visibility/account status.
```

### 23.2. Privacy Rules

```txt
- Reporter identity is not visible to target.
- Raw report description is not visible to target.
- Internal moderator notes are not visible to normal users.
- Message report content is shown only with limited context.
- Evidence files are shown only to authorized verification reviewers/admin.
- Analytics must not store raw content or report description.
- Notifications must use safe copy.
```

### 23.3. Sensitive Data Matrix

| Data                      | Sensitive? | Handling                         |
| ------------------------- | ---------: | -------------------------------- |
| Reporter identity         |        Yes | Restricted permission            |
| Report description        |        Yes | Moderator/admin only             |
| Reported message body     |        Yes | Limited context                  |
| Verification evidence     |        Yes | Verification reviewer/admin only |
| Moderator internal reason |        Yes | Audit/admin only                 |
| User-safe reason          |   Moderate | Can be sent to target            |
| Audit log                 |  Sensitive | Admin/audit viewer only          |
| Keyword matched phrase    |  Sensitive | Avoid storing raw phrase         |

## 27. Edge Cases

| Edge Case                                                | Expected Handling                         |
| -------------------------------------------------------- | ----------------------------------------- |
| Moderator acts without reason                            | Server rejects                            |
| Moderator lacks permission                               | 403                                       |
| Two moderators act same case                             | First wins, second sees stale state       |
| Target already deleted                                   | Show current status, allow close/escalate |
| Target already hidden                                    | Prevent duplicate hide or update case     |
| Restore hidden post but author banned                    | Reject or keep unavailable                |
| Restore comment but parent post removed                  | Reject or require parent restore          |
| Community owner reports moderator action                 | Escalate to admin                         |
| Report against admin/moderator                           | Escalate to higher admin                  |
| Reporter identity requested by normal moderator          | Hide unless permission                    |
| Message report accessed without permission               | Hide message body                         |
| Evidence report accessed without verification permission | Hide evidence                             |
| Notification fails                                       | Action succeeds, notification retries     |
| Audit fails                                              | Rollback action                           |
| Suspended user tries to interact                         | Block by account guard                    |
| Banned user content still appears in feed                | Bug; exclude by author account status     |
| Auto-hidden content receives restore                     | Remove auto-hidden flag and audit         |
| Keyword flag false positive                              | Dismiss flag with reason                  |
| Malicious mass reporting                                 | Group reports, review reporter history    |

```txt
- Load sensitive target detail only in detail view.
- Use cached report_count where needed.
- Avoid N+1 queries for target owner/profile/report counts.
- Audit timeline should be paginated if long.
```

Recommended indexes:

```txt

reports(target_type, target_id)
reports(reason)
reports(moderation_case_id)
moderation_cases(status, priority, created_at)
```

## 36. Final Feature Statement

```txt
Moderation là governance core của UEConnect. Feature này biến report, auto-hide và keyword flag thành quy trình xử lý có kiểm soát: moderator/admin review, action có reason, audit bắt buộc, notification an toàn, reporter identity được bảo vệ, content hidden có placeholder, account-level sanctions có permission mạnh hơn và mọi quyết định đều traceable. Moderation không phải là nút xóa cho admin bấm vui tay, mà là hệ thống bảo vệ cộng đồng HCMUE khỏi spam, quấy rối, giả mạo, lừa đảo, vi phạm bản quyền và nội dung không phù hợp.
```
