# Greeting & Connection

## 1. Feature Summary

### 1.1. One-line Description

Greeting & Connection cho phép verified users gửi lời chào có ngữ cảnh đến UEers khác, sau đó nếu được chấp nhận thì hệ thống tạo connection, conversation và mở đường cho messaging chính thức.

### 1.2. Feature Goal

Feature này tồn tại để:

- Thay thế hoàn toàn dating-style “match” bằng một interaction thân thiện, tin cậy và phù hợp môi trường HCMUE.
- Cho user chủ động mở kết nối với người khác thông qua lời chào có message.
- Giảm spam bằng trạng thái pending/accepted/declined/blocked rõ ràng.
- Cho người nhận quyền quyết định accept hoặc decline.
- Khi accepted, tự động tạo connection.
- Khi accepted, tự động tạo hoặc mở conversation.
- Khi accepted, gửi notification cho sender.
- Khi declined, cập nhật trạng thái rõ ràng và ngăn spam gửi lại.
- Cho phép report/block từ greeting context.
- Gắn chặt với Discovery Profile, Messaging, Notification, Safety Reporting và Privacy.
- Đảm bảo user không thể gửi greeting đến blocked, suspended, banned, hidden hoặc unverified users.

### 1.3. Product Value

| Value | Description |
|---|---|
| User value | User có cách mở lời lịch sự, rõ mục đích, không áp lực và không dating-like |
| Product value | Tạo conversion chính từ Discovery sang Messaging |
| Business value | Biến UEConnect thành nền tảng kết nối sinh viên thật thay vì chỉ có profile card để ngắm |
| Trust value | Chỉ verified active users mới gửi/nhận greeting |
| Safety value | Người nhận có quyền accept/decline/report/block |
| UX value | Flow rõ: gửi lời chào → chờ phản hồi → accepted tạo chat |

### 1.4. Non-goals

Feature này không làm:

- Không dùng từ match.
- Không dùng từ crush.
- Không dùng từ swipe.
- Không dùng dating language.
- Không tự động tạo connection chỉ vì hai người cùng tương tác.
- Không cho user spam greeting nhiều lần đến cùng một người.
- Không cho user gửi greeting tới người đã decline nếu policy khóa.
- Không cho user gửi greeting tới blocked user.
- Không cho user gửi greeting tới suspended/banned/hidden user.
- Không cho unverified user gửi greeting.
- Không cho gửi greeting ẩn danh.
- Không dùng greeting như hệ thống quảng cáo, spam, recruiter outreach hoặc confession trá hình.

### 4.3. Access Rules

```txt
- Only verified active users can send greetings.
- Sender cannot send greeting to self.
- Sender cannot send greeting to unverified user.
- Sender cannot send greeting to suspended/banned/deleted user.
- Sender cannot send greeting to a user hidden by moderation.
- Sender cannot send greeting if either side blocked the other.
- Sender cannot send duplicate pending greeting.
- Sender cannot send greeting again after declined if product policy locks declined state.
- Receiver can accept/decline only greetings addressed to them.
- Accepted greeting creates connection.
- Accepted greeting creates or opens conversation.
- Block/report from greeting must enforce block across Discovery, Messaging and Greeting.
```

## 7. Greeting Model

### 7.1. Greeting Meaning

Greeting là một **connection request** có message ngắn, dùng để mở lời giữa verified UEers.

Greeting không phải:

```txt
match
crush
dating request
follow request
friend request kiểu mạng xã hội đại trà
mentor request
message request tự do
```

Greeting là:

```txt
lời chào có ngữ cảnh
yêu cầu kết nối nhẹ nhàng
entry point để tạo connection
điều kiện để mở conversation chính thức
```

### 7.2. Greeting Sources

| Source              | Description                                                  |
| ------------------- | ------------------------------------------------------------ |
| `discovery`         | Sent from Discovery card                                     |
| `profile`           | Sent from public profile                                     |
| `mentor_context`    | Sent from mentor-related profile context, not mentor request |
| `community_context` | Sent from community member profile                           |
| `feed_context`      | Sent after opening author profile from post                  |
| `search_context`    | Sent from search result/profile                              |

### 7.3. Greeting Message

Recommended message constraints:

```txt
min length = 0
max length = 300 characters
message optional but strongly encouraged
```

Reason:

```txt
Greeting should be lightweight. A 300-character max prevents essays, spam scripts and the ancient human tendency to over-explain.
```

### 7.4. Suggested Greeting Prompts

P2 future, but copy-ready:

```txt
Chào bạn, mình thấy tụi mình cùng ngành nên muốn kết nối để học hỏi thêm.
Chào bạn, mình quan tâm đến chủ đề này và muốn trao đổi thêm.
Chào bạn, mình thấy bạn có kinh nghiệm phù hợp nên muốn kết nối.
```

Do not provide flirty templates. Đây không phải app “tìm người thương trong bán kính 5km”.

## 8. Greeting Status Model

### 8.1. Greeting Statuses

| Status                  | Meaning                                        |
| ----------------------- | ---------------------------------------------- |
| `draft`                 | Created locally/not submitted, optional future |
| `pending`               | Sent, waiting for receiver response            |
| `accepted`              | Receiver accepted                              |
| `declined`              | Receiver declined                              |
| `cancelled`             | Sender cancelled before response               |
| `expired`               | Pending too long and expired                   |
| `blocked`               | Greeting invalidated by block                  |
| `reported`              | Greeting reported                              |
| `removed_by_moderation` | Removed due to moderation                      |

### 8.2. Recommended Lifecycle

```txt
pending
→ accepted
→ connection_created
→ conversation_created
```

```txt
pending
→ declined
```

```txt
pending
→ cancelled
```

```txt
pending
→ expired
```

```txt
pending
→ reported
→ blocked
```

### 8.3. Invalid Transitions

```txt
accepted → pending
declined → pending
blocked → accepted
removed_by_moderation → accepted
expired → accepted
cancelled → accepted
```

### 8.4. Greeting Expiry

Recommended P1:

```txt
pending greeting expires after 30 days
```

MVP option:

```txt
No automatic expiry in P0, but status model should support expired.
```

### 8.5. Declined Policy

Product decision from prior planning:

```txt
A user cannot send another greeting if the previous greeting to the same target was declined.
```

Recommended rule:

```txt
declined greeting creates a permanent no-resend lock unless receiver later initiates greeting or admin/system resets by policy.
```

This prevents “từ chối lần một, gửi tiếp lần hai, ba, bốn” behavior, also known as harassment with extra steps.

## 9. Connection Model

### 9.1. Connection Meaning

Connection là quan hệ hai chiều được tạo sau khi Greeting được accepted.

Connection enables:

```txt
- direct messaging
- conversation creation
- stronger profile access if privacy allows
- connection-based feed/discovery relevance
```

Connection does not mean:

```txt
- dating match
- endorsement
- mentor relationship
- community membership
- public friendship badge by default
```

### 9.2. Connection Statuses

| Status       | Meaning                        |
| ------------ | ------------------------------ |
| `active`     | Users are connected            |
| `removed`    | Connection removed by one side |
| `blocked`    | Block relation exists          |
| `restricted` | One account suspended/banned   |
| `archived`   | Future state                   |

### 9.3. Connection Creation Rule

When Greeting accepted:

```txt
- Create connection if no active connection exists.
- If inactive/removed connection exists, restore or create new according to policy.
- Create or open conversation.
- Mark greeting as accepted.
- Notify sender.
```

### 9.4. Connection Uniqueness

```txt
There must be only one active connection between two users.
```

Recommended normalized pair:

```txt
user_low_id
user_high_id
```

Because duplicate connection rows are how databases quietly start plotting against you.

## 10. Conversation Integration

### 10.1. Accepted Greeting Flow

After receiver accepts:

```txt
1. Greeting status = accepted
2. Connection status = active
3. Conversation is created or reactivated
4. System message can be inserted: "Hai bạn đã kết nối qua lời chào."
5. Sender receives notification
6. Receiver may be redirected to conversation
7. Sender can open conversation from notification
```

### 10.2. Conversation Creation Rule

```txt
- If no conversation exists between users: create direct conversation.
- If conversation exists but restricted due to pending request: activate it.
- If conversation exists and active: reuse it.
- If block exists: do not create/activate.
```

### 10.3. Message Request Difference

Messaging spec may support message request/waiting room later, but Greeting creates the clean official path:

```txt
Greeting accepted → connected users → direct conversation
```

No accepted Greeting, no active connection-based direct message.

## 11. Main User Flows

### 11.1. Send Greeting from Discovery

```txt
User opens Discovery
→ User sees profile card
→ User clicks Gửi lời chào
→ Greeting composer opens
→ User writes optional message
→ System validates target eligibility
→ System checks duplicate/declined/block status
→ Greeting is created with pending status
→ Target receives notification
→ Candidate removed from Discovery stack
```

### 11.2. Send Greeting from Profile

```txt
User opens public profile
→ User clicks Gửi lời chào
→ Greeting composer opens
→ User writes optional message
→ System validates access and target state
→ Greeting is created
→ Target receives notification
```

### 11.3. Accept Greeting

```txt
Receiver opens Greeting notification or inbox
→ Receiver views greeting message and sender profile preview
→ Receiver clicks Chấp nhận
→ System validates greeting is pending
→ System marks greeting accepted
→ System creates connection
→ System creates or opens conversation
→ Sender receives notification
→ Receiver is redirected to conversation or connection success state
```

### 11.4. Decline Greeting

```txt
Receiver opens Greeting
→ Receiver clicks Từ chối
→ System validates greeting is pending
→ System marks greeting declined
→ Sender may receive notification depending product policy
→ Sender cannot send another greeting to same receiver by default
```

### 11.5. Cancel Sent Greeting

P1.

```txt
Sender opens sent greetings
→ Sender selects pending greeting
→ Clicks Hủy lời chào
→ System marks greeting cancelled
→ Receiver no longer sees pending greeting
```

### 11.6. Report Greeting

```txt
Receiver opens Greeting
→ Selects Báo cáo
→ Chooses reason
→ Optional description
→ System creates report
→ System auto-blocks sender
→ Greeting status becomes reported/blocked
→ Sender can no longer interact directly
→ Report enters moderation queue
```

### 11.7. Block Sender from Greeting

```txt
Receiver opens Greeting
→ Selects Chặn
→ Confirms
→ System creates block
→ Greeting status becomes blocked
→ Any connection is removed/deactivated
→ Messaging/greeting/discovery blocked
```

## 13. Failure & Alternative Flows

| Flow                                     | Condition                                                                    | Expected Behavior |
| ---------------------------------------- | ---------------------------------------------------------------------------- | ----------------- |
| Sender unverified                        | Reject and redirect verification                                             |                   |
| Sender suspended/banned                  | Reject and redirect restricted                                               |                   |
| Target suspended/banned                  | Reject and show unavailable                                                  |                   |
| Target disabled Discovery                | Still allow from profile only if profile visibility allows; otherwise reject |                   |
| Target blocked sender                    | Reject                                                                       |                   |
| Sender blocked target                    | Reject                                                                       |                   |
| Pending greeting exists                  | Show pending state                                                           |                   |
| Previous greeting declined               | Prevent re-send                                                              |                   |
| Already connected                        | Show connection/chat CTA instead                                             |                   |
| Greeting accepted twice                  | Idempotent, do not duplicate connection                                      |                   |
| Conversation creation fails              | Rollback or retry safely                                                     |                   |
| Notification fails                       | Greeting/connection succeeds, notification retries                           |                   |
| Receiver accepts after sender suspended  | Reject or create restricted state depending policy                           |                   |
| Receiver declines after greeting expired | Reject stale action                                                          |                   |
| User reports greeting                    | Report + auto-block                                                          |                   |
| Auto-block fails after report            | Report saved, block retry/log                                                |                   |
| Account deleted                          | Greeting becomes unavailable                                                 |                   |

## 20. Security & Privacy

### 20.1. Security Rules

```txt
- Greeting creation must be authorized server-side.
- Greeting target eligibility must be checked server-side.
- Accept/decline must be receiver-only.
- Cancel must be sender-only.
- Message body must be sanitized/escaped.
- Greeting action must enforce block relation.
- Accepted flow must be transactional.
- Connection creation must be idempotent.
- Conversation creation must be idempotent.
- Suspended/banned users cannot send/respond.
- Hidden/deleted users cannot be targets.
```

### 20.2. Privacy Rules

```txt
- Greeting target preview shows privacy-safe fields only.
- Full MSSV is never shown.
- Email/phone/evidence/admin notes never shown.
- Decline reason is not required and not shown.
- Receiver declining does not expose extra personal information.
- Greeting message visible only to sender, receiver, and authorized moderator if reported.
- Analytics must not store raw greeting message.
```

### 20.3. Sensitive Data Matrix

| Data              |             Sensitive? | Handling                                    |
| ----------------- | ---------------------: | ------------------------------------------- |
| Greeting message  |               Moderate | Sender/receiver only; moderator if reported |

| Receiver response |               Moderate | Sender sees status                          |

| Block relation    |               Moderate | Owner/system only                           |

## 33. Final Feature Statement

```txt
Greeting & Connection là connection conversion core của UEConnect. Feature này cho phép verified active users gửi lời chào có message đến UEers khác, receiver có thể accept hoặc decline, accepted sẽ tạo connection và conversation để nhắn tin chính thức. Greeting thay thế hoàn toàn dating-style match, phải enforce block/privacy/account status server-side, chống duplicate/spam, hỗ trợ report/block và không được leak dữ liệu nhạy cảm.
```
