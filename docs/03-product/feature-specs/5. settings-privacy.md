# Settings & Privacy

## 1. Feature Summary

### 1.1. One-line Description

Settings & Privacy cho phép user quản lý tài khoản, quyền riêng tư, khả năng xuất hiện trong Discovery, danh sách block, notification preferences và các thiết lập an toàn cá nhân trong UEConnect.

### 1.2. Feature Goal

Feature này tồn tại để:

- Cho user kiểm soát thông tin nào được hiển thị với UEers khác.
- Cho user bật/tắt khả năng xuất hiện trong Discovery.
- Cho user quản lý blocked users.
- Cho user điều chỉnh notification preferences.
- Cho user hiểu dữ liệu nào được thu thập, dữ liệu nào không public.
- Cho user quản lý các thiết lập tài khoản cơ bản.
- Hỗ trợ role-specific privacy cho Student, Alumni, Advisor/Mentor.
- Đảm bảo Feed, Discovery, Greeting, Messaging, Mentor và Community tôn trọng privacy rules.
- Chuẩn bị khả năng localization tiếng Việt/English.
- Làm nền cho data privacy, account deactivation và session/security settings sau này.

### 1.3. Product Value

| Value | Description |
|---|---|
| User value | User thấy an tâm vì kiểm soát được hồ sơ, Discovery, block và notification |
| Product value | Social interaction an toàn hơn, giảm spam và unwanted contact |
| Business value | Sản phẩm đủ nghiêm túc để dùng trong môi trường trường học |
| Trust value | User hiểu UEConnect không public dữ liệu nhạy cảm như MSSV, evidence, email |
| Safety value | Block/privacy settings giảm harassment và unwanted messaging |
| UX value | User có một nơi rõ ràng để chỉnh app theo nhu cầu cá nhân |

### 1.4. Non-goals

Feature này không làm:

- Không cho user tự đổi primary role.
- Không cho user xóa dấu vết audit/admin action.
- Không cho user ẩn toàn bộ khỏi admin/moderation.
- Không cho user bypass report/moderation bằng privacy setting.
- Không public email, MSSV, evidence hoặc report history.
- Không cho user tự xóa account vĩnh viễn trong MVP nếu chưa có retention policy.
- Không làm dashboard privacy phức tạp như sản phẩm enterprise SaaS 40 tab.
- Không cho user unblock để bypass moderation hoặc message restriction bất hợp lý.
- Không cho privacy setting phá integrity của Discovery, Greeting, Messaging.

| Registered user  |    Limited | Can see account/verification status                                |

| Admin            |    Limited | Admin sees user info through admin tools, not personal settings UI |
| Suspended user   |    Limited | Can see account restriction page, maybe support/privacy            |
| Banned user      | No/limited | Restricted state only                                              |

### 4.2. Required Permissions

| Action                          | Required Role        | Required Permission | Scope  |
| ------------------------------- | -------------------- | ------------------- | ------ |
| View own settings               | authenticated user   | none                | own    |
| Update own privacy              | verified active user | none                | own    |
| Toggle discovery visibility     | verified active user | none                | own    |
| View blocked users              | verified active user | none                | own    |
| Unblock user                    | verified active user | none                | own    |
| Update notification preferences | verified active user | none                | own    |
| View data privacy explanation   | authenticated user   | none                | own    |
| Deactivate account              | verified active user | future policy       | own    |
| View user privacy in admin      | admin                | `manage_users`      | global |
| Override user restriction       | admin                | `manage_users`      | global |

### 4.3. Access Rules

```txt
- User can only edit their own settings.
- Settings updates must be server-side authorized.
- Privacy settings do not hide data from authorized admin/moderation.
- Discovery visibility must be enforced in Discovery queries.
- Blocked users must be enforced in Discovery, Greeting and Messaging.
- Account status overrides settings.
- Suspended/banned users cannot use settings to bypass restrictions.
- Email, full MSSV and evidence must remain hidden from public surfaces regardless of settings.
```

## 8. Privacy Model

### 8.1. Profile Visibility Levels

| Level                    | Meaning                                |
| ------------------------ | -------------------------------------- |
| `public_to_verified`     | Visible to verified UEConnect users    |
| `connections_only`       | Visible only to connected users        |
| `community_context_only` | Visible when sharing community context |
| `private`                | Minimal profile only                   |
| `hidden_by_moderation`   | Admin/moderator-hidden profile         |

### 8.2. Discovery Visibility

| Value           | Meaning                                       |
| --------------- | --------------------------------------------- |
| `enabled`       | User can appear in Discovery suggestions      |
| `disabled`      | User will not appear in Discovery suggestions |
| `paused`        | Temporarily hidden                            |
| `forced_hidden` | Hidden by moderation/admin                    |

Discovery visibility must exclude:

```txt
- current user
- unverified users
- suspended users
- banned users
- users with discovery_visibility disabled
- blocked users
- users who blocked current user
- profiles hidden by moderation
```

### 8.3. Field-level Visibility

| Field                   | Default                   |            User Can Change? | Notes                       |
| ----------------------- | ------------------------- | --------------------------: | --------------------------- |
| Avatar                  | Visible to verified users |                     Limited | Required profile identity   |
| Display name            | Visible to verified users |      No/private not allowed | Basic identity              |
| Role badge              | Visible                   |                          No | Trust                       |
| Verified badge          | Visible                   |                          No | Trust                       |
| Faculty                 | Visible                   |                      Yes/P1 | Can hide if privacy concern |
| Major                   | Visible                   |                      Yes/P1 | Can hide                    |
| Cohort                  | Visible                   |                      Yes/P1 | Can hide                    |
| Class code              | Hidden by default         |                         Yes | More specific               |
| Bio                     | Visible                   |                         Yes | User text                   |
| Interests               | Visible                   |                         Yes | Discovery/Mentor            |
| Connection goals        | Visible                   |                         Yes | Discovery                   |
| Career info             | Optional                  |                         Yes | Alumni privacy              |
| Mentor topics           | Visible if mentor active  | Yes via mentor availability |                             |
| Community participation | Hidden/optional           |                         Yes | Can expose social graph     |
| Email                   | Hidden                    |           No public default | Sensitive                   |
| Phone                   | Hidden                    |           No public default | Sensitive                   |
| Full MSSV               | Never public              |                          No | Sensitive                   |
| Evidence                | Never public              |                          No | Sensitive                   |

### 8.4. Privacy Defaults

Recommended default after onboarding:

```txt
profile_visibility = public_to_verified
discovery_visibility = enabled
show_faculty = true
show_major = true
show_cohort = true
show_class_code = false
show_interests = true
show_connection_goals = true
show_communities = false
show_career_info = false for alumni until user enables
email_public = false
phone_public = false
```

Reason:

```txt
UEConnect needs enough context for social discovery, but must avoid exposing overly specific identity data by default.
```

## 10. Block System

### 10.1. Block Meaning

When User A blocks User B:

```txt
- Existing connection is removed or deactivated.
- User B cannot send message to User A.
- User B cannot send greeting to User A.
- User A and User B are hidden from each other's Discovery.
- Existing conversation becomes restricted.
- Profile access becomes unavailable or minimal.
- Future interaction is blocked unless unblocked.
```

### 10.2. Block vs Report

| Action              | Meaning                                     |
| ------------------- | ------------------------------------------- |
| Block               | User-level safety control                   |
| Report              | Sends issue to moderation                   |
| Report + auto block | Recommended default for harmful interaction |

Decision from product:

```txt
Report action should auto-block the reported target by default.
```

Implementation note:

```txt
Auto-block should be clearly communicated before final submit or included in report confirmation.
```

### 10.3. Blocked Users Page

Blocked users page shows:

```txt
- avatar or safe placeholder
- display name
- role badge if allowed
- blocked date
- unblock action
```

Blocked users page must not show:

```txt
- full private profile
- message history preview
- report detail
- sensitive moderation context
```

### 10.4. Unblock Rules

Unblock should:

```txt
- allow future interaction if both users are active
- not automatically restore connection
- not automatically reopen old conversation
- not automatically recreate greeting
- not delete report history
```

This matters because “unblock = everything magically normal again” is how apps summon edge-case demons.

### 11.3. Notification Privacy

Notification previews:

```txt
- Message notification may include preview in UI if user allows.
- Analytics must not store message preview.
- Browser push should avoid sensitive content by default.
```

Recommended browser push copy:

```txt
Bạn có tin nhắn mới trên UEConnect.
```

Not recommended:

```txt
Nguyễn Văn A: nội dung tin nhắn đầy đủ...
```

Vì lockscreen là nơi bí mật riêng tư đi chết một cái chết rất công khai.

### 11.4. Notification Retention

Product decision:

```txt
Notification retention = 7 days
```

Rule:

```txt
After 7 days, notifications can be archived or deleted depending on retention policy.
```

## 13. Account Settings

### 13.1. Account Summary

Account settings show:

```txt
- email
- account status
- primary role
- verification status summary
- profile completion status
- linked Microsoft account if available
```

Account settings must not allow:

```txt
- self-change primary role
- self-change verified MSSV
- self-remove moderation status
```

### 13.2. Email Change

MVP recommendation:

```txt
Email change is out of scope or requires re-verification/admin review.
```

Reason:

```txt
Email is part of institutional trust. Changing it casually is how identity integrity gets turned into soup.
```

### 13.3. Password Change

If using email/password account:

```txt
- User can change password in Security Settings P2.
- Password min 8.
- Current password required.
```

If Microsoft SSO-only account:

```txt
- Password setting is optional/future.
- Login primarily through Microsoft.
```

### 13.4. Account Deactivation

P2/Future.

Expected behavior:

```txt
- User can request deactivation.
- Profile hidden.
- Discovery disabled.
- Messaging disabled or restricted.
- Data retention follows policy.
- Admin/audit records remain.
```

## 14. Main User Flow

### 14.1. Update Privacy Flow

```txt
User opens Settings
→ Opens Privacy
→ Changes profile visibility or field visibility
→ System validates setting
→ System saves setting
→ Discovery/Profile/Messaging surfaces respect new setting
→ User sees success state
```

### 14.2. Toggle Discovery Visibility Flow

```txt
User opens Discovery Visibility
→ Turns off "Xuất hiện trong Khám phá"
→ System saves setting
→ User is excluded from Discovery suggestions
→ Existing greetings/connections remain unless blocked
```

### 14.3. Blocked Users Flow

```txt
User opens Blocked Users
→ Views list of blocked users
→ Chooses Unblock
→ Confirms action
→ System removes block
→ User sees success state
→ Connection is not automatically restored
```

### 14.4. Notification Preference Flow

```txt
User opens Notifications Settings
→ Toggles non-critical notification type
→ System saves preference
→ Future notifications respect preference
```

### 14.5. Mentor Availability Shortcut Flow

```txt
Mentor opens Settings
→ Opens Mentor Availability
→ Pauses requests
→ System updates mentor availability
→ Mentor hidden or marked unavailable in Mentor listing
```

## 15. Detailed Flow

| Step | Actor Action                      | System Response          | State   |
| ---: | --------------------------------- | ------------------------ | ------- |
|    1 | User opens settings               | Load settings data       | loading |
|    2 | User opens privacy                | Show current visibility  | default |
|    3 | User changes setting              | Validate                 | editing |
|    4 | User saves                        | Persist changes          | saving  |
|    5 | System updates dependent surfaces | Recalculate visibility   | applied |
|    6 | User opens blocked list           | Load blocked users       | loading |
|    7 | User unblocks                     | Confirm and remove block | updated |
|    8 | User updates notifications        | Save preference          | updated |
|    9 | Save fails                        | Show retry               | error   |

## 16. Alternative & Failure Flows

| Flow                                 | Condition                                          | Expected Behavior                  |
| ------------------------------------ | -------------------------------------------------- | ---------------------------------- |
| Save privacy fails                   | Network/server issue                               | Preserve selection and allow retry |
| User tries invalid visibility        | Server validation rejects                          |                                    |
| User turns off Discovery             | Immediately excluded from future Discovery queries |                                    |
| Blocked user opens profile           | Show unavailable state                             |                                    |
| User unblocks someone                | Do not auto-restore old connection                 |                                    |
| Suspended user opens settings        | Show restricted/account status only                |                                    |
| Banned user opens settings           | Block access                                       |                                    |
| Notification preference fails        | Show retry                                         |                                    |
| Critical notification disabled       | Prevent disabling                                  |                                    |
| Account status changed while editing | Reject save and redirect if restricted             |                                    |

## 23. Security & Privacy

### 23.1. Security Rules

```txt
- User can update only their own settings.
- Block/unblock must be server-side authorized.
- Privacy settings must be enforced in queries, not only UI.
- Discovery visibility must be enforced server-side.
- Messaging must check block state before sending.
- Greeting must check block state before sending.
- Profile public response must be sanitized.
- Suspended/banned users cannot use settings to bypass restriction.
```

### 23.2. Privacy Rules

```txt
- Full MSSV is never public regardless of setting.
- Evidence is never public regardless of setting.
- Email is hidden by default.
- Phone is hidden by default.
- Report history is not public.
- Admin notes are not public.
- Analytics does not store raw sensitive settings or message previews.
- Browser push should avoid sensitive previews by default.
```

### 23.3. Sensitive Data

| Data                     |   Sensitive? | Handling                          |
| ------------------------ | -----------: | --------------------------------- |
| MSSV                     |          Yes | Admin-only/full, masked elsewhere |
| Evidence                 |          Yes | Private storage, never public     |
| Email                    |          Yes | Hidden by default                 |
| Phone                    |          Yes | Hidden by default                 |
| Block list               |     Moderate | Owner/admin only                  |
| Privacy settings         |     Moderate | Owner/admin only                  |
| Notification preferences | Low/moderate | Owner only                        |
| Message preview          |    Sensitive | Avoid in push/analytics           |
| Report details           |    Sensitive | Reporter/admin only               |

## 27. Edge Cases

| Edge Case                                                   | Expected Handling                                                   |
| ----------------------------------------------------------- | ------------------------------------------------------------------- |
| Discovery shows user who disabled visibility                | Bug; query must exclude                                             |
| Blocked user can send greeting                              | Server must reject                                                  |
| Blocked user can send message                               | Server must reject                                                  |
| User unblocks someone                                       | Do not auto-restore connection/conversation                         |
| Report auto-block fails                                     | Report should still submit; show warning if needed                  |
| Privacy save fails                                          | Preserve selection and retry                                        |
| User toggles visibility while Discovery card already loaded | Greeting/message actions must recheck server-side                   |
| User becomes suspended while editing settings               | Save rejected, redirect restricted                                  |
| Notification preview leaks sensitive message                | Bug; block release                                                  |
| Critical notification disabled by request tampering         | Server ignores/rejects                                              |
| User attempts to block self                                 | Validation error                                                    |
| Duplicate block                                             | Return existing block state                                         |
| User blocks admin/system account                            | Block normal interaction if applicable, not system-critical notices |
| Deactivated account appears in Discovery                    | Exclude                                                             |
| Admin-hidden profile toggles Discovery on                   | forced_hidden overrides user setting                                |
