# Admin Operations

## 1. Feature Summary

### 1.1. One-line Description

Admin Operations là hệ thống vận hành nội bộ của UEConnect, cho phép admin quản lý verification, users, roles, permissions, moderation, communities, mentor access, announcements, audit logs và các chỉ số vận hành cốt lõi.

### 1.2. Feature Goal

Feature này tồn tại để:

- Cho admin nắm tình trạng tổng quan của toàn hệ thống.
- Cho admin duyệt xác thực identity của user.
- Cho admin quản lý user account status: active, suspended, banned, suspicious.
- Cho admin xử lý report, moderation queue và content safety.
- Cho admin quản lý role, permission và scoped permission grants.
- Cho admin tạo/quản lý community/club và cấp quyền cho club manager.
- Cho admin duyệt mentor access hoặc grant mentor capability.
- Cho admin quản lý admin/system announcements.
- Cho admin xem audit log cho các hành động nhạy cảm.
- Cho admin theo dõi KPI vận hành: verified accounts, daily posts, mentor requests, greetings sent, reports pending.
- Đảm bảo toàn bộ hành động admin có permission, reason và audit phù hợp.
- Chuẩn bị hướng tích hợp SQL Server Audit cho hành động database-level như DDL, user management, schema changes.

### 1.3. Product Value

| Value | Description |
|---|---|
| User value | User được xác thực, bảo vệ và hỗ trợ bởi quy trình vận hành rõ ràng |
| Product value | UEConnect có khả năng vận hành như platform thật, không phải app demo |
| Business value | Có thể trình bày với stakeholder trường như một hệ thống có governance |
| Trust value | Verification, moderation và audit tạo nền tin cậy |
| Safety value | Admin có công cụ phát hiện, xử lý, truy vết rủi ro |
| UX value | Admin không phải mò từng bảng database như sinh vật tiền sử cầm SQL query |

### 1.4. Non-goals

Feature này không làm:

- Không cho admin bypass permission.
- Không cho admin xóa audit log từ UI.
- Không cho admin tự grant quyền nguy hiểm cho mình mà không audit.
- Không thay thế SQL Server Audit ở database-level.
- Không cho admin xem password hoặc secret.
- Không public admin dashboard cho user thường.
- Không biến admin panel thành nơi chỉnh sửa tùy tiện mọi dữ liệu sống.
- Không cho community owner xử lý report có conflict of interest.
- Không hỗ trợ recruiter portal.
- Không hỗ trợ multi-university operation trong MVP.
- Không làm BI dashboard phức tạp như enterprise data warehouse trong MVP.

## 4. Actors & Permissions

### 4.1. Admin-side Actors

| Actor | Description |
|---|---|
| `super_admin` | Highest system operator, can manage permissions and critical operations |
| `admin` | General system admin, manages users, verification, communities depending permission |
| `verification_reviewer` | Reviews identity verification requests |
| `moderator` | Handles reports and moderation cases |
| `community_admin` | Manages community/club operations globally or scoped |
| `club_manager` | Scoped operator for a specific community/club |
| `mentor_reviewer` | Reviews mentor access requests |
| `audit_viewer` | Can view audit logs but not necessarily mutate data |
| `support_operator` | Limited user support actions |
| `system` | Automated system actor for auto-hide, scheduled jobs, expiry |

### 4.2. User-facing Actors Affected

| Actor | Managed By Admin? | Notes |
|---|---:|---|
| Student | Yes | Verification, account status, moderation |
| Alumni | Yes | Verification, mentor capability, moderation |
| Advisor | Yes | Verification, mentor/advisor capability |
| Mentor | Yes | Mentor access approval/grant |
| Club Manager | Yes | Permission grant scoped by community |
| Suspended/Banned User | Yes | Admin handles restrictions |

| Permission | Purpose | Scope |
|---|---|---|
| `view_admin_dashboard` | View admin home | global |
| `manage_users` | View/update users | global |
| `review_verification` | Review verification requests | global |

| `request_verification_info` | Need more information action | global |
| `mark_verification_conflict` | Mark identity conflict | global |
| `suspend_users` | Suspend account | global |
| `ban_users` | Ban account | global |
| `reactivate_users` | Reactivate account | global |
| `view_reports` | View report queue | global/scoped |
| `moderate_content` | Hide/delete/restore content | global/scoped |
| `warn_users` | Warn users | global |
| `manage_communities` | Create/update/suspend communities | global/scoped |
| `manage_club` | Manage specific community | scoped by `community_id` |
| `manage_community_members` | Manage membership | scoped by `community_id` |
| `manage_community_resources` | Manage resources | scoped by `community_id` |
| `manage_mentor_access` | Approve/grant mentor access | global |
| `create_announcement` | Create announcements | global |
| `manage_announcements` | Manage announcements | global |
| `manage_permissions` | Grant/revoke permissions | global |
| `view_audit_log` | View audit records | global |
| `view_sensitive_data` | View sensitive data when explicitly allowed | restricted |

### 4.4. Access Rules

```txt
- Admin routes require authenticated active admin account.
- Admin route access is permission-based, not just role-name-based.
- Account status overrides permission.
- UI hiding is not authorization.
- Every sensitive admin action requires server-side permission check.
- Every destructive/moderation/verification action requires reason.
- Every sensitive admin action creates audit log.
- Scoped permission must include scope_type and scope_id.
- Admin can be granted multiple permissions.
- Admin cannot use self-grant dangerous permission without audit.
- Super admin actions must be audited.
```

### 7.3. Verification Actions

| Action | Description | Reason Required | Audit |
|---|---|---:|---:|
| `approve` | Approve verification and role | Yes | Yes |
| `reject` | Reject request | Yes | Yes |

| `suspend_suspicious` | Suspend suspicious account | Yes | Yes |

### 7.4. Verification Admin Rules

```txt
- MSSV must be unique.
- Admin can correct submitted fields before approve, but must audit changed fields.
- Approved evidence remains stored according to retention policy.
- Rejected evidence remains stored for review/history unless retention policy says otherwise.
- Need more information must include instruction.
- User can resubmit but verification flow handles rate limit.
- Admin action must notify user with safe reason/instruction.
```

```txt

```

These are not always primary roles. They are granted capabilities/permissions.

### 9.3. Permission Grant Model

UEConnect should support:

```txt

```

Example scoped grant:

```txt
permission = manage_club
scope_type = community
scope_id = 123
```

### 9.4. Custom Permission Grants Table

Recommended because scoped permissions are important.

| Field | Type | Notes |
|---|---|---|
| `id` | bigint / ulid | Primary key |
| `user_id` | FK users | Receiver |
| `permission_key` | string | e.g. manage_club |
| `scope_type` | string nullable | global/community/mentor/etc |
| `scope_id` | bigint nullable | e.g. community_id |
| `granted_by` | FK users | Admin |
| `reason` | text | Required |
| `starts_at` | datetime nullable | Optional |
| `expires_at` | datetime nullable | Optional |
| `status` | string enum | active/revoked/expired |
| `created_at` | datetime |  |
| `updated_at` | datetime |  |
| `revoked_at` | datetime nullable |  |
| `deleted_at` | datetime nullable | Soft delete |

### 9.5. Permission Rules

```txt
- Permission grants require reason.
- Permission grants require audit.
- Revocation requires reason.
- Scoped permission must be enforced by scope.
- Admin cannot manage permission without manage_permissions.
- Self-grant of critical permission must be audited and may require future dual approval.
```

## 14. Audit Log Operations

### 14.1. Audit Log Goal

Audit log answers:

```txt
- Who did what?
- To which target?
- When?
- From where?
- Why?
- What changed?
```

### 14.2. Actions Requiring Audit

```txt
- verification approve/reject/need_more_information/mark_conflict
- admin edits verification submitted data
- suspend/ban/reactivate user
- grant/revoke permission
- grant/revoke mentor access
- create/update/suspend community
- moderation hide/delete/restore/warn/suspend/ban
- create/publish critical announcement
- access sensitive evidence if enabled
- SQL/database-level changes through SQL Server Audit
```

### 14.3. Audit Log Fields

| Field | Type | Notes |
|---|---|---|
| `id` | bigint / ulid | Primary key |
| `actor_id` | FK users nullable | Admin/system |
| `actor_type` | string | user/system/job |
| `action` | string | Action key |
| `target_type` | string | user/post/report/etc |
| `target_id` | string | Target id |
| `before_values` | json nullable | Sensitive, restricted |
| `after_values` | json nullable | Sensitive, restricted |
| `reason` | text nullable | Required for sensitive actions |
| `ip_address` | string nullable | Security |
| `user_agent` | string nullable | Security |
| `metadata` | json nullable | Safe metadata |
| `created_at` | datetime |  |

### 14.4. SQL Server Audit Direction

UEConnect uses MS SQL Server, so database-level audit should be considered for:

```txt
- DDL changes
- permission changes
- schema changes
- direct table modification by privileged accounts
- login/security events
- database user changes
- suspicious admin database operations
```

Application audit and SQL Server Audit are different:

| Layer | Captures |
|---|---|
| Application Audit | Product-level action: approve verification, suspend user, hide post |
| SQL Server Audit | Database-level action: ALTER TABLE, direct UPDATE, role/permission changes |

Rule:

```txt
Application audit is required for product governance.
SQL Server Audit is required for database governance.
They complement each other, not replace each other.
```

Nói cách khác, đừng mong SQL Server Audit hiểu “admin đã reject verification vì minh chứng không hợp lệ”. Database không đọc được ý định con người, thật đáng ghen tị.

## 18. Main Admin Flows

### 18.1. Admin Login Flow

```txt
Admin logs in
→ Account active check
→ Admin permission check
→ If opening /admin and has permission: show dashboard
→ If lacks permission: show 403
```

Important:

```txt
Admin is not auto-forced to /admin after normal login.
Only route access determines admin panel entry.
```

### 18.2. Verification Review Flow

```txt
Admin opens verification queue
→ Filters pending requests
→ Opens request detail
→ Reviews evidence and submitted info
→ Optionally edits fields before approval
→ Chooses approve / reject / need more information / mark conflict / suspend suspicious
→ Enters reason/instruction
→ System applies action
→ User notified
→ Audit created
```

### 18.3. User Suspension Flow

```txt
Admin opens user detail
→ Reviews account/report/moderation history
→ Chooses suspend
→ Enters reason and duration
→ Confirms action
→ System updates account_status
→ Active sessions restricted on next request
→ User notified with safe reason
→ Audit created
```

### 18.4. Permission Grant Flow

```txt
Admin opens user detail
→ Opens permissions tab
→ Chooses permission key
→ Selects scope if scoped
→ Enters reason
→ Grants permission
→ Audit created
→ Permission takes effect immediately
```

### 18.5. Community Manager Grant Flow

```txt
Admin opens community detail
→ Opens members/operators
→ Selects user
→ Grants manage_club scoped to community_id
→ Enters reason
→ Audit created
→ User can manage only that community scope
```

### 18.6. Mentor Access Approval Flow

```txt
Admin opens mentor access queue
→ Reviews user profile and mentor fields
→ Approves / rejects / requests more info
→ Enters reason/instruction
→ System updates mentor_access_status
→ User notified
→ Audit created
```

## 25. Security & Privacy

### 25.1. Security Rules

```txt
- Admin routes require auth + active account + permission.
- Every sensitive action is server-side authorized.
- Every destructive/sensitive action requires reason.
- Every sensitive action creates audit log.
- Permission grants/revocations require audit.
- Admin cannot see secrets/passwords.
- Evidence preview requires verification permission.
- Report details require moderation permission.
- Message report content requires restricted permission.
- SQL Server Audit should be enabled for database-level changes.
```

### 25.2. Privacy Rules

```txt
- Admin panel must not expose more data than permission allows.
- Evidence is preview-only for verification reviewers.
- Reporter identity hidden unless permission allows.
- Private messages are not generally browsable.
- Admin notes are never visible to normal users.
- Audit logs are restricted.
- Analytics must not contain raw sensitive content.
```

| Message content | Yes | Only report context |
| Admin notes | Yes | Admin-only |

## 29. Edge Cases

```txt
Laravel Policies/Gates
Laravel Form Requests
Laravel Events
Laravel Notifications
Laravel Queues
Laravel Scheduler
Laravel Soft Deletes
Laravel Transactions
Laravel Eloquent Scopes
Custom scoped permission grants
MS SQL Server
SQL Server Audit for database-level auditing
```

### 34.2. Permission Direction

Recommended:

```txt
Use custom permission_grants table for scoped permissions.
Use Laravel Gates/Policies to enforce permission.
Spatie Permission can be used for simple global roles only if it does not complicate scoped grants.
```

### 34.3. Admin UI Direction

Recommended stack:

```txt
Blade Components
TailwindCSS
Alpine.js
Livewire where useful for tables/forms
Lucide Icons
Vite
```

Livewire good for:

```txt
- admin tables
- filters
- verification action panels
- permission grants
- dashboard widgets
```

Không cần biến mọi dropdown thành Livewire component có đời sống riêng. Một số thứ chỉ cần HTML tử tế, xin đừng over-engineer cho vui tay.

### 34.4. Transaction Direction

Use transactions for:

```txt
- verification action
- user suspend/ban/reactivate
- permission grant/revoke
- mentor access decision
- community suspend/reactivate
- moderation action
- announcement publish if audit required
```

After commit:

```txt
- send notification
- fire analytics event
- clear cache if needed
```

## 38. Final Feature Statement

```txt
Admin Operations là operational control center của UEConnect. Feature này cho phép admin vận hành trust, safety, identity, community, mentor và governance thông qua dashboard, verification queue, user management, moderation, permission grants, community/mentor operations, announcements và audit logs. Mọi hành động nhạy cảm phải được permission-check server-side, có reason, có audit và không leak dữ liệu nhạy cảm. Admin panel không phải CRUD cho vui, mà là lớp kiểm soát để UEConnect có thể tồn tại như một product thật trong môi trường HCMUE.
```
