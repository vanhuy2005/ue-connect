# Search & Filter

## 1. Feature Summary

### 1.1. One-line Description

Search & Filter cho phép verified users tìm kiếm và lọc người dùng, bài viết, cộng đồng, mentor, career pathways, tài nguyên và nội dung liên quan trong UEConnect theo cách nhanh, an toàn, privacy-aware và phù hợp với bối cảnh HCMUE.

### 1.2. Feature Goal

Feature này tồn tại để:

- Giúp user tìm đúng nội dung/người/cộng đồng/mentor nhanh hơn.
- Hỗ trợ global search trên toàn app.
- Hỗ trợ scoped search trong từng module như Feed, Discovery, Community, Mentor, Career Pathway, Messaging.
- Hỗ trợ filter theo role, khoa, ngành, khóa, topic, community, career pathway, thời gian, trạng thái.
- Tôn trọng privacy settings, block, account status, verification status và moderation status.
- Không leak thông tin nhạy cảm qua search result.
- Cung cấp ranking hợp lý, không chỉ “LIKE query rồi cầu trời”.
- Hỗ trợ search UX thân thiện trên PWA/mobile.
- Hỗ trợ admin search riêng với permission cao hơn, audit khi truy cập dữ liệu nhạy cảm.
- Chuẩn bị kiến trúc để mở rộng từ SQL search sang full-text search hoặc search engine sau này.

### 1.3. Product Value

| Value | Description |
|---|---|
| User value | User tìm được UEers, mentor, community, bài viết, tài nguyên, pathway nhanh hơn |
| Product value | Tăng discoverability, retention và khả năng điều hướng trong PWA |
| Business value | UEConnect trông như platform thật, không phải app demo phải bấm mò từng tab |
| Trust value | Search tôn trọng verified-only, privacy, block và moderation |
| Safety value | Không để search trở thành công cụ stalk, scrape, lộ MSSV/email/evidence |
| UX value | Search/filter giúp user kiểm soát trải nghiệm thay vì bị feed/discovery quyết định hết |

### 1.4. Non-goals

Feature này không làm:

- Không cho guest search dữ liệu nội bộ UEConnect.
- Không public search index ra ngoài HCMUE.
- Không search theo full MSSV public.
- Không search evidence xác thực.
- Không search private messages toàn cục.
- Không cho user thường tìm report, audit log, admin notes.
- Không cho search bypass privacy/block/moderation.
- Không tạo people search kiểu stalking.
- Không expose ranking score.
- Không làm search engine phức tạp như Elasticsearch ngay MVP nếu chưa cần.
- Không để filter UI thành bảng điều khiển máy bay cho sinh viên năm nhất hoảng loạn.

## 4. Actors & Permissions

### 4.1. Actors

| Actor                          |             Can Search? | Notes                                     |
| ------------------------------ | ----------------------: | ----------------------------------------- |
| Guest                          | No/Limited landing only | No internal data                          |
| Registered but unverified user |                      No | Redirect verification                     |
| Pending user                   |                      No | Cannot access app search                  |
| Verified Student               |                     Yes | Main actor                                |
| Verified Alumni                |                     Yes | Privacy-filtered                          |
| Verified Advisor               |                     Yes | Privacy-filtered                          |
| Mentor                         |                     Yes | Same as verified user                     |
| Club Manager                   |                     Yes | Can search within managed community scope |
| Admin                          |                     Yes | Admin search with permission              |
| Moderator                      |                     Yes | Moderation search with permission         |
| Suspended user                 |              Limited/No | Usually restricted                        |
| Banned user                    |                      No | No access                                 |

### 4.2. Required Permissions

| Action                          | Required Actor/Permission           | Scope                  |
| ------------------------------- | ----------------------------------- | ---------------------- |
| Use global app search           | verified active user                | app                    |
| Search public/visible profiles  | verified active user                | privacy-filtered       |
| Search feed posts               | verified active user                | visible posts only     |
| Search community list           | verified active user                | visible communities    |
| Search inside private community | active member                       | community_id           |
| Search community resources      | member/visibility policy            | community_id           |
| Search mentors                  | verified active user                | visible active mentors |
| Search career pathways          | verified active user                | published pathways     |
| Search own conversations        | conversation participant            | own conversations      |
| Admin user search               | `manage_users`                      | global                 |
| Admin verification search       | `review_verification`               | global                 |
| Admin audit search              | `view_audit_log`                    | global                 |
| Admin moderation search         | `view_reports` / `moderate_content` | global/scoped          |

### 4.3. Access Rules

```txt
- Search results must be filtered server-side.
- UI filter hiding is not authorization.
- User must be verified and active to use app search.
- Search result must not include blocked users either direction.
- Suspended/banned/deleted users are excluded from user-facing search.
- Hidden/removed content is excluded unless viewer has moderation/admin permission.
- Private community content appears only to active members or authorized managers/admins.
- Private messages are searchable only inside participant's own conversations.
- Admin search result visibility depends on permission.
```

| Filter           | Type        | Description                                   |
| ---------------- | ----------- | --------------------------------------------- |
| `category`       | enum        | people/post/community/mentor/pathway/resource |
| `keyword`        | string      | User query                                    |
| `date_range`     | date range  | Created/updated date                          |
| `role`           | enum        | student/alumni/advisor/mentor                 |
| `faculty`        | enum/string | Faculty                                       |

| `community_type` | enum        | club/academic/etc                             |
| `career_pathway` | id          | Related pathway                               |
| `topic`          | tag/id      | Topic                                         |
| `status`         | enum        | Only where allowed                            |
| `has_media`      | boolean     | Posts/resources                               |

| `source_type`    | enum        | Official/admin/mentor/community               |

### 7.2. Filter UX Rules

```txt
- Mobile filters use bottom sheet.
- Desktop filters use sidebar or top filter bar.
- Active filters show as chips.
- User can clear individual filter.

- Filter options must be privacy-safe.
```

MVP:

```txt
- Preserve filters in current page URL.
- Do not persist globally across sessions unless recent search feature implemented.
```


## 8. Search Ranking MVP

### 8.1. Ranking Principles

Search ranking should prioritize:

```txt
- exact match
- verified/active visibility
- user context relevance
- HCMUE academic context
- recency where appropriate
- profile/content completeness
- community/mentor availability
- safety/moderation quality
```

Search ranking should not prioritize:

```txt
- attractiveness
- social clout only
- spammy keyword stuffing
- hidden popularity score
- sensitive identity fields
```

### 8.2. Ranking Formula Concept

```txt
score =
  exact_match_score
+ prefix_match_score
+ field_weight_score
+ context_relevance_score
+ recency_score
+ quality_score
- moderation_penalty
- stale_content_penalty
```

### 8.3. Field Weights

People:

```txt
display_name: high
major/faculty: medium
bio/interests: medium
private fields: excluded
```

Mentor:

```txt
expertise_topics: high
headline: high
can_help_with: medium
experience_summary: medium
```

Community:

```txt
name: high
type/category: high
description: medium
related pathway/program: medium
```

Career Pathway:

```txt
title: high
domain cluster: high
skills/programs: medium
description: medium
```

Posts:

```txt
body/title: high
tags/topics: medium
author context: low/medium
```

Resources:

```txt
title: high
description: medium
community/pathway: medium
```

### 8.4. Vietnamese Search Handling

MVP should support basic Vietnamese-friendly behavior:

```txt
- case-insensitive search
- accent-insensitive search if feasible
- trim whitespace
- normalize repeated spaces
- safe escaping
```

Examples:

```txt
"cong nghe giao duc" should match "Công nghệ giáo dục"
"su pham tin hoc" should match "Sư phạm Tin học"
```

Implementation can use normalized columns:

```txt
search_text_normalized
```

Because asking SQL Server to read Vietnamese like a patient librarian without preparation is optimistic, bordering on spiritual.

## 10. Privacy & Safety Rules

### 10.1. Privacy Enforcement

Every search result must check:

```txt
- account_status
- verification_status
- profile visibility
- field-level privacy
- discovery/profile visibility where relevant
- block relationship both directions
- community visibility/membership
- content visibility
- moderation status
```

### 10.2. Search Result Redaction

If field is private:

```txt
- Do not include it in result payload.
- Do not use it as snippet.
- Do not show "hidden because private" unless useful.
```

### 10.3. Block Rules

```txt
- If viewer blocked target, target excluded.
- If target blocked viewer, target excluded.
- Related content authored by blocked users should be hidden or reduced according to safety policy.
- Conversations with blocked user become restricted.
```

### 10.4. Moderation Rules

```txt
- hidden_by_moderation content excluded from normal search.
- removed content excluded.
- suspended community excluded or shown locked depending context.
- banned/suspended users excluded from people search.
- reported but not hidden content can remain visible unless threshold/auto-hide applies.
```

### 10.5. Anti-abuse Rules

```txt
- Rate limit search requests eventually.
- Reject extremely long queries.
- Log abnormal search patterns.
- Do not allow user enumeration through exact email/MSSV search.
- Do not expose result count for sensitive scopes if it enables inference.
```

MVP may not need login rate limit per earlier auth decision, but search absolutely needs guardrails eventually, because “search” is what turns a social app into a scraping buffet.

## 11. Admin Search

### 11.1. Admin Search Scope

Admin search is separate from user search.

Admin can search depending permission:

```txt
- users
- verification requests
- reports
- moderation cases
- communities
- mentor access requests
- audit logs
- announcements
```

### 11.2. Admin Search Rules

```txt
- Admin search requires admin route + permission.
- Admin result sections depend on permission.
- Sensitive search access may be audited.
- Search over evidence metadata requires verification permission.
- Search over reports requires moderation permission.
- Audit log search requires audit permission.
```

### 11.3. Admin Search Result Groups

```txt
Users
Verification
Reports
Moderation Cases
Communities
Mentor Access
Audit Logs
Announcements
```

### 11.4. Admin Sensitive Search

For sensitive categories:

```txt
- Do not show raw evidence content in list.
- Do not show raw report description in search suggestion.
- Do not show private message body unless in reported context and permission allows.
- Audit search should show action summary, actor, target, time.
```

## 12. Main User Flows

### 12.1. Global Search Flow

```txt
User clicks search bar
→ Enters keyword
→ System validates query
→ User submits or chooses suggestion
→ System checks verified active account
→ Searches allowed categories
→ Applies privacy/permission/block/moderation filters
→ Ranks results
→ Shows grouped results
→ User opens result
```

### 12.2. Filter Flow

```txt
User opens module page
→ Opens filters
→ Selects filter values
→ Applies filters
→ URL updates
→ Results reload
→ Active filter chips displayed
→ User clears filter or opens result
```

### 12.3. Empty Result Flow

```txt
User searches
→ No results after filters
→ Show empty state
→ Suggest removing filters or trying broader keyword
```

### 12.4. Search to Greeting Flow

```txt
User searches people
→ Opens visible profile result
→ Clicks Gửi lời chào
→ Greeting composer opens
```

### 12.5. Search to Mentor Request Flow

```txt
User searches mentor/topic
→ Opens mentor profile
→ Clicks Gửi yêu cầu mentor
→ Mentor request form opens with topic prefilled if possible
```

### 12.6. Search to Community Join Flow

```txt
User searches community
→ Opens community detail
→ Clicks Tham gia
→ Join flow starts
```

### 12.7. Admin Search Flow

```txt
Admin opens admin topbar search
→ Enters query
→ System checks admin permissions
→ Shows grouped admin results
→ Admin opens allowed result
→ Sensitive access optionally audited
```

## 19. Security & Privacy

### 19.1. Security Rules

```txt
- Search must use parameterized queries.
- Search must enforce authorization after query and before response.
- Sensitive fields must not be indexed into user-facing search.
- Admin search must be permission-gated.
- Private content cannot appear in suggestions.
- Search result payload must be sanitized.
- Search endpoint should have rate limiting eventually.
- Long/expensive queries must be limited.
```

### 19.2. Privacy Rules

```txt
- Do not show full MSSV.
- Do not show private email/phone.
- Do not show verification evidence.
- Do not show private messages outside participant scope.
- Do not show report/audit/admin data to normal users.
- Do not reveal that a blocked/private user exists through search.
- Do not show hidden/deleted content snippets.
```

### 19.3. Sensitive Data Matrix

| Data                 | Sensitive? | User Search           | Admin Search            |
| -------------------- | ---------: | --------------------- | ----------------------- |
| Display name         |        Low | Allowed if visible    | Allowed                 |
| Full MSSV            |       High | No                    | Permission only         |
| Email                |       High | No by default         | Permission only         |
| Phone                |       High | No                    | Restricted              |
| Evidence             |       High | No                    | Verification permission |
| Private message body |       High | Own conversation only | Reported context only   |
| Report detail        |       High | No                    | Moderator permission    |
| Audit log            |       High | No                    | Audit permission        |
| Admin note           |       High | No                    | Admin permission        |
| Search history       |   Moderate | Owner only            | Not casually visible    |

## 23. Edge Cases

| Edge Case                                   | Expected Handling                                |
| ------------------------------------------- | ------------------------------------------------ |

| Hidden post appears                         | Must not happen                                  |

```txt
- Module search can paginate deeper.
- Use normalized search columns where needed.
- Use indexes for filter fields.

```
