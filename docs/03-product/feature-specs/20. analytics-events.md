# Analytics Events

## 1. Feature Summary

### 1.1. One-line Description

Analytics Events cho phép UEConnect ghi nhận các sự kiện hành vi và vận hành quan trọng vào database để đo product health, activation, engagement, trust, safety và MVP success metrics mà không xâm phạm dữ liệu riêng tư của user.

### 1.2. Feature Goal

Feature này tồn tại để:

- Đo mức độ thành công của UEConnect bằng các chỉ số đã định nghĩa.
- Theo dõi activation: verified account, onboarding completed, profile completed.
- Theo dõi engagement: posts, comments, greetings, messages, community joins.
- Theo dõi mentor value: mentor requests, accepted requests, conversations created.
- Theo dõi trust/safety: reports, moderation actions, blocks, auto-hide.
- Theo dõi PWA usage: install prompt, push permission, notification clicks.
- Cung cấp dữ liệu cơ bản cho admin dashboard sau này.
- Hỗ trợ ra quyết định product dựa trên dữ liệu thay vì “cảm giác hôm nay app có vẻ ổn”.
- Không lưu nội dung nhạy cảm như message body, report description, verification evidence, admin notes.
- Thiết kế đủ đơn giản cho MVP nhưng đủ chuẩn để scale.

### 1.3. Product Value

| Value | Description |
|---|---|
| Product value | Biết feature nào có người dùng thật, feature nào chỉ nằm đó làm cảnh |
| Business value | Chứng minh UEConnect có traction khi triển khai thật hoặc đưa vào portfolio |
| UX value | Tìm được điểm drop-off trong auth, verification, onboarding, mentor request |
| Trust value | Theo dõi verification và safety mà không leak thông tin nhạy cảm |
| Admin value | Có số liệu vận hành cơ bản cho dashboard |
| Engineering value | Event schema nhất quán giúp debug và phân tích dễ hơn |

### 1.4. Non-goals

Feature này không làm:

- Không xây dashboard phân tích phức tạp trong MVP.
- Không dùng external analytics service bắt buộc trong MVP.
- Không lưu raw message body.
- Không lưu raw comment/post body trong analytics payload.
- Không lưu report description.
- Không lưu verification evidence path.
- Không lưu private profile fields.
- Không tracking kiểu theo dõi quá mức.
- Không bán dữ liệu.
- Không dùng analytics để xếp hạng user công khai.
- Không biến UEConnect thành một cái camera hành vi có logo trường.

## 6. Event Design Principles

### 6.1. Principle 1: Privacy First

Analytics event must not contain:

```txt
- raw message body
- raw post body
- raw comment body
- report description
- verification evidence file path
- verification evidence notes
- admin internal notes
- private profile fields
- full MSSV
- email
- phone number
- raw search query if it may contain personal data
```

### 6.2. Principle 2: Event is Not Audit Log

Analytics tells product behavior.

Audit log tells accountability.

Example:

```txt
analytics_event:
admin_dashboard_opened

audit_log:
admin_suspended_user with reason, actor, target, before/after state
```

Không dùng analytics thay audit. Đó là như dùng nhật ký ăn uống làm hồ sơ pháp lý, nghe đã thấy sai.

### 6.3. Principle 3: Event Payload Must Be Small

Good payload:

```json
{
  "source": "discovery",
  "has_message": true,
  "target_role": "student"
}
```

Bad payload:

```json
{
  "message": "full greeting message here",
  "target_email": "student@hcmue.edu.vn",
  "private_note": "..."
}
```

### 6.4. Principle 4: Use Categories Instead of Raw Content

Use:

```txt
reason_category = spam
file_type = pdf
query_length = 12
query_hash = abc123
has_attachment = true
```

Do not use:

```txt
full report text
full uploaded file path
full search query
raw message content
```

### 6.5. Principle 5: Events Should Be Stable

Event names should not change casually.

Bad:

```txt
user_click_button_thing
clicked_new_ui_final_v2
```

Good:

```txt
greeting_sent
mentor_request_submitted
community_join_requested
```

## 7. Event Naming Convention

### 7.1. Format

Use snake_case:

```txt
module_action
```

Examples:

```txt
verification_submitted
greeting_sent
mentor_request_accepted
community_join_requested
message_sent
notification_opened
```

### 7.2. Naming Rules

```txt
- Use past tense for completed events.
- Use started/opened for user begins flow.
- Use submitted for form submission.
- Use failed for failed action.
- Use blocked/prevented for system prevention.
- Use category prefix if module-specific.
```

### 7.3. Recommended Action Words

| Action      | Meaning                            |
| ----------- | ---------------------------------- |
| `opened`    | User opens page/modal              |
| `started`   | User starts flow                   |
| `submitted` | User submits form                  |
| `completed` | Flow completed                     |
| `sent`      | Message/greeting/notification sent |
| `received`  | Event received by user/system      |
| `accepted`  | Request accepted                   |
| `declined`  | Request declined                   |
| `failed`    | Action failed                      |
| `prevented` | System prevented action            |
| `created`   | Entity created                     |
| `updated`   | Entity updated                     |
| `deleted`   | Entity deleted                     |
| `reported`  | Report submitted                   |
| `blocked`   | Block created                      |
| `viewed`    | Detail viewed                      |
| `clicked`   | CTA/result clicked                 |

| ----------------- | ---------------------- | -------: | -------------------------- |
| `id`              | bigint / ulid          |      Yes | Primary key                |

| `event_category`  | string enum            |      Yes | auth/feed/community/etc    |
| `actor_id`        | FK users nullable      |       No | Null for system/guest      |

| `session_id`      | string nullable        |       No | App session                |
| `request_id`      | string nullable        |       No | Request correlation        |
| `source`          | string nullable        |       No | page/module/source         |
| `target_type`     | string nullable        |       No | user/post/community/etc    |
| `target_id`       | string/bigint nullable |       No | Target record id           |
| `scope_type`      | string nullable        |       No | community/conversation/etc |
| `scope_id`        | string/bigint nullable |       No | Scope id                   |
| `metadata`        | json nullable          |       No | Safe payload only          |
| `ip_hash`         | string nullable        |       No | Optional hashed IP         |

| `device_type`     | string nullable        |       No | mobile/desktop/tablet      |

## 11. Event Triggering Rules

### 11.1. Server-side vs Client-side Events

| Event Type                        | Preferred Source                       |
| --------------------------------- | -------------------------------------- |
| Entity created/updated/deleted    | Server-side                            |
| Verification approved/rejected    | Server-side                            |
| Message sent                      | Server-side after DB commit            |
| Greeting sent/accepted            | Server-side                            |
| Mentor request submitted/accepted | Server-side                            |
| Report/moderation/admin action    | Server-side                            |
| Page opened/viewed                | Client-side or server-side             |
| Button/modal opened               | Client-side                            |
| PWA install prompt                | Client-side                            |
| Push permission                   | Client-side + server-side subscription |

### 11.2. After Commit Rule

For important server events:

```txt
Create analytics event after successful database commit.
```

Examples:

```txt
post_created
greeting_sent
message_sent
mentor_request_submitted
community_joined
verification_approved
```

Do not track success if transaction failed. Incredible that this has to be said, but software has a gift for lying politely.

### 11.3. Failed Action Events

Failed events should track reason category only:

```txt
message_send_failed:
- blocked
- validation_error
- permission_denied
- account_restricted
- network_error
- server_error
```

Do not include raw exception stack trace in analytics metadata.

## 12. Deduplication & Idempotency

### 12.1. Why Needed

Some events can be duplicated because of:

```txt
- double click
- browser retry
- websocket reconnect
- queue retry
- back button resubmit
- mobile network weirdness
```

### 12.2. Deduplication Fields

Add optional:

```txt
event_dedup_key
```

Recommended format:

```txt
{event_name}:{actor_id}:{target_type}:{target_id}:{client_event_id}
```

### 12.3. Deduplication Examples

| Event                         | Dedup Key                 |
| ----------------------------- | ------------------------- |
| `message_sent`                | message id                |
| `greeting_sent`               | greeting id               |
| `mentor_request_submitted`    | mentor request id         |
| `post_created`                | post id                   |
| `notification_opened`         | user id + notification id |
| `pwa_install_prompt_accepted` | user id + device/session  |

### 12.4. Duplicate Handling

```txt
- If same dedup key exists, ignore new duplicate.
- If event is harmless page view, dedup may not be needed.
- Queue retry should not duplicate important success events.
```

## 13. Privacy & Compliance Rules

### 13.1. Data Minimization

Only collect what is useful.

Bad:

```txt
Track every mouse movement.
```

Good:

```txt
Track onboarding step completed.
```

### 13.2. No Raw Sensitive Content

Forbidden in analytics:

```txt
- message body
- post body
- comment body
- report description
- appeal description
- evidence note
- evidence file path
- email
- phone
- full MSSV
- IP address raw
- user agent raw if avoidable
```

### 13.3. Hashing

If needed:

```txt
ip_hash = hash(ip + app_salt)
user_agent_hash = hash(user_agent + app_salt)
query_hash = hash(normalized_query + app_salt)
```

Do not use analytics DB as a cute little surveillance bunker.

### 13.4. Retention

Recommended MVP:

```txt
Raw analytics events retention = 180 days
Aggregated daily metrics retention = long-term
```

Alternative for student project:

```txt
Raw analytics events retention = 90 days
Aggregates retained indefinitely
```

### 13.5. Deletion / Anonymization

If user account is deleted:

```txt
- Keep aggregate metrics.
- Raw event actor_id can be anonymized/nullified depending policy.
- Do not keep personal identifiers unnecessarily.
```

## 17. Security Requirements

### 17.1. Security Rules

```txt
- Client event endpoint must require authentication except explicitly public PWA events if needed.
- Event names must be whitelisted.
- Metadata must be sanitized.
- User cannot spoof actor_id.
- actor_id is derived from authenticated session.
- Admin analytics access is permission-gated.
- Raw analytics access is restricted.
- Analytics must not contain secrets/tokens.
- Sensitive events should be server-generated only.
```

### 17.2. Spoofing Prevention

Client cannot send:

```json
{
  "actor_id": 999,
  "event_name": "verification_approved"
}
```

Server must derive actor and reject server-only events from client.

### 17.3. Server-only Events

## 23. Final Feature Statement

```txt
Analytics Events là measurement layer của UEConnect. Feature này ghi nhận các sự kiện product quan trọng như verification, onboarding, post, greeting, mentor request, community, messaging, notification, search, report và moderation vào database theo schema thống nhất. Analytics phục vụ đo lường product health và success metrics, không thay thế audit log, không lưu nội dung nhạy cảm, không dùng để giám sát cá nhân quá mức. MVP ưu tiên event tracking an toàn và aggregation cơ bản, dashboard nâng cao để P2/P3.
```
