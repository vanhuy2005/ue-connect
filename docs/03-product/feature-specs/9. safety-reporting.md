# Safety Reporting

## 1. Feature Summary

### 1.1. One-line Description

Safety Reporting cho phép verified users báo cáo nội dung, hồ sơ, tin nhắn, cộng đồng, mentor request hoặc hành vi vi phạm để hệ thống tự động bảo vệ user, đưa report vào moderation queue và hỗ trợ admin xử lý an toàn.

### 1.2. Feature Goal

Feature này tồn tại để:

- Cho user báo cáo nội dung hoặc hành vi vi phạm trong UEConnect.
- Bảo vệ user ngay sau report bằng auto-block mặc định.
- Đưa report vào moderation queue để admin/moderator xử lý.
- Hỗ trợ report cho nhiều target type: profile, post, comment, message, community, community post, mentor request, evidence abuse.
- Ngăn duplicate pending report cùng target từ cùng reporter.
- Hỗ trợ auto-hide nếu một target nhận nhiều report vượt threshold.
- Hỗ trợ keyword flagging để phát hiện nội dung rủi ro.
- Bảo vệ reporter khỏi bị lộ danh tính.
- Bảo vệ target khỏi việc bị public report detail chưa được xác minh.
- Tạo audit/moderation trail cho quyết định xử lý.
- Chuẩn bị cho appeal/kháng nghị.

### 1.3. Product Value

| Value | Description |
|---|---|
| User value | User có công cụ tự bảo vệ khi gặp spam, quấy rối, giả mạo, lừa đảo hoặc nội dung không phù hợp |
| Product value | UEConnect có safety loop rõ ràng thay vì để social platform tự cháy |
| Business value | Sản phẩm đủ nghiêm túc để triển khai trong môi trường trường học |
| Trust value | Verified identity + report + moderation tạo niềm tin cho cộng đồng HCMUE |
| Safety value | Report auto-block, queue xử lý, auto-hide và keyword flag giảm rủi ro lan rộng |
| UX value | User biết phải làm gì khi gặp vấn đề, không cần tự chịu đựng hoặc rời app |

### 1.4. Non-goals

Feature này không làm:

- Không tự động ban user chỉ dựa trên một report đơn lẻ.
- Không public danh tính reporter.
- Không public report detail cho target.
- Không cho user gửi nhiều pending report vào cùng một target.
- Không cho report trở thành công cụ spam/harassment ngược.
- Không thay thế moderation review của admin.
- Không dùng report để xóa audit log hoặc che giấu hành vi vi phạm.
- Không xử lý tranh chấp học thuật/bản quyền phức tạp như tòa án thu nhỏ.
- Không cho user bypass block/messaging bằng cách report rồi nhắn tiếp.
- Không dùng AI moderation như quyết định cuối cùng trong MVP.

### 4.1. Actors

| Actor            |   Can Use? | Notes                                                                |
| ---------------- | ---------: | -------------------------------------------------------------------- |
| Guest            |         No | Cannot report internal content                                       |
| Registered user  |    Limited | Can report verification/evidence issue only if relevant later        |
| Pending user     |    Limited | Can report account/support issue later; not social content           |
| Verified Student |        Yes | Can report visible content/users                                     |
| Verified Alumni  |        Yes | Can report visible content/users                                     |
| Verified Advisor |        Yes | Can report visible content/users                                     |
| Mentor           |        Yes | Can report mentor request/message/profile                            |
| Club Manager     |        Yes | Can report content; may also moderate scoped community if permission |
| Community Owner  |        Yes | Can report/escalate community content                                |
| Moderator        |        Yes | Can view reports if permission                                       |
| Admin            |        Yes | Full report management                                               |
| Suspended user   | Limited/No | Cannot abuse report system unless appeal/support path                |
| Banned user      |         No | Restricted                                                           |

### 4.2. Required Permissions

| Action                       | Required Role                            | Required Permission                 | Scope                |
| ---------------------------- | ---------------------------------------- | ----------------------------------- | -------------------- |
| Report visible profile       | verified active user                     | none                                | target visible       |
| Report visible post          | verified active user                     | none                                | target visible       |
| Report visible comment       | verified active user                     | none                                | target visible       |
| Report conversation message  | conversation participant                 | none                                | own conversation     |
| Report community             | community member/verified user if public | none                                | allowed context      |
| Report community post        | member/allowed viewer                    | none                                | community visibility |
| Report mentor request        | participant                              | none                                | own mentor request   |
| Report evidence abuse        | admin/reviewer/user depending context    | `review_verification` or owner      | scoped               |
| View report queue            | moderator/admin                          | `view_reports` / `moderate_content` | global/scoped        |
| Take report action           | moderator/admin                          | `moderate_content`                  | global/scoped        |
| Escalate report              | moderator/admin                          | `escalate_report`                   | global/scoped        |
| View reporter identity       | admin/moderator                          | `view_reporter_identity`            | restricted           |
| View sensitive report detail | admin/moderator                          | `moderate_content`                  | restricted           |

### 4.3. Access Rules

```txt
- User can report only content/profile/message they are allowed to see or participated in.
- User cannot report the same target again while a pending report exists.
- User cannot report themselves.
- Report action auto-blocks target by default for user safety.
- Auto-block must block message, greeting and Discovery both directions.
- Auto-block does not delete the report.
- Reporter identity is hidden from target.
- Report detail is visible only to authorized moderation/admin roles.
- Suspended/banned users cannot spam report system.
- Admin/moderator action requires permission and reason.
```

### 8.2. Reason Definitions

| Reason                          | Description                                                             | Severity Default  |
| ------------------------------- | ----------------------------------------------------------------------- | ----------------- |
| `spam`                          | Repetitive, irrelevant, bot-like, promotional or disruptive content     | low/medium        |
| `harassment`                    | Targeted attack, bullying, unwanted repeated contact                    | high              |
| `impersonation`                 | Pretending to be another student/alumni/advisor/admin                   | high              |
| `sexual_or_dating_content`      | Dating, sexual, flirtatious, explicit or inappropriate romantic content | high              |
| `copyright_violation`           | Unauthorized sharing of copyrighted resources/materials                 | medium/high       |
| `personal_information_leak`     | Sharing private info like phone, email, MSSV, address, documents        | high/critical     |
| `scam_or_fraud`                 | Phishing, fake offers, money scam, suspicious links                     | critical          |
| `hate_or_offensive_language`    | Insults, slurs, discriminatory or aggressive language                   | high              |
| `politically_sensitive_content` | Sensitive political content inappropriate for school environment        | medium/high       |
| `other`                         | Does not fit above categories                                           | medium by default |

## 9. Report Status Model

### 9.1. Report Status

| Status         | Meaning                               |
| -------------- | ------------------------------------- |
| `submitted`    | Report created                        |
| `queued`       | In moderation queue                   |
| `under_review` | Moderator/admin reviewing             |
| `action_taken` | Moderator took action                 |
| `dismissed`    | No violation or insufficient evidence |
| `duplicate`    | Merged with existing report           |
| `escalated`    | Sent to higher-level admin            |
| `resolved`     | Closed                                |
| `appealed`     | Target submitted appeal               |
| `reopened`     | Reopened after appeal/new evidence    |

### 9.2. Recommended Initial State

```txt
submitted → queued
```

Implementation can create directly as `queued`.

### 9.3. Invalid Transitions

```txt
resolved → under_review without reopen
dismissed → action_taken without reopen
duplicate → action_taken directly
submitted → appealed
```

### 9.4. Report Lifecycle

```txt
submitted
→ queued
→ under_review
→ action_taken
→ resolved

submitted
→ queued
→ under_review
→ dismissed
→ resolved

submitted
→ duplicate

action_taken
→ appealed
→ reopened
→ under_review
```

## 11. Auto-block Rule

### 11.1. Product Decision

```txt
When a user submits a report, the target user is auto-blocked by default.
```

### 11.2. Auto-block Applies To

| Target Type             | Block Target                                                               |
| ----------------------- | -------------------------------------------------------------------------- |
| `user_profile`          | Reported user                                                              |
| `post`                  | Post author                                                                |
| `comment`               | Comment author                                                             |
| `message`               | Message sender                                                             |
| `community`             | No direct user block by default, unless reporting owner/manager separately |
| `community_post`        | Post author                                                                |
| `community_resource`    | Uploader/owner depending context                                           |
| `mentor_request`        | Other participant                                                          |
| `mentor_profile`        | Mentor user                                                                |
| `verification_evidence` | No auto-block by default; admin flow                                       |

### 11.3. Auto-block Effects

```txt
- Remove/deactivate existing connection.
- Prevent new greeting.
- Prevent messaging.
- Hide both users from each other's Discovery.
- Restrict profile access.
- Restrict conversation access or interaction.
```

### 11.4. Auto-block Communication

Report confirmation should say:

```txt
Báo cáo đã được gửi. Để bảo vệ bạn, UEConnect cũng đã chặn tương tác trực tiếp với người này.
```

### 11.5. Auto-block Failure

If report succeeds but auto-block fails:

```txt
- Report must still be saved.
- System logs failure.
- UI should show report submitted and a warning if needed.
- Background job or retry should attempt block again if safe.
```

Report không nên thất bại chỉ vì block phụ trợ bị lỗi. Nhưng nếu block lỗi mà không ai biết thì lại là kiểu “safety bằng niềm tin”, rất thời thượng và rất nguy hiểm.

## 12. Duplicate Report Rule

### 12.1. Rule

```txt
A reporter cannot submit another active/pending report for the same target.
```

### 12.2. Duplicate Definition

Duplicate if:

```txt
same reporter_id
same target_type
same target_id
status in [submitted, queued, under_review, escalated]
```

### 12.3. Duplicate Behavior

If duplicate pending report exists:

```txt
- Do not create new report.
- Show user that their previous report is already being reviewed.
- Optionally allow user to add additional information in future/P2.
```

UI copy:

```txt
Bạn đã báo cáo nội dung này. Báo cáo đang được xem xét.
```

### 12.4. After Resolution

After report is resolved/dismissed:

```txt
- User may report same target again only if new issue occurs.
- System should prevent rapid repeated reports with cooldown if abuse appears.
```

## 13. Auto-hide & Keyword Flagging

### 13.1. Auto-hide Threshold

P1 decision:

```txt
If a content target receives multiple valid reports, system can auto-hide it before moderator review.
```

Recommended MVP/P1 thresholds:

| Target Type        |                                                               Auto-hide Threshold |
| ------------------ | --------------------------------------------------------------------------------: |
| Post               |                                                     3 reports from distinct users |
| Comment            |                                                     3 reports from distinct users |
| Message            | No global auto-hide; restrict reported message in conversation/moderation context |
| Profile            |                                 Do not auto-hide full account; flag high priority |
| Community post     |                                                   3 reports from distinct members |
| Community resource |                                                 2 copyright/personal data reports |
| Mentor request     |                                                            No auto-hide; escalate |
| Community          |                                                            No auto-hide; escalate |

### 13.2. Auto-hide Rules

```txt
- Auto-hide applies only to content, not automatically to user account.
- Auto-hide status must be visible to moderators.
- Auto-hide must create audit/system moderation event.
- Target user may receive safe notification depending policy.
- Auto-hide can be restored by moderator.
```

### 13.3. Keyword Flagging

Keyword flagging can detect:

```txt
- sexual/dating words
- harassment/offensive language
- scam/phishing keywords
- personal information patterns
- politically sensitive keywords
- spam patterns
```

### 13.4. Keyword Flag Behavior

```txt
- Keyword flag does not automatically punish user in MVP.
- Keyword flag increases moderation priority.
- Keyword flag can trigger "needs review" state for risky content.
- Keyword flag can combine with reports to trigger auto-hide faster.
```

### 13.5. Important Safety Rule

```txt
Keyword flagging is an assistive signal, not final judgment.
```

Máy soi từ khóa mà quyết định thay con người thì kiểu gì cũng có ngày phạt nhầm một câu vô tội. Công nghệ rất thích tự tin sai.

### 14.1. Report Post Flow

```txt

| Auto-hide threshold reached            | Hide content and mark urgent                          |                                                                |

## 23. Security & Privacy

### 23.1. Security Rules

```txt
- Report creation must be authorized server-side.
- User cannot report target they are not allowed to access.
- User cannot report self.
- Duplicate active report must be prevented.
- Reporter identity must not be exposed to target.
- Report detail must be admin/moderator-only.
- Auto-block must be server-side, not UI-only.
- Messaging/greeting/discovery must enforce block server-side.
- Report snapshot must be protected.
- Admin actions require permission and reason.
```

### 23.2. Privacy Rules

```txt
- Target user must not see reporter identity.
- Target user must not see raw report description.
- Notification to target must use safe copy.
- Report analytics must not store raw description.
- Message report should expose only limited context to moderator.
- Evidence report must not expose evidence to unauthorized users.
- Report queue should show sensitive data only to permitted roles.
```

### 23.3. Sensitive Data Matrix

| Data                     | Sensitive? | Handling                              |

## 27. Edge Cases

| Edge Case                                             | Expected Handling                                                    |
| ----------------------------------------------------- | -------------------------------------------------------------------- |
| User reports same target twice                        | Show duplicate pending state                                         |
| User reports self                                     | Reject                                                               |
| User reports target after it was deleted              | Allow if recently visible and snapshot exists; otherwise unavailable |
| User reports private content by guessed ID            | Reject                                                               |

  right = actions/status/history
```

### 29.3. PWA Notes

```txt
- Report requires network.
- Offline report should show retry, not fake success.
- If user loses connection mid-report, preserve reason/description locally until retry if feasible.
```

## 30. Performance Requirements

```txt
- Report submission must be fast.
- Duplicate check must use indexed query.
- Moderation queue must be paginated.
- Report grouping must avoid expensive full-table scans.
- Auto-hide threshold must use count/index or cached counter.
- Admin report queue must support filters by status, severity, target type, reason and date.
```

Recommended indexes:

```txt
reports(reporter_id, target_type, target_id, status)
reports(target_type, target_id)
reports(status, severity, created_at)
reports(reason)
reports(moderation_case_id)
moderation_cases(target_type, target_id)
moderation_cases(status, priority)
blocks(blocker_id, blocked_id)
keyword_flags(target_type, target_id)
```

## 36. Final Feature Statement

```txt
Safety Reporting là trust & safety core của UEConnect.
