# Home Feed

## 1. Feature Summary

### 1.1. One-line Description

Home Feed là social surface chính của UEConnect, nơi verified users xem, tạo và tương tác với bài viết theo ngữ cảnh HCMUE, có thuật toán gợi ý, visibility, safety, moderation và media support.

### 1.2. Feature Goal

Feature này tồn tại để:

- Cung cấp trang chính sau khi user vào app.
- Cho user xem bài viết từ cộng đồng HCMUE.
- Cho user tạo bài viết có text và hình ảnh.
- Hỗ trợ tương tác social: xem bài, reaction, comment, save, report, block.
- Hiển thị nội dung theo thuật toán MVP lấy cảm hứng từ Threads/Instagram nhưng phù hợp môi trường trường học.
- Ưu tiên nội dung có ngữ cảnh: cùng khoa, cùng ngành, cùng khóa, cùng community, mentor/career, verified users.
- Hỗ trợ admin/system announcement.
- Không hỗ trợ anonymous post.
- Không biến Feed thành nơi dating, confession, spam hoặc chợ tài liệu vi phạm bản quyền.
- Tạo dữ liệu đầu vào cho notification, analytics, moderation và community.

### 1.3. Product Value

| Value | Description |
|---|---|
| User value | User có nơi cập nhật, chia sẻ, hỏi đáp, tìm bạn học, mentor và cộng đồng |
| Product value | Tạo daily habit và social activity cho UEConnect |
| Business value | Chứng minh app không chỉ là profile/discovery mà là social platform hoàn chỉnh |
| Trust value | Chỉ verified users được đăng và tương tác |
| Safety value | Report/block/moderation gắn ngay vào feed |
| UX value | User vào app là thấy nội dung sống, không bị rơi vào màn hình trống |

### 1.4. Non-goals

Feature này không làm:

- Không hỗ trợ anonymous post.
- Không hỗ trợ confession ẩn danh.
- Không hỗ trợ dating content.
- Không hỗ trợ livestream, reels, stories trong MVP.
- Không biến feed thành marketplace.
- Không biến feed thành job board.
- Không cho unverified/suspended/banned user đăng bài.
- Không public feed ra ngoài UEConnect.
- Không hiển thị nội dung bị hidden/deleted/banned cho normal users.
- Không dùng thuật toán “black box” quá phức tạp trong MVP.
- Không ưu tiên vanity metrics làm user chạy theo like như một cuộc thi vô nghĩa khác của nhân loại.

```txt id="y2bh0a"
- Only active verified users can access Home Feed.
- Pending/rejected/unverified users cannot access feed.
- Suspended users cannot create posts or interact.
- Banned users cannot access feed.
- User can edit/delete only own posts.
- Admin/moderator action requires permission and reason.
- Feed must exclude blocked users both directions.
- Feed must exclude hidden/deleted/banned content.
- Feed must respect post visibility and community visibility.
- Public internet users cannot view feed.
```

## 8. Feed Ranking / Algorithm MVP

### 8.1. Algorithm Goal

Feed algorithm should prioritize:

```txt id="nam45c"
- relevance
- recency
- trust
- shared HCMUE context
- engagement quality
- safety
- diversity
```

Not prioritize:

```txt id="b1lucw"
- outrage
- spam engagement
- dating-like attention
- pure popularity
- clickbait
- unsafe content
```

### 8.2. MVP Ranking Inputs

| Signal                   | Meaning                     | Weight Direction |
| ------------------------ | --------------------------- | ---------------- |
| Recency                  | Newer posts higher          | High             |
| Same faculty             | Same faculty as viewer      | Medium           |
| Same major               | Same major as viewer        | Medium/High      |
| Same cohort              | Same cohort as viewer       | Medium           |
| Same community           | Shared community/club       | Medium           |
| Connected user           | Existing connection         | Medium/High      |
| Mentor/advisor relevance | Mentor/career context       | Medium           |
| Engagement count         | Comments/reactions/saves    | Medium           |
| Report count             | More reports reduce ranking | Negative         |
| Hidden/moderated         | Excluded                    | Hard filter      |
| Block relationship       | Excluded                    | Hard filter      |
| Author status            | Only active verified users  | Hard filter      |
| Post visibility          | Must match viewer           | Hard filter      |

### 8.3. MVP Feed Scoring Direction

Conceptual scoring:

```txt id="a0lfdb"
score =
  recency_score
+ shared_context_score
+ connection_score
+ engagement_score
+ content_type_boost
- report_penalty
- repetition_penalty
```

Hard filters happen before scoring:

```txt id="vgm77a"
- viewer must be verified active
- post must be active/visible
- author must be active verified
- not blocked either direction
- post visibility must allow viewer
- not hidden/deleted/moderated
```

### 8.4. Fallback Feed

If algorithm cannot produce enough posts:

```txt id="vu8am5"
fallback to latest verified posts that pass hard filters
```

### 8.5. Ranking Fairness Rules

```txt id="57dxvp"
- Do not show only one author repeatedly.
- Avoid duplicate posts in one page.
- Ensure admin announcement can be pinned without flooding.
- Do not expose why a specific private context boosted a post.
- Do not leak hidden profile/private community membership in ranking explanation.
```

### 8.6. Implementation Level

MVP does not need ML.

Recommended MVP:

```txt id="ts73d0"
SQL query + weighted scoring + pagination + fallback latest
```

Yes, boring SQL. Boring is excellent when the alternative is pretending you built Instagram’s recommendation system between lunch and a deadline.

## 11. Main User Flow

### 11.1. View Feed Flow

```txt id="kq0bbv"
User opens /app/home
→ System checks auth/account/verification/profile completion
→ System loads feed candidates
→ Apply hard filters
→ Apply ranking score
→ Return paginated feed
→ UI renders post cards
→ User scrolls for next page
```

### 11.2. Create Post Flow

```txt id="yx2vnk"
User clicks create post
→ Composer opens
→ User enters content
→ User optionally uploads images
→ User selects visibility if available
→ System validates content/media/permission
→ Post is created as active
→ Feed updates or redirects to post detail
→ Analytics event recorded
```

### 11.3. React / Save Flow

```txt id="mr80la"
User taps reaction/save
→ System checks account active + post visible
→ System creates/updates reaction/save
→ UI updates optimistically or after response
```

### 11.4. Report Post Flow

```txt id="l1lfn0"
User opens post action menu
→ Selects Report
→ Chooses reason
→ Optional description
→ System creates report
→ System auto-blocks author if safety flow says so
→ Report enters moderation queue
→ UI confirms report submitted
```

### 11.5. Block Author Flow

```txt id="n4twag"
User opens post action menu
→ Selects Block
→ Confirms
→ System creates block
→ Existing connection is removed/deactivated
→ Author posts disappear from feed
→ Messaging/greeting blocked
```

## 13. Alternative & Failure Flows

| Flow                                     | Condition                          | Expected Behavior             |
| ---------------------------------------- | ---------------------------------- | ----------------------------- |
| Feed empty                               | New user/no posts                  | Show helpful empty state      |
| Feed query fails                         | Server/network error               | Show retry                    |
| User offline                             | PWA/offline                        | Show offline state            |
| Post deleted while visible               | User opens detail                  | Show unavailable              |
| Post hidden by moderation                | User refreshes                     | Show placeholder/unavailable  |
| User blocked author                      | Cached post still visible          | Action recheck removes/denies |
| User suspended while composing           | Submit rejected                    |                               |
| Image upload succeeds, post submit fails | Keep draft or cleanup orphan media |                               |
| Algorithm returns duplicates             | Deduplicate before response        |                               |
| Announcement expired                     | Do not show                        |                               |
| Visibility changed                       | Recheck on every request           |                               |
| User tries to react to hidden post       | Server rejects                     |                               |

## 20. Security & Privacy

### 20.1. Security Rules

```txt id="mmb4r2"
- Server must authorize every feed/action request.
- Never rely on hidden UI for create/react/report.
- Feed query must exclude blocked users both directions.
- Feed query must exclude private/invisible content.
- Post visibility must be checked on post detail and actions.
- Suspended/banned users cannot create or interact.
- Media IDs must belong to user before attaching to post.
- Moderated/deleted posts must not expose body/media to normal users.
- Admin announcement creation requires permission.
```

### 20.2. Privacy Rules

```txt id="zxbrvv"
- Feed is verified-only.
- No public internet feed in MVP.
- Do not expose private profile fields in feed card.
- Do not reveal why a post was ranked using private context.
- Do not show posts from blocked users.
- Do not show hidden community posts to non-members.
- Analytics must not store raw post body.
```

### 20.3. Sensitive Data

| Data                         | Sensitive? | Handling                                                  |
| ---------------------------- | ---------: | --------------------------------------------------------- |
| Post body                    |   Moderate | Visible by post visibility; do not store raw in analytics |
| Post image                   |   Moderate | Follows post visibility                                   |
| Author profile context       |   Moderate | Use public-safe fields only                               |
| Report reason/detail         |  Sensitive | Reporter/admin only                                       |
| Ranking signals              |   Internal | Do not expose raw                                         |
| Block relationship           |   Moderate | Enforced server-side                                      |
| Admin announcement targeting |   Moderate | Admin only                                                |

## 24. Edge Cases

| ---------------------------------------- | ---------------------------------------------------- |
| New user has no relevant posts           | Show empty state + actions                           |

| Algorithm duplicates posts               | Deduplicate before response                          |
| Post deleted while visible               | Post detail shows unavailable                        |
| Post hidden after feed load              | Action requests are rejected                         |
| Blocked author remains in cached feed    | Server rejects interactions and next refresh removes |

| User attaches media not owned by them    | Reject                                               |

| Announcement target changes              | Recalculate visibility                               |

| Post body contains banned content        | Keyword flag/future moderation                       |
| Copyright resource shared in post        | Report/moderation flow                               |
