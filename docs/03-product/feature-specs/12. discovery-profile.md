# Discovery Profile

## 1. Feature Summary

### 1.1. One-line Description

Discovery Profile cho phép verified users khám phá UEers phù hợp trong HCMUE dựa trên hồ sơ, khoa/ngành/khóa, mục tiêu kết nối, sở thích học tập, community, mentor context và privacy settings.

### 1.2. Feature Goal

Feature này tồn tại để:

- Giúp user khám phá người phù hợp trong cộng đồng HCMUE.
- Tạo cảm giác mỗi UEer là một hồ sơ có giá trị, có bối cảnh, không phải một avatar vô danh.
- Hỗ trợ kết nối học tập, mentor, cùng khoa, cùng ngành, cùng khóa, cùng community.
- Tận dụng interaction pattern nhanh, trực quan, dễ dùng như Tinder nhưng loại bỏ hoàn toàn dating language/vibe.
- Cho user gửi Greeting thay vì “match”.
- Cho user xem profile context đủ tin cậy trước khi gửi lời chào.
- Tôn trọng privacy, block, account status, discovery visibility.
- Làm nền cho Messaging: chỉ khi Greeting/Connection được accepted mới có conversation chính thức.
- Giữ trải nghiệm thân thiện, trẻ trung, nhưng vẫn trusted và enterprise-ready.

### 1.3. Product Value

| Value | Description |
|---|---|
| User value | User tìm được bạn cùng học, bạn cùng khoa/ngành, mentor, alumni hoặc cộng đồng phù hợp |
| Product value | Tạo interaction loop chính ngoài Feed |
| Business value | UEConnect khác portal trường vì có discovery social layer |
| Trust value | Chỉ hiển thị verified users, có badge, context và privacy rule |
| Safety value | Block/report/filter giúp tránh unwanted interaction |
| UX value | Khám phá nhanh, rõ, mobile-first, không dating hóa sản phẩm |

### 1.4. Non-goals

Feature này không làm:

- Không dùng từ match.
- Không dùng từ swipe trong UI copy.
- Không dùng crush, hot, dating, hẹn hò, ghép đôi.
- Không xếp hạng người theo độ hấp dẫn.
- Không có public attractiveness score.
- Không ép user xuất hiện trong Discovery nếu họ tắt visibility.
- Không hiển thị full MSSV, email, phone, evidence.
- Không cho unverified/suspended/banned users xuất hiện.
- Không cho blocked users thấy nhau.
- Không mở Discovery ra ngoài HCMUE.
- Không biến profile card thành CV dài như LinkedIn.
- Không biến card thành dating profile trá hình, vì rõ ràng nhân loại chưa đủ rắc rối.

## 7. Discovery Strategy

### 7.1. Product Direction

Discovery should feel:

```txt
trusted
student-first
minimal
human
HCMUE-contextual
friendly
quick
safe
```

Discovery should not feel:

```txt
dating
thirsty
premium luxury
childish
portal-like
AI-generated
over-colorful
```

### 7.2. Interaction Direction

Inspired by Tinder only in:

```txt
- card-based browsing
- quick decision pattern
- gesture support on mobile
- primary action + secondary action
```

Not copied from Tinder in:

```txt
- language
- dating framing
- match mechanic
- attractiveness ranking
- hot/crush/swipe UI copy
- romantic visual tone
```

### 7.3. UX Principle

```txt
Discovery is for connection context, not judgment.
```

Card should answer:

```txt
- Người này là ai?
- Có phải UEer verified không?
- Có điểm chung nào với tôi?
- Người này muốn kết nối vì điều gì?
- Tôi có thể gửi lời chào phù hợp không?
```

## 8. Discovery Candidate Rules

### 8.1. Candidate Eligibility

A user can appear in Discovery only if:

```txt
account_status = active
verification_status = approved
profile_completion_status = basic_complete or higher
discovery_visibility = enabled
profile not hidden by moderation
not blocked either direction
not current viewer
```

### 8.2. Candidate Exclusion

Exclude:

```txt
- current user
- unverified users
- pending/rejected users
- profile incomplete users
- suspended users
- banned users
- deleted/deactivated users
- users hidden by moderation
- users who disabled Discovery
- users blocked by viewer
- users who blocked viewer
- users with active unresolved severe safety restriction
```

### 8.3. Repeated Candidate Control

```txt
- If viewer passes/skips a profile, do not show again immediately.
- Skipped profile can reappear after a cooldown if no block/report exists.
- If Greeting is sent, do not show same candidate as normal Discovery card until status changes.
- If Greeting is declined, do not allow new Greeting to same target unless product policy allows after long cooldown.
```

Recommended MVP:

```txt
skip cooldown = 7 days
declined greeting = do not allow another greeting to same target
pending greeting = hide from Discovery normal stack
accepted greeting = connection created, hide from Discovery normal stack
blocked/reported = never show while block active
```

## 9. Discovery Ranking MVP

### 9.1. Ranking Goal

Discovery ranking should prioritize:

```txt
- shared HCMUE context
- relevance to connection goals
- trust
- profile completeness
- mentor/community fit
- recency/activity
- diversity
- safety
```

Not prioritize:

```txt
- attractiveness
- gender/romantic assumptions
- popularity-only ranking
- vanity metrics
- spam activity
```

### 9.2. Ranking Inputs

| Signal                    | Meaning                                      | Direction                    |
| ------------------------- | -------------------------------------------- | ---------------------------- |
| Same faculty              | Candidate has same faculty                   | Positive                     |
| Same major                | Candidate has same major                     | Strong positive              |
| Same cohort               | Candidate has same cohort/year               | Positive                     |
| Same class code           | Candidate shares class if visible            | Positive, privacy-aware      |
| Shared learning interests | Overlap in interests                         | Positive                     |
| Shared career interests   | Overlap in career goals                      | Positive                     |
| Shared community          | Same community/club                          | Positive                     |
| Connection goals match    | Candidate goals match viewer goals           | Strong positive              |
| Mentor availability       | Candidate can mentor and viewer wants mentor | Positive                     |
| Alumni/advisor context    | Useful for mentor/career                     | Positive                     |
| Profile completeness      | More complete safe profile                   | Positive                     |
| Recent activity           | Recently active user                         | Mild positive                |
| Already connected         | Exclude or deprioritize                      | Exclude from discovery stack |
| Pending greeting          | Exclude                                      |                              |
| Report count              | Negative                                     |                              |
| Moderation status         | Hard exclude if hidden/restricted            |                              |
| Block relation            | Hard exclude                                 |                              |

### 9.3. MVP Scoring Direction

Conceptual scoring:

```txt
score =
  shared_hcmue_context_score
+ interest_overlap_score
+ connection_goal_score
+ mentor_context_score
+ community_overlap_score
+ profile_quality_score
+ recency_score
- report_risk_penalty
- repetition_penalty
```

Hard filters run before scoring:

```txt
active verified profile
privacy allowed
discovery enabled
not blocked
not suspended/banned
not hidden by moderation
not pending/accepted/declined greeting according to policy
```

### 9.4. Ranking Explanation

Discovery card may show safe reasons:

```txt
- Cùng khoa
- Cùng ngành
- Cùng khóa
- Cùng quan tâm: Thiết kế UI
- Cùng cộng đồng
- Có thể hỗ trợ mentor
```

Do not show creepy/private reasons:

```txt
- Người này vừa hoạt động 2 phút trước
- Người này xem profile bạn
- Người này bị nhiều người bỏ qua
- Người này có điểm tương hợp 98%
```

Ứng dụng trường học không cần biến thành phòng thí nghiệm thao túng hành vi. Đủ app ngoài kia làm trò đó rồi.

Because laptop users should not have to do weird mouse rituals just to meet a classmate.

## 13. Greeting Integration

### 13.1. Greeting Meaning

Greeting is a polite connection request with optional message.

It replaces dating-style match.

### 13.2. Greeting From Discovery

Flow:

```txt
User sees candidate card
→ User clicks Gửi lời chào
→ Greeting composer opens
→ User writes optional message or uses suggested intro
→ System validates length/status/access
→ Greeting is sent
→ Candidate receives notification
→ Candidate can accept/decline
```

### 13.3. Greeting Rules

```txt
- Cannot send Greeting to self.
- Cannot send Greeting to blocked user.
- Cannot send Greeting if target blocked viewer.
- Cannot send duplicate pending Greeting.
- Cannot send again if previous Greeting declined, according to product decision.
- Cannot send to suspended/banned/hidden user.
- Greeting message has max length.
- Greeting must not contain prohibited content.
```

Details belong to `greeting-connection.md`.

### 13.4. Discovery After Greeting

After Greeting sent:

```txt
- Remove candidate from current Discovery stack.
- Show success state.
- Do not show same candidate again while Greeting is pending.
```

## 14. Safety & Privacy Rules

### 14.1. Privacy Enforcement

Discovery must enforce:

```txt
profile_visibility
discovery_visibility
field-level visibility
block relationship
account status
moderation status
community privacy
mentor availability
```

### 14.2. Report From Discovery

User can report:

```txt
- suspicious profile
- impersonation
- harassment
- sexual/dating content
- scam/fraud
- personal information leak
- other
```

Report from Discovery should:

```txt
- create report
- auto-block target
- remove profile from Discovery
- show safe confirmation
```

### 14.3. Block From Discovery

Block should:

```txt
- create block relationship
- remove candidate from Discovery
- prevent Greeting
- prevent Messaging
- remove/deactivate existing connection if any
- hide both users from each other
```

### 14.4. Moderated Profile

If profile is hidden by moderation:

```txt
- Do not include in Discovery.
- If user opens old URL, show unavailable/restricted state.
```

## 15. Main User Flow

### 15.1. View Discovery Flow

```txt
User opens /app/discovery
→ System checks auth/account/verification/profile completion
→ System loads discovery candidates
→ Apply hard filters
→ Apply ranking
→ Return first candidate/card batch
→ UI renders card
```

### 15.2. Pass Profile Flow

```txt
User clicks Bỏ qua
→ System records pass event
→ Candidate removed from current stack
→ Next candidate shown
```

### 15.3. Send Greeting Flow

```txt
User clicks Gửi lời chào
→ Greeting composer opens
→ User writes message
→ System validates target and duplicate state
→ Greeting sent
→ Candidate removed from Discovery
→ Notification sent to target
```

### 15.4. View Profile Flow

```txt
User clicks Xem hồ sơ
→ System checks profile access
→ Opens public profile detail
→ User can send Greeting/report/block from profile
```

### 15.5. Report Profile Flow

```txt
User opens More menu
→ Selects Báo cáo
→ Chooses reason
→ Optional description
→ System creates report
→ System auto-blocks target
→ Target removed from Discovery
→ Report enters moderation queue
```

### 15.6. Block Profile Flow

```txt
User opens More menu
→ Selects Chặn
→ Confirms
→ System creates block
→ Target removed from Discovery
→ Greeting/messaging blocked
```

## 17. Alternative & Failure Flows

| Flow                                     | Condition                               | Expected Behavior |
| ---------------------------------------- | --------------------------------------- | ----------------- |
| No candidates                            | All filtered/exhausted                  | Show empty state  |
| Candidate becomes suspended while shown  | Action recheck rejects and removes card |                   |
| Candidate disables Discovery while shown | Action recheck rejects and removes card |                   |
| Viewer becomes suspended                 | Redirect restricted                     |                   |
| Greeting duplicate                       | Show existing pending state             |                   |
| Greeting declined before card action     | Prevent re-send                         |                   |
| Block exists after cached card loaded    | Action recheck removes/denies           |                   |
| Report succeeds but block fails          | Report saved, block retry/log           |                   |
| Ranking returns duplicates               | Deduplicate                             |                   |
| Profile image missing                    | Show fallback avatar                    |                   |

## 24. Security & Privacy

### 24.1. Security Rules

```txt
- Discovery candidates are resolved server-side.
- Client cannot request hidden/suspended/private users by ID and bypass checks.
- Every Greeting/report/block action rechecks candidate access.
- Sensitive profile fields must not be included in candidate payload.
- Block relationship must be enforced both directions.
- Suspended/banned users cannot appear or interact.
- Discovery visibility must be enforced in query, not only UI.
```

### 24.2. Privacy Rules

```txt
- Full MSSV never appears.
- Email never appears by default.
- Phone never appears by default.
- Evidence never appears.
- Admin notes never appear.
- Report history never appears.
- Private profile fields are excluded.
- Private community memberships are excluded.
- Candidate view tracking is not shown to users in MVP.
```

### 24.3. Sensitive Data Matrix

| Data                    | Sensitive? | Handling                      |
| ----------------------- | ---------: | ----------------------------- |
| MSSV                    |        Yes | Never shown                   |
| Email/phone             |        Yes | Never on card                 |
| Evidence                |        Yes | Never shown                   |
| Discovery visibility    |   Moderate | Owner/admin only              |
| Block relation          |   Moderate | Enforced, not public          |
| Candidate ranking score |   Internal | Never shown                   |
| Candidate view tracking |   Moderate | Internal analytics only       |
| Shared context          |   Moderate | Show only public-safe context |
