## 4. Realtime Architecture Decision

### 4.1. Business Requirement

Messaging cß║¦n:

```txt
- Realtime message delivery
- Typing indicator
- Read receipts
- Browser push notification
- PWA-friendly behavior
- Laravel-native integration
- Scalable enough for school social platform
- Maintainable for student/final project but enterprise-ready
```

### 4.2. Options Comparison

| Option                | Pros                                                                                           | Cons                                                                                     | Fit                        |
| --------------------- | ---------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- | -------------------------- |
| Laravel Reverb + Echo | Laravel-native, self-hosted, good with broadcasting, no vendor lock-in, fits PWA/Laravel stack | Need server/runtime setup, scaling requires planning, WebSocket infra must be maintained | Best default for UEConnect |
| Pusher                | Mature, easy Laravel Echo integration, fast setup                                              | Vendor cost, external dependency, less control                                           | Good if muß╗æn launch nhanh  |
| Ably                  | Very robust realtime infra, strong reliability, presence, global scale                         | Cost/vendor dependency, more external service complexity                                 | Strong enterprise option   |
| Soketi                | Pusher-compatible, self-hosted, fast                                                           | Extra service to operate, not Laravel-native like Reverb                                 | Good alternative           |
| Polling only          | Simple                                                                                         | Not true realtime, bad UX, wasteful                                                      | Not acceptable             |
| Server-Sent Events    | Simpler than WS for one-way updates                                                            | Weak for bidirectional typing/presence                                                   | Not enough alone           |

### 4.3. Recommended Choice

Recommended for UEConnect:

```txt
Primary MVP/P1:
Laravel Reverb + Laravel Echo + private/presence channels

Fallback:
HTTP polling for failed WebSocket connection

Future enterprise alternative:
Ably or Pusher if deployment environment cannot reliably host WebSocket server
```

### 4.4. Why Laravel Reverb

```txt
- Fits Laravel ecosystem.
- Works with Laravel Broadcasting.
- Works with Echo on frontend.
- Self-hosted, no vendor lock-in.
- Good enough for HCMUE-only platform scale.
- Can be deployed beside Laravel app with queue workers.
- Easier to explain in portfolio/case study as intentional architecture.
```

### 4.5. Realtime Channel Strategy

| Channel Type          | Use                                                    |
| --------------------- | ------------------------------------------------------ |
| Private channel       | Direct conversation messages                           |
| Presence channel      | Typing indicator, online participants, future presence |
| Private user channel  | Notifications for specific user                        |
| Admin private channel | Critical admin/moderation notifications                |

Recommended channels:

```txt
private-conversation.{conversationId}
presence-conversation.{conversationId}
private-user.{userId}
private-admin.{adminId}
```

### 4.6. Realtime Events

| Event                        | Channel                      | Payload                      |
| ---------------------------- | ---------------------------- | ---------------------------- |
| `MessageSent`                | `private-conversation.{id}`  | message summary              |
| `MessageEdited`              | `private-conversation.{id}`  | message id + edited body     |
| `MessageDeleted`             | `private-conversation.{id}`  | message id + delete status   |
| `MessageRead`                | `private-conversation.{id}`  | user id + message id/read_at |
| `TypingStarted`              | `presence-conversation.{id}` | user id                      |
| `TypingStopped`              | `presence-conversation.{id}` | user id                      |
| `ConversationRestricted`     | `private-conversation.{id}`  | reason category              |
| `MessageReported`            | admin/moderation channel     | report metadata              |
| `ConversationCreated`        | `private-user.{userId}`      | conversation summary         |
| `MessageNotificationCreated` | `private-user.{userId}`      | notification summary         |

### 4.7. Realtime Fallback

If WebSocket fails:

```txt
- Show connection indicator.
- Continue allowing message send through HTTP.
- Poll conversation every 10-15 seconds while open.
- Poll conversation list every 30-60 seconds.
- Reconnect WebSocket automatically.
```

### 4.8. Realtime Reliability Rules

```txt
- Message is persisted to database first.
- Realtime event is broadcast after commit.
- Client must not treat realtime event as source of truth.
- Client should reconcile by message_id/client_message_id.
- Duplicate events must be ignored.
- Failed broadcast must not lose persisted message.
- Sender should use optimistic UI only with pending state.
```

| Banned user      |                 No | Cannot access                                        |

### 5.2. Required Permissions

| Action                        | Required Role                | Required Permission      | Scope          |
| ----------------------------- | ---------------------------- | ------------------------ | -------------- |
| View conversation list        | verified active user         | none                     | own            |
| View conversation             | participant                  | none                     | conversation   |
| Send message                  | participant + allowed status | none                     | conversation   |
| Send message request          | verified active user         | none                     | allowed target |
| Accept message request        | request receiver             | none                     | own            |
| Decline/delete request        | request receiver             | none                     | own            |
| Edit own message              | message sender               | none                     | own            |
| Delete own message            | message sender               | none                     | own            |
| Read message                  | conversation participant     | none                     | conversation   |
| Send attachment               | conversation participant     | none                     | conversation   |
| Report message                | conversation participant     | none                     | conversation   |
| Block user                    | verified active user         | none                     | own safety     |
| View reported message context | moderator/admin              | `review_message_reports` | restricted     |
| Moderate message              | moderator/admin              | `moderate_content`       | global/scoped  |

### 5.3. Access Rules

```txt
- User can only view conversations where they are participant.
- User can only send message in active conversation.
- Direct active conversation requires active connection, accepted greeting, accepted mentor request, or accepted message request.
- Blocked users cannot message each other.
- Suspended/banned users cannot send messages.
- Deleted/deactivated users cannot send messages.
- Message attachments follow conversation access.
- Message edit/delete only allowed by sender.
- Reported message context visible only to authorized moderator/admin.
```

## 10. Realtime UX Rules

### 10.1. Sending Message UX

Flow:

```txt
User types message
ŌåÆ Clicks send
ŌåÆ Client creates temporary message with client_message_id
ŌåÆ UI shows sending state
ŌåÆ Server validates and stores message
ŌåÆ Server returns official message_id
ŌåÆ UI replaces temp message with official message
ŌåÆ Server broadcasts MessageSent after DB commit
ŌåÆ Other participant receives realtime event
```

### 10.2. Duplicate Prevention

Use:

```txt
client_message_id
sender_id
conversation_id
```

Rule:

```txt
If same client_message_id is submitted twice by same sender in same conversation, return existing message instead of creating duplicate.
```

Because double-clicking send should not create two identical messages, unless your app is secretly a spam cannon.

### 10.3. Typing Indicator

Rules:

```txt
- Send TypingStarted when user begins typing.
- Send TypingStopped after 3-5 seconds inactivity or input empty.
- Do not persist typing events in DB.
- Do not send typing events to blocked/restricted conversations.
- Throttle typing events.
```

Recommended timing:

```txt
typing started throttle = 2 seconds
typing stopped timeout = 5 seconds
```

### 10.4. Read Receipts

Rules:

```txt
- Mark messages as read when conversation is opened and messages are visible.
- Store read_at per participant/conversation or per message receipt.
- For 1:1 chat, conversation-level last_read_message_id is enough.
- Broadcast MessageRead after update.
```

MVP recommended:

```txt
conversation_participants.last_read_message_id
conversation_participants.last_read_at
```

Avoid per-message read receipts unless needed. Yes, less granular. Also less database suffering.

### 10.5. Online/Presence

P1/P2.

MVP can support presence only inside active conversation for typing indicator.

Do not show global online status in MVP unless privacy policy is ready.

## 16. Security & Privacy

### 16.1. Security Rules

```txt
- Every conversation/message route requires participant authorization.
- Message send must check account status and block status server-side.
- Realtime channel subscription must be private/presence authorized.
- Client cannot subscribe to conversation channel without participant permission.
- Message attachments served through protected route.
- Raw storage path never exposed.
- Message edit/delete only by sender.
- Moderator cannot view private messages unless message is reported and permission allows.
- Message body must be escaped/sanitized.
- Realtime events must not include sensitive data beyond what participants can access.
```

### 16.2. Privacy Rules

```txt
- Private messages visible only to participants.
- Reported message context visible only to authorized moderator/admin.
- Browser push should avoid full sensitive message preview by default.
- Analytics must not store message body.
- Attachment filename may be sensitive; avoid showing in notification.
- Blocked conversation should not reveal new activity.
- Deleted/hidden messages should not expose original body to normal users.
```

### 16.3. Sensitive Data Matrix

| Data                      | Sensitive? | Handling                             |
| ------------------------- | ---------: | ------------------------------------ |
| Message body              |        Yes | Participants only, no analytics body |
| Attachments               |        Yes | Protected access                     |
| Reported message snapshot |        Yes | Moderator/admin restricted           |
| Read receipts             |   Moderate | Participants only                    |
| Typing indicator          |   Moderate | Conversation participants only       |
| Conversation membership   |   Moderate | Participants/admin if needed         |
| Notification preview      |   Moderate | Safe preview only                    |
| Attachment path           |        Yes | Internal only                        |

## 19. Edge Cases

| Edge Case                                        | Expected Handling                                              |
| ------------------------------------------------ | -------------------------------------------------------------- |
| User sends message twice by double-click         | client_message_id prevents duplicate                           |
| WebSocket event duplicated                       | Client deduplicates by message_id                              |
| Message persisted but broadcast fails            | Message still visible after fetch/poll                         |
| Broadcast received before HTTP response          | Reconcile temp message with official id                        |
| User blocked while typing                        | Send rejected, conversation restricted                         |
| User blocked after message sent                  | Existing message remains unless moderated; future send blocked |
| Receiver suspended                               | New messages blocked                                           |
| Sender suspended while composer open             | Submit rejected                                                |
| Conversation opened by non-participant           | 403                                                            |
| Attachment uploaded but message fails            | Orphan cleanup job                                             |
| Message deleted while receiver views it          | Realtime event replaces with placeholder                       |
| Message edited after recipient loads old version | Realtime event updates                                         |
| User reports deleted message                     | Allow if recently visible/snapshot exists                      |
| Read receipt points to invalid message           | Reject                                                         |
| Typing indicator stuck                           | Client timeout clears after 5 seconds                          |
| WebSocket disconnected                           | Fallback polling                                               |
| Browser push denied                              | In-app notification still works                                |
| Moderator tries to view unreported message       | Deny                                                           |
| Community suspended                              | Community chat restricted                                      |
| Mentor request cancelled                         | Mentor conversation restricted if policy says                  |
