# Media Upload

## 1. Feature Summary

### 1.1. One-line Description

Media Upload cho phép UEConnect xử lý upload, lưu trữ, preview, kiểm soát quyền truy cập và quản lý lifecycle của file/media trong verification, profile, post, message, community và resource library.

### 1.2. Feature Goal

Feature này tồn tại để:

- Cho user upload evidence khi xác thực tài khoản.
- Cho user upload avatar bắt buộc khi hoàn thiện profile.
- Cho user upload ảnh trong bài viết.
- Cho user gửi ảnh/file trong messaging.
- Cho community/club upload resource có kiểm soát bản quyền.
- Đảm bảo file nhạy cảm như evidence luôn private.
- Đảm bảo file public như avatar/post image được validate, optimize và serve đúng cách.
- Đảm bảo upload có security, privacy, moderation và audit direction rõ ràng.
- Tránh việc hệ thống trở thành “Google Drive mini không kiểm soát”, thứ mà chưa ai cần thêm trong đời.

### 1.3. Product Value

| Value | Description |
|---|---|
| User value | User có thể xác thực, cá nhân hóa hồ sơ, chia sẻ hình ảnh/tài liệu đúng ngữ cảnh |
| Product value | Profile, Feed, Messaging, Community và Verification đầy đủ hơn |
| Business value | Tạo nền kỹ thuật đủ enterprise cho PWA thật |
| Trust value | Evidence được bảo vệ, không public |
| Safety value | File upload được validate, scan-ready, moderation-ready |
| UX value | Upload rõ ràng, có trạng thái progress/error/retry |

### 1.4. Non-goals

Feature này không làm:

- Không public evidence.
- Không cho upload file không kiểm soát.
- Không cho upload executable/script.
- Không cho upload tài liệu vi phạm bản quyền.
- Không làm cloud drive cá nhân.
- Không cho anonymous file sharing.
- Không cho user gửi file cho người chưa có permission.
- Không lưu file nhạy cảm vào analytics.
- Không expose direct storage path.
- Không tin MIME type từ client, vì client nói dối chuyên nghiệp hơn nhiều người tưởng.

### 4.1. Actors

| Actor            |   Can Use? | Notes                                            |

| Guest            |         No | Cannot upload                                    |
| Registered user  |        Yes | Can upload verification evidence                 |

| Verified Student |        Yes | Avatar, posts, messages, community if allowed    |
| Verified Alumni  |        Yes | Same as verified user                            |
| Verified Advisor |        Yes | Same as verified user                            |
| Mentor           |        Yes | Mentor profile/resources if enabled              |
| Club Manager     |        Yes | Community resources/posts if scoped permission   |
| Community Owner  |        Yes | Community resources/settings                     |
| Moderator        |        Yes | Can review flagged media                         |
| Admin            |        Yes | Can preview evidence/admin media with permission |
| Suspended user   | Limited/No | Cannot upload social media                       |
| Banned user      |         No | Cannot upload                                    |

### 4.2. Required Permissions

| Action                     | Required Role                 | Required Permission                         | Scope            |
| -------------------------- | ----------------------------- | ------------------------------------------- | ---------------- |
| Upload evidence            | authenticated unverified user | none                                        | own verification |
| Preview own evidence       | owner                         | none                                        | own              |
| Admin preview evidence     | admin/reviewer                | `review_verification`                       | global           |
| Upload avatar              | verified active user          | none                                        | own              |
| Upload post image          | verified active user          | none                                        | own post         |
| Upload message attachment  | verified active user          | conversation participant                    | conversation     |
| Upload community resource  | club manager/owner/admin      | `manage_club` / `manage_community_resource` | scoped community |
| Preview community resource | member/allowed user           | membership/visibility                       | community        |
| Moderate media             | moderator/admin               | `moderate_content`                          | global/scoped    |
| Delete media by moderation | moderator/admin               | `moderate_content`                          | global/scoped    |

### 4.3. Access Rules

```txt id="czk0yq"
- Evidence can only be accessed by owner and authorized admin/reviewer.
- Avatar is public to verified users and can be cached.
- Post image visibility follows post visibility.
- Message attachment visibility follows conversation participant permission.
- Community resource visibility follows community membership and resource policy.
- Suspended/banned users cannot upload new social media.
- Blocked users cannot access new message media from restricted conversations.
- Direct storage paths must never be exposed.
- Every file access must be authorized through controller/signed route where needed.
```

## 7. Storage Strategy

### 7.1. Recommended Hybrid Storage Strategy

UEConnect should use a hybrid storage strategy.

```txt id="t1spve"
P0 MVP:
- Laravel local storage
- private disk for evidence and sensitive files
- public disk for avatar/post images if appropriate
- protected controller for private previews

P1/P2 Enterprise-ready:
- S3-compatible object storage for scalable files
- signed URLs for temporary access
- CDN for public optimized images
- private bucket for evidence/message attachments/resources
```

### 7.2. Why Hybrid

| Media Type             | Storage Type                           | Reason                   |
| ---------------------- | -------------------------------------- | ------------------------ |
| Evidence               | Private                                | Sensitive identity proof |
| Avatar                 | Public or protected public-to-verified | Needed across app        |
| Post image             | Public-to-verified / controlled        | Follows post visibility  |
| Message attachment     | Private                                | Conversation-sensitive   |
| Community resource     | Private/protected                      | Membership/copyright     |
| Admin evidence preview | Protected route                        | Permission required      |

### 7.3. MVP Disk Direction

Recommended Laravel disks:

```txt id="661gxz"
storage/app/private/evidence
storage/app/private/messages
storage/app/private/community-resources
storage/app/public/avatars
storage/app/public/post-images
```

Do not expose:

```txt id="bz17xv"
storage/app/private/*
```

### 7.4. Future Object Storage Direction

Future S3-compatible layout:

```txt id="71x677"
private/evidence/{user_id}/{request_id}/...
private/messages/{conversation_id}/...
private/community-resources/{community_id}/...
public/avatars/{user_id}/...
public/post-images/{post_id}/...
```

### 10.2. Rate Limits

Recommended MVP/P1:

```txt id="9r7qnz"
- Evidence resubmission follows verification rate limit.
- Avatar upload limited to prevent spam.
- Post image upload tied to post creation limits.
- Message attachment upload tied to messaging rate limits.
- Community resource upload tied to community manager permission.
```

Suggested starting points:

```txt id="7rqwdr"
avatar: 20 uploads/day/user
post images: 50 files/day/user
message attachments: 100 files/day/user
community resources: 30 files/day/community
evidence: max 3 active items/request, resubmission limited by verification flow
```

These are product-safe defaults, not sacred scripture chiseled into stone.

## 11. Main User Flow

### 11.1. Evidence Upload Flow

```txt id="hn1ro9"
User opens verification evidence step
→ User selects file or adds evidence URL
→ User adds note per evidence item
→ System validates type, size and item count
→ System stores file in private storage
→ System creates media/evidence metadata
→ User sees uploaded item
→ User submits verification
```

### 11.2. Avatar Upload Flow

```txt id="lmgj3k"
Verified user opens profile setup
→ User selects avatar image
→ System validates image type and size
→ Optional crop/preview
→ System stores avatar
→ System creates/updates media record
→ Profile references avatar_media_id
→ Profile can become basic_complete
```

### 11.3. Post Image Upload Flow

```txt id="75mrmf"
User opens post composer
→ User selects images
→ System validates file count/type/size
→ System uploads images
→ Images attach to draft post
→ User publishes post
→ Post visibility controls image visibility
```

### 11.4. Message Attachment Flow

```txt id="45l87i"
User opens conversation
→ System verifies participant access
→ User selects image/file
→ System validates file type/size
→ System stores file privately
→ Message is sent with attachment metadata
→ Only participants can preview/download
```

### 11.5. Community Resource Flow

```txt id="mse8l6"
Community owner/manager opens resource upload
→ Adds title, description, file
→ Confirms copyright/permission
→ System validates permission, file type and size
→ Resource stored privately/protected
→ Members can access if community/resource visibility allows
```

## 12. Detailed Flow

| Step | Actor Action                 | System Response                  | State             |
| ---: | ---------------------------- | -------------------------------- | ----------------- |
|    1 | User selects file            | Client shows local preview       | selected          |
|    2 | User uploads                 | Server validates auth/permission | validating        |
|    3 | Server validates file        | Check size/MIME/extension        | validating_file   |
|    4 | Server stores file           | Save to disk/object storage      | storing           |
|    5 | Server creates metadata      | Save media_files record          | saved             |
|    6 | UI shows uploaded media      | Attach to parent resource        | uploaded          |
|    7 | User submits parent resource | Parent links media               | attached          |
|    8 | Access requested later       | Server checks permission         | authorized/denied |
|    9 | Media removed                | Soft delete or detach            | deleted/detached  |

## 13. Failure Flows

| Failure                           | Expected Handling                                |
| --------------------------------- | ------------------------------------------------ |
| File too large                    | Reject with size message                         |
| Invalid file type                 | Reject with allowed types                        |
| MIME spoofing                     | Reject after server sniffing                     |
| Upload interrupted                | Show retry                                       |
| Storage write fails               | Do not create metadata                           |
| Metadata save fails               | Delete stored file or mark orphan cleanup        |
| File missing on preview           | Show unavailable state                           |
| Unauthorized private file request | 403                                              |
| User suspended mid-upload         | Reject final attach                              |
| Parent resource deleted           | Hide/detach media                                |
| Media reported/hidden             | Show placeholder                                 |
| Community suspended               | Resources inaccessible except admin/owner policy |

## 20. Security & Privacy

### 20.1. Security Rules

```txt id="iq87lv"
- Validate file type server-side.
- Validate MIME using server inspection where possible.
- Do not trust client-provided MIME.
- Block executable/script-like files.
- Store private files outside public web root.
- Serve private files only through authorized controller.
- Do not expose storage_path in frontend.
- Sanitize original filename before storage.
- Generate safe random filenames.
- Use CSRF protection for uploads.
- Recheck account status before attaching media to parent resource.
- Recheck parent resource permission before preview/download.
```

### 20.2. Privacy Rules

```txt id="a9fp4y"
- Evidence is never public.
- Message attachments are participant-only.
- Community resources follow community access.
- Original filenames may contain sensitive info; avoid showing broadly.
- Analytics must not store storage path or original filename.
- Browser push must not include file names for sensitive attachments.
- Admin evidence preview must be permission-checked.
```

### 20.3. Sensitive Data Matrix

| Data               |    Sensitive | Handling                           |
| ------------------ | -----------: | ---------------------------------- |
| Evidence file      |          Yes | Private storage, protected preview |
| Evidence URL       |          Yes | Admin/owner only                   |
| Message attachment |          Yes | Participant-only                   |
| Community resource | Moderate/Yes | Member/permission access           |
| Avatar             | Low/moderate | Public/protected                   |
| Post image         |     Moderate | Follows post visibility            |
| Storage path       |          Yes | Internal only                      |
| Original filename  |     Moderate | Avoid broad exposure               |
| File hash          |     Moderate | Internal only                      |

| User deletes post                       | Soft delete media relationship, retain according to policy |
| Duplicate upload spam                   | Rate limit by surface                                      |
| Original filename has private data      | Store internally, do not expose broadly                    |
| Browser cannot preview PDF              | Show download/preview fallback if authorized               |
| Virus scan future flags file            | Quarantine and notify admin/user                           |
