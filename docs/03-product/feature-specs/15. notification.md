# Notification

## 1. Feature Summary

### 1.1. One-line Description

Notification cho phép UEConnect gửi thông báo **in-app** và **browser push** cho user về verification, greeting, message, mentor request, moderation action, community activity và system announcement theo cách an toàn, realtime-friendly và phù hợp PWA.

### 1.2. Feature Goal

Feature này tồn tại để:

- Cho user biết khi có sự kiện quan trọng trong UEConnect.
- Hỗ trợ notification trong app qua notification center.
- Hỗ trợ browser push notification cho PWA/web.
- Hỗ trợ installed PWA trên điện thoại hiển thị thông báo như app nhỏ.
- Hỗ trợ notification badge/count trong UI.
- Hỗ trợ mark as read, mark all as read.
- Hỗ trợ notification retention 7 ngày.
- Hỗ trợ safe preview, không leak dữ liệu nhạy cảm.
- Hỗ trợ realtime notification delivery qua Laravel Reverb/Echo nếu user đang online.
- Hỗ trợ queued notification delivery nếu user offline.
- Hỗ trợ phân loại notification theo type, priority, action URL.
- Tôn trọng user privacy settings.
- Đảm bảo các notification quan trọng như moderation/account restriction không bị tắt hoàn toàn.

### 1.3. Product Value

| Value | Description |
|---|---|
| User value | User không bỏ lỡ verification, greeting, message, mentor request hoặc moderation update |
| Product value | Tăng activation, retention và phản hồi kịp thời |
| Business value | Giúp UEConnect hoạt động như một PWA thật, không phải web tĩnh giả dạng app |
| Trust value | Notification quan trọng được gửi an toàn, không leak thông tin nhạy cảm |
| Safety value | Moderation/account notifications giúp user hiểu trạng thái tài khoản/nội dung |
| UX value | Notification rõ ràng, có badge, có action link, không làm phiền quá mức |

### 1.4. Non-goals

Feature này không làm:

- Không gửi SMS trong MVP.
- Không gửi email notification trong MVP, trừ email hệ thống auth/verification nếu module auth cần.
- Không hỗ trợ mobile native push qua app store vì UEConnect là PWA.
- Không gửi notification public ngoài hệ thống.
- Không hiển thị nội dung nhạy cảm đầy đủ trong browser push.
- Không hiển thị raw report detail, evidence path, message attachment path.
- Không cho user tắt notification bắt buộc liên quan đến account/security/moderation.
- Không biến notification thành spam machine để “tăng engagement”, một ý tưởng rất phổ biến và rất đáng bị khóa trong tủ.

## 4. Notification Channels

### 4.1. Supported Channels

| Channel               |      Supported | Priority | Description                                    |
| --------------------- | -------------: | -------: | ---------------------------------------------- |
| In-app notification   |            Yes |       P0 | Notification center inside UEConnect           |
| Realtime in-app event |            Yes |       P0 | Delivered while user is online                 |
| Browser push          |            Yes |       P1 | Web Push API notification                      |
| PWA installed push    |            Yes |       P1 | Same browser push, shown like app notification |
| Email                 | No for feature |      Out | Only auth/system email if needed               |
| SMS                   |             No |      Out | Not MVP                                        |
| Native mobile push    |             No |      Out | No native app                                  |

### 4.2. Channel Rules

```txt
- Every notification should be stored as in-app notification first.
- Browser push is optional delivery layer on top of stored notification.
- Browser push requires explicit user permission.
- Realtime event can update UI badge/panel instantly.
- Browser push must use safe preview.
- Critical account/security/moderation notifications must always appear in-app.
```

### 4.3. Delivery Strategy

Recommended:

```txt
Event occurs
→ Create notification record in database
→ Queue push delivery if enabled
→ Broadcast realtime event to private-user.{userId}
→ UI updates badge/list if user online
→ User opens notification center or clicks push
→ Notification marked read according to action
```

## 5. PWA / Browser Push Explanation

### 5.1. Important Concept

UEConnect định hướng là **PWA**, nghĩa là user có thể mở trên trình duyệt hoặc cài vào màn hình chính điện thoại. Khi PWA được cài, notification vẫn là **browser push notification**, nhưng người dùng thường cảm nhận nó giống thông báo của một app nhỏ trên điện thoại.

Nói cách khác:

```txt
UEConnect không cần app native để có notification cơ bản.
PWA dùng Web Push + Service Worker để gửi thông báo.
```

### 5.2. Installed PWA Behavior

Khi user cài UEConnect PWA trên điện thoại:

```txt
- App có icon trên màn hình chính.
- App mở trong cửa sổ riêng giống app.
- Notification có thể hiện trong notification tray của điện thoại nếu user cấp quyền.
- Khi user bấm notification, PWA mở đúng route liên quan.
- Notification vẫn phụ thuộc vào browser/PWA support của thiết bị.
```

### 5.3. PWA Notification Deep Link

Push notification should include:

```txt
notification_id
type
title
body
action_url
icon
badge icon if supported
timestamp
```

Example action URL:

```txt
/app/messages/{conversationId}
/app/greetings
/app/verification/status
/app/mentor/requests/{requestId}
/app/moderation/status
```

### 5.4. Service Worker Responsibilities

Service Worker should:

```txt
- receive push event
- show notification with safe title/body
- handle notification click
- open existing UEConnect PWA window if possible
- navigate to action_url
- focus existing client if already open
- avoid showing sensitive content
```

### 5.5. PWA Limitations

```txt
- Browser push requires user permission.
- iOS support depends on installed PWA and browser/platform limitations.
- User can deny notification permission.
- Push delivery is not guaranteed instantly.
- Offline device receives notification only when browser push service can deliver.
- Browser/system may throttle push.
```

### 5.6. PWA Widget / Small App Icon Badge

Nếu user nói “widget nhỏ trên điện thoại”, trong ngữ cảnh PWA nên hiểu là:

```txt
- icon PWA ngoài màn hình chính
- notification badge/count nếu nền tảng hỗ trợ
- notification tray item
- app-like notification click behavior
```

UEConnect should support:

```txt
- App icon
- Manifest name and short_name
- Theme color
- Service worker
- Push permission
- Notification click deep link
- Badge icon where supported
```

But should not promise:

```txt
- Native home screen widget with dynamic UI
- Guaranteed iOS badge count
- Guaranteed background sync on every browser
```

Con người đặt tên “PWA” nghe rất thần thánh, nhưng support giữa browser và OS thì vẫn là một nồi lẩu. Tài liệu phải nói rõ để khỏi tự lừa mình.

## 6. Notification Types

| Type                                 | Trigger                            | Receiver          | Priority |
| ------------------------------------ | ---------------------------------- | ----------------- | -------- |
| `verification_approved`              | Admin approves verification        | User              | P0       |
| `verification_rejected`              | Admin rejects verification         | User              | P0       |
| `verification_need_more_information` | Admin requests more info           | User              | P0       |
| `greeting_received`                  | User receives Greeting             | Receiver          | P0       |
| `greeting_accepted`                  | Receiver accepts Greeting          | Sender            | P0       |
| `message_received`                   | New message                        | Other participant | P0       |
| `mentor_request_received`            | Student sends mentor request       | Mentor            | P0/P1    |
| `mentor_request_accepted`            | Mentor accepts request             | Student           | P0/P1    |
| `mentor_request_declined`            | Mentor declines request            | Student           | P1       |
| `mentor_request_need_more_info`      | Mentor asks more info              | Student           | P1       |
| `moderation_action`                  | Content/account action             | Target user       | P0       |
| `account_suspended`                  | Admin suspends account             | User              | P0       |
| `account_banned`                     | Admin bans account                 | User              | P0       |
| `community_invite_or_update`         | Community membership/status update | User/member       | P1       |
| `announcement_published`             | Admin/system announcement          | Target audience   | P1       |

### 6.2. P1/P2 Notification Types

| Type                       | Trigger                          | Receiver       | Priority |
| -------------------------- | -------------------------------- | -------------- | -------- |
| `post_commented`           | New comment on own post          | Post author    | P1       |
| `comment_replied`          | Reply to own comment             | Comment author | P1       |
| `post_reacted`             | Reaction on post                 | Post author    | P2       |
| `community_join_approved`  | Community join approved          | User           | P1       |
| `community_resource_added` | New resource in joined community | Members        | P2       |
| `report_reviewed`          | Report processed                 | Reporter       | P1       |
| `content_restored`         | Moderated content restored       | Owner          | P1       |
| `system_maintenance`       | Maintenance notice               | All/target     | P1       |

### 6.3. Notification Priority Levels

| Priority   | Meaning                      | Behavior                                              |
| ---------- | ---------------------------- | ----------------------------------------------------- |
| `low`      | Nice to know                 | In-app only by default                                |
| `normal`   | Regular update               | In-app + push if enabled                              |
| `high`     | Important user action needed | In-app + push if enabled                              |
| `critical` | Account/security/moderation  | Always in-app, push if allowed, cannot disable in-app |

## 7. Notification Status Model

### 7.1. Notification Statuses

| Status        | Meaning                             |
| ------------- | ----------------------------------- |
| `unread`      | Created, not read                   |
| `read`        | User marked/read                    |
| `archived`    | Hidden from active list             |
| `expired`     | Retention expired                   |
| `deleted`     | Soft deleted/system cleanup         |
| `failed_push` | In-app exists, push delivery failed |

### 7.2. Delivery Statuses

| Status               | Meaning                        |
| -------------------- | ------------------------------ |
| `pending`            | Waiting for delivery job       |
| `broadcasted`        | Realtime event emitted         |
| `push_sent`          | Browser push sent              |
| `push_failed`        | Push failed                    |
| `push_denied`        | User/browser denied permission |
| `push_not_supported` | Browser/device unsupported     |

### 7.3. Lifecycle

```txt
created/unread
→ realtime broadcast if online
→ browser push if enabled
→ user opens/clicks
→ read
→ expired after retention window
```

### 7.4. Retention Rule

Product decision:

```txt
Notification retention = 7 days
```

Rules:

```txt
- Keep active notifications for 7 days.
- Expire old notifications after 7 days.
- Critical account/moderation events may remain visible in account/history pages elsewhere.
- Notification expiration does not delete source data.
- Audit logs are not notification records and do not expire by this rule.
```

## 9. Browser Push Permission Strategy

### 9.1. Permission Timing

Do not ask for push permission immediately on first page load. That is how apps train users to click “Block” faster than their brain boots.

Recommended prompt timing:

```txt
- After user is verified.
- After profile setup is complete.
- After user has seen app value.
- When user visits notification settings.
- After first meaningful event, such as accepted greeting or first message.
```

### 9.2. Permission Prompt Flow

```txt
User opens Notification Settings
→ App explains benefit
→ User clicks Enable browser push
→ Browser native permission prompt appears
→ If granted: subscribe push endpoint
→ Save subscription
→ Send test/success state if needed
```

### 9.3. Soft Prompt Copy

Title:

```txt
Bật thông báo trên thiết bị này?
```

Body:

```txt
UEConnect có thể gửi thông báo khi bạn có tin nhắn, lời chào, cập nhật mentor hoặc trạng thái xác thực.
```

CTA:

```txt
Bật thông báo
```

Secondary:

```txt
Để sau
```

### 9.4. Denied Permission

If user denies:

```txt
- Do not keep asking aggressively.
- Show explanation in settings.
- Provide browser setting guidance if needed.
- Continue in-app notifications normally.
```

Copy:

```txt
Trình duyệt đang chặn thông báo. Bạn vẫn sẽ nhận thông báo trong UEConnect.
```

### 9.5. Unsupported Browser

Copy:

```txt
Trình duyệt hoặc thiết bị hiện không hỗ trợ thông báo đẩy. Bạn vẫn có thể xem thông báo trong UEConnect.
```

## 10. User Notification Preferences

### 10.1. Preference Categories

| Category                    |              Default |   Can Disable? | Notes                                     |
| --------------------------- | -------------------: | -------------: | ----------------------------------------- |
| Verification updates        |                   On | No/P0 critical | Account flow                              |
| Account/security/moderation |                   On |             No | Critical                                  |
| Greeting                    |                   On |            Yes | Connection                                |
| Message                     |                   On |            Yes | Messaging                                 |
| Mentor request              |                   On |            Yes | Mentor                                    |
| Community updates           |                   On |            Yes | Community                                 |
| Post/comment activity       |                   On |            Yes | Feed interaction                          |
| Announcements               |                   On |        Partial | Critical announcements cannot be disabled |
| Browser push                | Off until permission |            Yes | User-controlled                           |

### 10.2. Channel Preferences

For each category:

```txt
in_app_enabled
push_enabled
```

Rules:

```txt
- in_app critical notification cannot be disabled.
- push always requires browser permission.
- if push disabled, in-app still works.
- if user logs out, push subscription should be invalidated or scoped safely.
```

### 10.3. Default Recommendation

```txt
In-app: enabled by default
Browser push: disabled until explicit permission
Critical in-app: always enabled
```

## 11. Safe Preview Rules

### 11.1. General Rule

Notification preview must show enough context to be useful but not leak sensitive content.

### 11.2. Allowed Preview Examples

Message safe preview:

```txt
Bạn có tin nhắn mới từ một UEer.
```

Greeting:

```txt
Bạn có một lời chào mới.
```

Verification:

```txt
Yêu cầu xác thực của bạn đã được duyệt.
```

Mentor:

```txt
Bạn có một yêu cầu mentor mới.
```

Moderation:

```txt
Một nội dung của bạn đang được xem xét theo quy định cộng đồng.
```

### 11.3. Avoid in Push Notification

```txt
- full message body
- full report description
- evidence note/path
- attachment filename if sensitive
- MSSV
- email/phone
- private profile data
- moderator internal reason
- hidden/deleted content body
```

### 11.4. In-app Preview vs Browser Push

| Data                  | In-app                       | Browser Push          |
| --------------------- | ---------------------------- | --------------------- |
| Message short preview | Allowed if preference allows | Safer generic default |
| Greeting sender name  | Allowed                      | Allowed if safe       |
| Verification result   | Allowed                      | Allowed               |
| Report detail         | No                           | No                    |
| Moderation reason     | User-safe summary only       | Generic               |
| Evidence info         | No path/content              | No                    |
| Mentor request topic  | Allowed if safe              | Generic/short         |

## 12. Deep Link / Action URL Rules

### 12.1. Action URL

Each notification should have `action_url`.

Examples:

| Type                                 | Action URL                                                      |
| ------------------------------------ | --------------------------------------------------------------- |
| `verification_approved`              | `/app/welcome` or `/app/profile/setup`                          |
| `verification_rejected`              | `/verification/status`                                          |
| `verification_need_more_information` | `/verification/evidence`                                        |
| `greeting_received`                  | `/app/greetings/received`                                       |
| `greeting_accepted`                  | `/app/messages/{conversationId}`                                |
| `message_received`                   | `/app/messages/{conversationId}`                                |
| `mentor_request_received`            | `/app/mentor/requests/{requestId}`                              |
| `mentor_request_accepted`            | `/app/messages/{conversationId}`                                |
| `moderation_action`                  | `/app/notifications/{notificationId}` or related content/status |
| `announcement_published`             | `/app/announcements/{announcementId}`                           |

### 12.2. Deep Link Security

```txt
- Clicking notification must recheck auth.
- If user is logged out, redirect login then intended URL.
- If user lacks access, show unavailable/permission state.
- If content deleted/hidden, show safe unavailable page.
```

### 12.3. PWA Click Behavior

When push notification clicked:

```txt
Service Worker receives notificationclick
→ closes notification
→ checks existing PWA windows
→ focuses existing window if possible
→ navigates/open action_url
→ app fetches notification details
→ marks notification read if policy allows
```

## 13. Main User Flows

### 13.1. Receive In-app Notification While Online

```txt
Event occurs
→ Notification record created
→ Realtime event broadcast to private-user.{userId}
→ UI badge increments
→ Notification dropdown/list updates
→ User clicks notification
→ Opens action_url
→ Notification marked read
```

### 13.2. Receive Browser Push While PWA Not Open

```txt
Event occurs
→ Notification record created
→ Push job checks subscription/preference
→ Browser push sent
→ Device shows notification
→ User taps notification
→ Installed PWA/browser opens action_url
→ App authenticates/checks access
→ Notification marked read
```

### 13.3. Enable Browser Push

```txt
User opens Settings → Notifications
→ User sees explanation
→ User clicks Enable browser push
→ Browser permission prompt
→ If granted: frontend gets PushSubscription
→ Backend stores subscription
→ Push enabled for this device
```

### 13.4. Mark Notification as Read

```txt
User opens notification center
→ Clicks notification or mark read
→ System updates read_at
→ UI unread count decreases
```

### 13.5. Mark All as Read

```txt
User opens notification center
→ Clicks Mark all as read
→ System updates unread notifications for user
→ Badge count becomes 0
```

## 14. Failure & Alternative Flows

| Flow                                              | Condition                               | Expected Behavior |
| ------------------------------------------------- | --------------------------------------- | ----------------- |
| Push permission denied                            | Store no subscription, continue in-app  |                   |
| Push unsupported                                  | Show unsupported state                  |                   |
| Push delivery fails                               | Mark push_failed, keep in-app           |                   |
| User logged out after push click                  | Redirect login, then intended URL       |                   |
| Notification target deleted                       | Show unavailable                        |                   |
| Notification action no longer allowed             | Show permission/unavailable             |                   |
| Realtime disconnected                             | Notification appears after refresh/poll |                   |
| Duplicate event received                          | Client deduplicates by notification_id  |                   |
| Notification already read                         | Keep read state                         |                   |
| Subscription expired                              | Remove or deactivate subscription       |                   |
| User disables push                                | Stop push delivery, keep in-app         |                   |
| Critical notification preference disabled attempt | Prevent disabling in-app                |                   |
| Notification older than 7 days                    | Expired/hidden from center              |                   |

## 21. Security & Privacy

### 21.1. Security Rules

```txt
- User can only view own notifications.
- User can only mark own notifications.
- Push subscription belongs to authenticated user.
- Push endpoint must not be exposed to other users.
- Service worker must not include secrets.
- VAPID private key must be stored in environment variables.
- Notification click must recheck route authorization.
- Browser push payload must be safe.
- Critical notifications cannot be disabled in-app.
```

### 21.2. Privacy Rules

```txt
- Do not store raw message body in analytics.
- Do not send full sensitive message in push.
- Do not send report detail in push or notification preview.
- Do not include evidence path or file path.
- Do not include full MSSV/email/phone.
- Do not show hidden/deleted content body.
- Do not expose moderator internal reason.
```

### 21.3. Sensitive Data Matrix

| Data                  | Sensitive? | Handling              |
| --------------------- | ---------: | --------------------- |
| Push endpoint         |        Yes | Owner/system only     |
| Message body          |        Yes | Safe preview only     |
| Report detail         |        Yes | Never in notification |
| Evidence info         |        Yes | Never path/content    |
| Moderation reason     |  Sensitive | User-safe copy only   |
| Action URL            |   Moderate | Owner only            |
| Actor identity        |   Moderate | Show only when safe   |
| Notification metadata |   Moderate | Safe fields only      |

### 22.1. Audit Needed?

| Action                                           |                  Audit Required | Reason                 |
| ------------------------------------------------ | ------------------------------: | ---------------------- |
| Create normal notification                       |                              No | Operational            |
| Mark read                                        |                              No | User action            |

### 22.2. Audit Rule

Notification is usually not the source of truth for sensitive actions.

```txt
Example:

→ audit belongs to admin/moderation action
→ notification tells user safely
```

Không audit mỗi cái chuông kêu. Audit quyết định khiến chuông kêu. Cũng giống không ghi biên bản từng tiếng thở, cảm ơn trời đất.

## 27. Edge Cases

| Edge Case                                        | Expected Handling                               |
| ------------------------------------------------ | ----------------------------------------------- |
| User denies push permission                      | Keep in-app notification                        |
| Browser unsupported                              | Show unsupported state                          |
| Push subscription expires                        | Mark inactive and ask re-enable                 |
| User logs out                                    | Do not show private data in current app session |
| Push clicked while logged out                    | Redirect login then intended URL                |
| Notification target deleted                      | Show unavailable                                |
| Notification target hidden by moderation         | Show unavailable/safe status                    |
| Realtime event duplicated                        | Deduplicate by notification_id                  |
| Notification created but broadcast fails         | Fetch later shows notification                  |
| Push sent but in-app not created                 | Should not happen; in-app first                 |
| User disables message push                       | In-app still created                            |
| Critical notification disabled attempt           | Prevent disabling in-app                        |
| Notification older than 7 days                   | Expire/hide from center                         |
| User has multiple devices                        | Send push to all active subscriptions           |
| One device subscription fails                    | Deactivate that subscription only               |
| PWA opened from notification with stale route    | App shows safe fallback                         |
| Message notification after block                 | Suppress if block created before send           |
| Moderation notification contains reporter detail | Must never happen                               |

## 36. Final Feature Statement

```txt
Notification là communication feedback layer của UEConnect. Feature này phải lưu mọi notification dưới dạng in-app notification trước, sau đó mới dispatch realtime và browser push nếu phù hợp. Vì UEConnect là PWA, browser push thông qua Service Worker giúp installed PWA trên điện thoại hiển thị notification gần giống app native, nhưng không được hứa các khả năng native widget/badge tuyệt đối. Notification phải có safe preview, action URL, unread state, mark as read, retention 7 ngày, realtime update, push permission rõ ràng và không leak dữ liệu nhạy cảm.
```

## 37. Short Version

```txt
Notification chỉ gồm in-app và browser push.
In-app là source of truth.
Browser push là delivery layer.
PWA cài trên điện thoại có thể hiện thông báo như app nếu browser/OS hỗ trợ.
Không promise native widget thật.
Có notification center.
Có badge/unread count.
Có mark as read.
Retention 7 ngày.
Push phải xin permission đúng lúc.
Push không leak message/report/evidence.
Critical in-app notification không được tắt.
Realtime dùng private-user channel.
Notification mà leak dữ liệu nhạy cảm thì không phải tiện lợi, nó là tai nạn có icon cái chuông.
```
