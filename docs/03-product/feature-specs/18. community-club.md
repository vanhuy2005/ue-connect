# Community & Club

## 1. Feature Summary

### 1.1. One-line Description

Community & Club cho phép UEConnect tổ chức các cộng đồng/CLB trong HCMUE với membership, community feed, community chat, resources, owner/manager roles, join request, moderation, scoped permissions và admin governance rõ ràng.

### 1.2. Feature Goal

Feature này tồn tại để:

- Cho sinh viên HCMUE tìm và tham gia cộng đồng/CLB phù hợp.
- Hỗ trợ CLB, khoa, nhóm học tập, nhóm học thuật, nhóm định hướng nghề nghiệp và cộng đồng sở thích trong trường.
- Cho admin tạo và quản lý community chính thức.
- Cho user đề xuất community/CLB thông qua form hoặc kênh liên hệ.
- Cho admin gán `community_owner` và `club_manager` theo scope từng community.
- Cho community owner/club manager vận hành bài viết, thành viên, tài nguyên và chat trong phạm vi được cấp quyền.
- Hỗ trợ community feed tương tự feed chính nhưng scoped trong community.
- Hỗ trợ community resources để lưu tài liệu/học liệu/link hợp lệ.
- Hỗ trợ community chat realtime hoặc near-realtime theo định hướng Messaging.
- Hỗ trợ join request và approval flow.
- Hỗ trợ report, moderation, suspend community, hidden content placeholder.
- Hỗ trợ audit log cho các hành động admin/manager nhạy cảm.
- Chuẩn bị kiến trúc để scale thành nhiều loại cộng đồng khác nhau, không chỉ CLB.

### 1.3. Product Value

| Value | Description |
|---|---|
| User value | User tìm được cộng đồng học tập, CLB, mentor group, career group hoặc nhóm sở thích trong HCMUE |
| Product value | Tạo retention loop bên cạnh Feed, Discovery, Mentor và Messaging |
| Business value | Làm UEConnect giống một campus social platform thật, không chỉ là app profile/card |
| Trust value | Community được admin tạo/duyệt, owner/manager có scoped permission |
| Safety value | Có report, moderation, role kiểm soát, audit, community suspension |
| UX value | User có nơi tập trung bài viết, chat, tài nguyên, thành viên theo từng cộng đồng |

### 1.4. Non-goals

Feature này không làm:

- Không cho user tự do tạo community public ngay lập tức trong MVP.
- Không cho club manager có quyền toàn hệ thống.
- Không thay thế hệ thống quản lý CLB chính thức của nhà trường nếu có.
- Không tạo cộng đồng ngoài phạm vi HCMUE.
- Không cho cộng đồng chứa tài liệu vi phạm bản quyền.
- Không cho community chat thành kênh spam không kiểm soát.
- Không cho community owner tự xử lý report có conflict of interest.
- Không cho community bị suspended vẫn hoạt động như bình thường.
- Không tạo marketplace/job board trong community.
- Không biến community thành Facebook Group clone vô pháp, vì nhân loại đã thử rồi và kết quả không quá rực rỡ.

## 4. Actors & Permissions

### 4.1. Actors

| Actor                 | Description                                                    |
| --------------------- | -------------------------------------------------------------- |
| Verified User         | Can view/join public or eligible communities                   |
| Student               | Main community member                                          |
| Alumni                | Can join alumni/career/community groups if allowed             |
| Advisor               | Can join/manage academic communities if granted                |
| Community Member      | User accepted into community                                   |
| Community Owner       | Highest community-level operator for one community             |
| Club Manager          | Scoped operator with granted permissions for one community     |
| Admin                 | Creates, manages, suspends, audits communities                 |
| Moderator             | Reviews community reports/content                              |
| Suspended/Banned User | Cannot participate                                             |
| System                | Automated actions: auto-hide, notifications, scheduled cleanup |

### 4.2. Community Role Types

| Role                | Scope         | Meaning                                   |
| ------------------- | ------------- | ----------------------------------------- |
| `member`            | community     | Normal accepted member                    |
| `pending_member`    | community     | Waiting for approval                      |
| `community_owner`   | community     | Main responsible user                     |
| `club_manager`      | community     | Operator with scoped permissions          |
| `content_moderator` | community     | Can moderate community content if granted |
| `resource_manager`  | community     | Can manage resources if granted           |
| `chat_moderator`    | community     | Can manage chat if granted                |
| `admin`             | global        | Platform admin                            |
| `moderator`         | global/scoped | Platform moderation                       |

### 4.3. Required Permissions

| Action                        | Required Permission                         | Scope               |
| ----------------------------- | ------------------------------------------- | ------------------- |
| View public community         | verified active user                        | app                 |
| View private community detail | member/admin/manager                        | community           |
| Request join                  | verified active user                        | community           |
| Approve join request          | `manage_community_members`                  | community_id        |
| Reject join request           | `manage_community_members`                  | community_id        |
| Remove member                 | `manage_community_members`                  | community_id        |
| Create community              | `manage_communities`                        | global              |
| Edit community                | `manage_community`                          | community_id/global |
| Suspend community             | `manage_communities` / `moderate_content`   | global              |
| Archive community             | `manage_communities`                        | global              |
| Grant club manager            | `manage_permissions` / `manage_communities` | global              |
| Create community post         | member                                      | community           |
| Manage community post         | author or `moderate_content`                | community_id        |
| Create resource               | member/resource_manager depending policy    | community           |
| Approve resource              | `manage_community_resources`                | community_id        |
| Send community chat message   | member                                      | community           |
| Moderate community chat       | `moderate_community_chat`                   | community_id        |
| Report community/content      | verified active user/member                 | target              |
| View community audit          | `view_audit_log` / `manage_communities`     | global/scoped       |

### 4.4. Scoped Permission Model

Community management must use scoped permissions.

Example:

```txt
permission_key = manage_club
scope_type = community
scope_id = 123
```

More specific examples:

```txt
manage_community_members:community:123
manage_community_posts:community:123
manage_community_resources:community:123
moderate_community_chat:community:123
```

### 4.5. Permission Rules

```txt
- Community owner and club manager permissions are scoped by community_id.
- Club manager cannot manage communities outside assigned scope.
- Club manager cannot grant global permissions.
- Community owner can manage community-level operations only.
- Admin can override community operation if permission allows.
- Suspended/banned users lose community participation abilities.
- Community suspended state overrides all member/manager permissions for normal operations.
- UI hiding is not authorization.
- Server-side policy is required for every action.
```

## 6. Community Status Model

### 6.1. Community Statuses

| Status                 | Meaning                                     |
| ---------------------- | ------------------------------------------- |
| `draft`                | Created but not public                      |
| `pending_review`       | Suggested/community proposal awaiting admin |
| `active`               | Normal operation                            |
| `inactive`             | Visible but low/no operation                |
| `suspended`            | Locked by admin/moderation                  |
| `archived`             | Historical, read-only or hidden             |
| `hidden_by_moderation` | Hidden due to safety                        |
| `deleted`              | Soft deleted                                |

### 6.2. Status Effects

| Status                 | User View                         | Member Actions       | Manager Actions              |
| ---------------------- | --------------------------------- | -------------------- | ---------------------------- |
| `draft`                | Admin only                        | No                   | Admin edit                   |
| `pending_review`       | Admin/reviewer only               | No                   | Review only                  |
| `active`               | Visible by policy                 | Yes                  | Yes                          |
| `inactive`             | Visible with inactive badge       | Limited/Yes          | Yes                          |
| `suspended`            | Locked page with reason-safe copy | No posting/chat/join | Limited admin/appeal actions |
| `archived`             | Optional read-only                | No new activity      | Admin restore/archive        |
| `hidden_by_moderation` | Unavailable                       | No                   | Admin/mod only               |
| `deleted`              | Not visible                       | No                   | Admin restore maybe          |

### 6.3. Suspended Community UI

When community is suspended:

```txt
Cộng đồng này hiện đang bị tạm khóa.
```

Optional safe reason:

```txt
Một số hoạt động của cộng đồng đã bị hạn chế theo quy định cộng đồng của UEConnect.
```

Do not expose:

```txt
- reporter identity
- raw report description
- internal admin notes
- sensitive moderation details
```

## 7. Community Membership

### 7.1. Membership Status

| Status                  | Meaning                                     |
| ----------------------- | ------------------------------------------- |
| `pending`               | Join request waiting                        |
| `active`                | Accepted member                             |
| `rejected`              | Join request rejected                       |
| `left`                  | User left                                   |
| `removed`               | Manager/admin removed user                  |
| `banned_from_community` | User blocked/banned from specific community |
| `muted`                 | User can view but not post/chat             |
| `restricted`            | Limited due to moderation                   |
| `invited`               | User invited but not accepted yet           |

### 7.2. Membership Rules

```txt
- User must be verified active to join.
- Suspended/banned platform users cannot join or participate.
- Community-specific banned users cannot rejoin unless restored.
- Private community requires invite or approval.
- Approval-required community creates membership status pending.
- User can leave community.
- Manager/admin can remove member with reason.
- Removing member should not delete their old posts unless moderation requires.
```

### 7.3. Join Request Required Fields

| Field          |    Required | Notes                                       |
| -------------- | ----------: | ------------------------------------------- |
| `community_id` |         Yes | Target community                            |
| `join_reason`  | Conditional | Required for restricted/private communities |
| `role_context` |          No | Student/alumni/advisor                      |
| `note`         |          No | Additional note                             |

Recommended:

```txt
For club community, join_reason max = 500 characters.
```

### 7.4. Join Request Flow

```txt
User opens community
→ Clicks Tham gia
→ If open: membership active
→ If approval_required: join request pending
→ Owner/manager receives notification
→ Owner/manager approves/rejects
→ User receives notification
```

### 7.5. Member Removal Flow

```txt
Manager/Admin opens member list
→ Selects member
→ Chooses remove/restrict/mute
→ Enters reason
→ System applies action
→ Audit/community activity record created
→ User notified with safe copy
```

## 8. Community Roles & Governance

### 8.1. Community Owner

Community owner can:

```txt
- edit community profile if allowed
- manage join requests
- manage members
- assign limited internal roles if allowed
- create announcements
- moderate posts/resources/chat within community if granted
- view community management dashboard
```

Community owner cannot:

```txt
- grant global permissions
- modify platform role
- access admin audit globally
- override platform moderation
- handle report where they are the target
```

### 8.2. Club Manager

Club manager is an operator, not a global admin.

Club manager can have scoped permissions such as:

```txt
- manage_community_posts
- manage_community_members
- manage_community_resources
- moderate_community_chat
```

Rule:

```txt
Club manager permission must be explicitly scoped to community_id.
```

### 8.3. Internal Community Roles

Future-ready internal roles:

```txt
president
vice_president
secretary
event_lead
content_lead
resource_manager
chat_moderator
member
```

MVP should store role label flexibly but enforce permissions separately.

Because titles are politics. Permissions are reality. Humanity often confuses the two.

### 8.4. Conflict of Interest Rules

```txt
- Community owner/manager cannot dismiss report against themselves.
- Community owner/manager cannot approve resource/post they authored if community policy requires review.
- Reports involving community leadership should escalate to admin/moderator.
- Admin can override community-level decisions.
```

## 9. Community Feed

### 9.1. Community Feed Meaning

Community feed là post stream scoped trong một community.

It can include:

```txt
- member posts
- owner/manager announcements
- pinned posts
- resource updates
- event updates future
- mentor/career pathway posts future
```

### 9.2. Community Post Types

| Type             | Description                      |
| ---------------- | -------------------------------- |
| `standard_post`  | Normal member post               |
| `announcement`   | Owner/manager/admin announcement |
| `resource_share` | Resource-related post            |
| `question`       | Member question                  |
| `discussion`     | Discussion                       |
| `event_notice`   | Future event                     |
| `system_update`  | System/community status update   |

### 9.3. Post Visibility

| Visibility          | Meaning                           |
| ------------------- | --------------------------------- |
| `community_members` | Only members                      |
| `public_preview`    | Preview visible to verified users |
| `managers_only`     | Internal management post          |
| `admin_only`        | Admin/system                      |

MVP recommendation:

```txt
Default community post visibility = community_members.
Community announcements can have public preview if community visibility allows.
```

### 9.4. Community Feed Rules

```txt
- Only active members can post unless community allows admin-only posting.
- Suspended/muted/restricted members cannot post.
- Community must be active.
- Posts follow same content policy as home feed.
- Report/hide/delete/restore post must integrate with Moderation.
- Auto-hide threshold can apply to community posts.
- Hidden content shows placeholder.
```

### 9.5. Pinning Posts

P1:

```txt
- Owner/manager can pin important posts.
- Max pinned posts per community = configurable, default 3.
- Pinned posts appear above feed.
```

## 10. Community Resources

Community resources giúp lưu trữ tài nguyên hợp lệ:

```txt
- syllabus links if allowed
- learning notes written by members
- slides created by club/community
```

### 10.4. Resource Status

| ----------------------- | ----------------------------- |

| `rejected`              | Not approved                  |
| `hidden_by_moderation`  | Hidden                        |
| `removed_by_moderation` | Removed                       |

### 10.5. Resource Review Rule

Recommended MVP:

```txt
All uploaded files in community resources should be pending_review before public display.
Links can be published by manager/resource_manager but reportable.
```

```txt
All resources, including links, require approval before public display.
```

Enterprise-safe recommendation:

```txt
Use strict review for MVP. Fewer lawsuits, fewer headaches, fewer “ai up giáo trình lậu vậy?” moments.
```

## 11. Community Chat

### 11.1. Community Chat Purpose

Community chat cho phép thành viên trao đổi nhanh trong community.

It should support:

```txt
- text messages
- image/file attachments if allowed
- realtime delivery using Messaging architecture
- report message
- block user
- chat moderation
- community suspended state
```

### 11.2. Chat Channel Model

MVP:

```txt
one default chat channel per active community
```

Future:

```txt
multiple channels per community:
- general
- announcements
- resources
- questions
- events
```

### 11.3. Community Chat Access

```txt
- Only active members can view/send.
- Muted/restricted members can view but cannot send depending restriction.
- Suspended community disables sending.
- Archived community disables sending.
- Community chat messages can be reported.
- Manager/moderator can hide/delete messages if granted.
```

### 11.4. Realtime Direction

Use same realtime stack as Messaging:

```txt
Laravel Reverb + Echo
private-community-chat.{communityId}
presence-community-chat.{communityId}
```

Events:

```txt
CommunityMessageSent
CommunityMessageEdited
CommunityMessageDeleted
CommunityTypingStarted
CommunityTypingStopped
CommunityChatRestricted
```

### 11.5. Chat Safety

```txt
- Keyword flagging can apply.
- Auto-hide can apply to reported messages.
- Reported messages enter moderation queue.
- Message body must not appear in analytics.
- Attachment access must be community-member protected.
```

## 12. Community Suggestion Flow

### 12.1. Why Suggestion Exists

User cannot freely create community in MVP, but should be able to suggest one.

Channels:

```txt
- built-in suggestion form
- email/contact fallback
- private chat/support future
```

Recommended MVP:

```txt
Built-in community suggestion form.
```

### 12.2. Suggestion Fields

| Field                    |       Required | Notes                   |
| ------------------------ | -------------: | ----------------------- |
| `suggested_name`         |            Yes | Community/club name     |
| `community_type`         |            Yes | club/academic_group/etc |
| `purpose`                |            Yes | Why it should exist     |
| `target_members`         |            Yes | Who it serves           |
| `proposed_owner_id`      |             No | If known                |
| `related_faculty`        |             No | Faculty/department      |
| `related_program`        |             No | Academic program        |
| `related_career_pathway` |             No | Optional                |
| `supporting_note`        |             No | Extra                   |
| `contact_email`          | Auto/from user | User email              |

### 12.3. Suggestion Status

| Status                   | Meaning                    |
| ------------------------ | -------------------------- |
| `submitted`              | User submitted             |
| `under_review`           | Admin reviewing            |
| `need_more_information`  | Admin asks more            |
| `approved`               | Community created/accepted |
| `rejected`               | Not accepted               |
| `converted_to_community` | Community record created   |
| `duplicate`              | Similar community exists   |

## 13. Main User Flows

### 13.1. Discover Community

```txt
User opens Communities
→ System shows communities by category/status/relevance
→ User filters by type/faculty/pathway
→ User opens community detail
→ User requests join or joins directly depending policy
```

### 13.2. Join Community

```txt
User opens active community
→ Clicks Tham gia
→ If join_policy = open: active membership created
→ If approval_required: join request pending
→ Manager/owner notified
→ Manager approves/rejects
→ User notified
```

### 13.3. Community Owner Approves Member

```txt
Owner/manager opens Join Requests
→ Reviews user profile + join reason
→ Approves/rejects
→ Membership status updated
→ User notified
→ Audit/community activity created
```

### 13.4. Create Community by Admin

```txt
Admin opens Admin Communities
→ Clicks Create Community
→ Fills community info
→ Assigns owner/manager if needed
→ Sets visibility/join policy/status
→ Publishes as active
→ Audit created
```

### 13.5. Suggest Community

```txt
User opens Suggest Community
→ Fills suggestion form
→ Suggestion submitted
→ Admin reviews
→ Admin approves/rejects/asks more info
→ User notified
```

### 13.6. Post in Community Feed

```txt
Member opens community feed
→ Creates post
→ System validates membership/status/content
→ Post published or pending review depending policy
→ Members may see in community feed
```

### 13.7. Upload Community Resource

```txt
Member opens Resources
→ Submits file/link with copyright attestation
→ Resource status = pending_review
→ Resource manager/admin reviews
→ Approves/rejects
→ Published resource appears in community
```

### 13.8. Use Community Chat

```txt
Member opens Chat tab
→ System checks active membership
→ Loads chat messages
→ Subscribes to realtime channel
→ User sends message
→ Message stored and broadcast
```

### 13.9. Suspend Community

```txt
Admin/moderator opens community admin
→ Chooses Suspend
→ Enters reason
→ Community status = suspended
→ Posting/chat/join disabled
→ Members see locked state
→ Audit created
```

### 17.7. `community_chat_messages`

Option A:

```txt
Reuse messaging tables with conversation.type = community_channel.
```

Option B:

```txt
Separate community_chat_messages table.
```

Recommended:

```txt
Reuse messaging/conversation architecture where possible:
conversations.type = community_channel
conversations.source_type = community
conversations.source_id = community_id
```

Reason:

```txt
One chat architecture is already painful enough. Creating two separate chat systems is how future you learns regret-driven development.
```

### 17.8. Constraints

```txt
- community slug unique.
- community owner must be verified active user.
- only one active membership per user/community.
- join request unique pending per user/community.
- suspended community blocks new posts/chat/join.
- published resources require copyright_attestation = true.
- pending resources are not visible to normal members.
- manager permissions must be scoped by community_id.
- all community tables use soft delete.
```

## 20. Security & Privacy

### 20.1. Security Rules

```txt
- Community routes require verified active user unless public preview allowed.
- Private/restricted community detail requires membership or permission.
- Community chat requires active membership.
- Resource files are served through protected route.
- Raw storage paths are never exposed.
- Club manager permissions are scoped and checked server-side.
- Community owner/manager cannot act outside community scope.
- Suspended community blocks posting/chat/join.
- Reported community content can be viewed only by authorized moderator/admin.
- Admin/community manager actions require reason and audit where sensitive.
```

### 20.2. Privacy Rules

```txt
- Private community membership list can be hidden based on community settings.
- User profile privacy applies inside member list.
- Join reason visible only to managers/admin.
- Resource submitter visible according to community policy.
- Chat messages visible only to active members and authorized moderators for reports.
- Report details are not exposed to community owner if conflict exists.
- Admin notes are never visible to normal users.
```

### 20.3. Sensitive Data Matrix

| Data                         |         Sensitive? | Handling                 |
| ---------------------------- | -----------------: | ------------------------ |
| Join reason                  |           Moderate | Managers/admin only      |
| Private community membership |           Moderate | Community settings       |

| Resource file                | Moderate/Sensitive | Access controlled        |
| Report detail                |          Sensitive | Moderator/admin only     |

## 24. Edge Cases

| Edge Case                                     | Expected Handling                      |
| --------------------------------------------- | -------------------------------------- |
| User tries to join suspended community        | Reject with locked state               |
| User already active member joins again        | Idempotent/show already joined         |
| User has pending join request                 | Show pending state                     |
| User banned from community tries join         | Reject                                 |
| Manager approves same request twice           | Idempotent/stale state                 |
| Manager loses permission while page open      | Action rechecks and rejects            |
| Community suspended while user composing post | Submit rejected                        |
| Community archived while chat open            | Chat becomes read-only                 |
| Owner removed from platform                   | Admin must assign new owner            |
| Club manager acts outside scope               | 403                                    |
| Manager tries to remove owner                 | Require admin or stronger permission   |
| Resource file deleted                         | Show unavailable                       |
| Resource copyright violation reported         | Hide/send moderation                   |
| Private community accessed by non-member      | Show locked/private state              |
| Report against community owner                | Escalate to admin                      |
| Community chat realtime fails                 | Fallback fetch/poll                    |
| User leaves community                         | Loses chat/resource/member access      |
| Suspended user remains member                 | Membership remains but actions blocked |
| Duplicate community suggestion                | Mark duplicate/link existing           |
| Admin audit fails during suspend              | Rollback suspend action                |

## 33. Final Feature Statement

```txt
Community & Club là campus community core của UEConnect. Feature này cho phép verified users khám phá, tham gia và tương tác trong các cộng đồng/CLB thuộc HCMUE thông qua community profile, membership, feed, resources và chat. Community phải được admin tạo/duyệt, có owner/club manager với scoped permissions theo community_id, có join request, resource review, moderation, report, suspended state, audit và notification. Community không phải group chat vô luật; nó là không gian campus được quản trị rõ ràng để sinh viên, alumni, advisor và CLB hoạt động an toàn.
```
