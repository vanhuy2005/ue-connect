# Profile Management

## 1. Feature Summary

### 1.1. One-line Description

Profile Management cho phép user đã verified thiết lập, chỉnh sửa và kiểm soát hồ sơ cá nhân của mình để xuất hiện đáng tin trong Feed, Discovery, Greeting, Messaging, Mentor và Community.

### 1.2. Feature Goal

Feature này tồn tại để:

- Biến verified account thành một identity có ngữ cảnh, không chỉ là email + role.
- Giúp user thể hiện mình là ai trong cộng đồng HCMUE.
- Cung cấp thông tin cần thiết cho Discovery, Mentor matching, Community và Feed.
- Tạo sự tin cậy thông qua verified badge, faculty, major, cohort, role badge và profile completeness.
- Cho phép user kiểm soát mức độ hiển thị thông tin cá nhân.
- Bảo vệ dữ liệu nhạy cảm như MSSV, email, evidence, admin note, report history.
- Tránh profile bị dating hóa hoặc LinkedIn hóa quá mức.
- Làm nền cho recommendation, search, filter và analytics sau này.

### 1.3. Product Value

| Value | Description |
|---|---|
| User value | User có hồ sơ rõ ràng, đáng tin, thể hiện được bản thân và mục tiêu kết nối |
| Product value | Feed, Discovery, Mentor và Community có context để hoạt động tốt |
| Business value | UEConnect trông giống sản phẩm thật, không phải clone mạng xã hội thiếu identity |
| Trust value | Verified badge + HCMUE context giúp user tin người mình đang tương tác |
| Safety value | Privacy controls giúp user tránh lộ thông tin quá mức |
| UX value | Profile giúp người khác hiểu “vì sao nên kết nối với người này” |

### 1.4. Non-goals

Feature này không làm:

- Không public đầy đủ MSSV.
- Không public email, phone, evidence hoặc admin note.
- Không biến profile thành dating profile.
- Không dùng các từ như crush, match, hot, dating, swipe trên profile.
- Không biến profile thành CV dài như LinkedIn.
- Không yêu cầu user điền tất cả mọi thứ trước khi dùng app.
- Không cho user chỉnh sửa role chính sau verification.
- Không cho user giả mạo faculty, major, cohort đã được xác thực mà không qua review.
- Không hiển thị report history hoặc moderation history cho user khác.
- Không có “profile score” công khai, vì app này không cần chấm điểm con người như món hàng.

## 4. Actors & Permissions

### 4.1. Actors

| Actor            | Can Use? | Notes                                                        |
| ---------------- | -------: | ------------------------------------------------------------ |
| Guest            |       No | Cannot view app profiles                                     |
| Registered user  |       No | Must verify first                                            |
| Pending user     |       No | Can only see verification status                             |
| Verified Student |      Yes | Can setup/edit own profile                                   |
| Verified Alumni  |      Yes | Can setup/edit alumni profile                                |
| Verified Advisor |      Yes | Can setup/edit advisor profile                               |
| Mentor           |      Yes | Can add mentor-specific sections if approved                 |
| Club Manager     |      Yes | Profile still user-owned                                     |
| Admin            |      Yes | Can view profile admin detail; limited edit if policy allows |
| Moderator        |  Limited | Can view profile for reports/moderation                      |
| Suspended user   |  Limited | Can view account status; profile may be restricted           |
| Banned user      |       No | No product access                                            |

### 4.2. Required Permissions

| Action                                    | Required Role   | Required Permission                     | Scope          |
| ----------------------------------------- | --------------- | --------------------------------------- | -------------- |
| View own profile                          | verified user   | none                                    | own            |
| Edit own profile                          | verified user   | none                                    | own            |
| View public profile                       | verified user   | none                                    | public/allowed |
| Upload own avatar                         | verified user   | none                                    | own            |
| Change visibility settings                | verified user   | none                                    | own            |
| View full profile in admin                | admin/moderator | `manage_users` or `moderate_content`    | global/scoped  |
| Hide profile by moderation                | admin/moderator | `moderate_content`                      | global         |
| Edit verified identity fields             | admin/reviewer  | `review_verification` or `manage_users` | global         |
| View sensitive verification-linked fields | admin           | `review_verification` / `manage_users`  | global         |

### 4.3. Access Rules

```txt
- Only active verified users can create full profile.
- User can edit only their own editable profile fields.
- User cannot edit primary role directly.
- User cannot edit verified identity fields without re-review or admin correction.
- Full MSSV must never appear on public profile.
- Email and phone are private by default.
- Admin-only fields must never leak into public profile.
- Blocked users cannot view each other's full public profile.
- Discovery visibility controls whether profile appears in Discovery.
- Account status overrides profile visibility.
```

## 9. Profile Completion Model

### 9.1. Completion Levels

| Level            | Meaning                   | Access                                                 |
| ---------------- | ------------------------- | ------------------------------------------------------ |
| `not_started`    | User has not opened setup | Redirect onboarding                                    |
| `incomplete`     | Missing required fields   | Limited app / setup prompt                             |
| `basic_complete` | Required profile done     | Full app access                                        |
| `enhanced`       | Added interests/goals     | Better Discovery/Mentor                                |
| `mentor_ready`   | Mentor fields complete    | Can receive mentor request if `mentor_access` approved |

### 9.2. Required for Basic Complete

```txt
avatar
display_name
faculty
role-specific required fields
short_bio
privacy defaults confirmed
```

### 9.3. Profile Completion Rules

```txt
- Avatar is required.
- Required fields must be completed before entering full app.
- Optional fields should be encouraged, not forced.
- User can skip onboarding but not required profile fields.
- Profile completion should be recalculated after edit.
```

## 10. Main User Flow

### 10.1. Happy Path: Student Profile Setup

```txt
Verification approved
→ User lands on Welcome screen
→ User enters Profile Setup
→ User uploads avatar
→ User confirms display name
→ User confirms faculty / major / cohort
→ User writes short bio
→ User selects learning interests
→ User selects connection goals
→ User confirms privacy defaults
→ Profile becomes basic_complete
→ Redirect to /app/home
```

### 10.2. Happy Path: Alumni Profile Setup

```txt
Alumni verification approved
→ User lands on Alumni Welcome screen
→ User uploads avatar
→ User confirms HCMUE background
→ User adds current field
→ User writes short bio
→ User selects can-help-with topics
→ User chooses mentor availability if desired
→ User confirms privacy defaults
→ Profile becomes basic_complete
```

### 10.3. Happy Path: Advisor Profile Setup

```txt
Advisor verification approved
→ User lands on Advisor Welcome screen
→ User uploads avatar
→ User confirms department and position
→ User adds support area
→ User writes short bio
→ User chooses advising/mentor topics
→ User confirms privacy defaults
→ Profile becomes basic_complete
```

### 10.4. Edit Profile Flow

```txt
User opens My Profile
→ Clicks Edit Profile
→ Updates editable fields
→ System validates fields
→ User saves
→ System updates profile
→ Recalculates completion
→ Creates analytics event
→ Shows success state
```

### 10.5. Public Profile View Flow

```txt
User opens another profile
→ System checks access
→ System checks block relationship
→ System checks profile visibility
→ System renders public-safe fields
→ User can send greeting / message if allowed / report / block
```

## 12. Alternative & Failure Flows

| Flow                         | Condition                   | Expected Behavior                    |
| ---------------------------- | --------------------------- | ------------------------------------ |
| Avatar upload invalid        | File too large/wrong type   | Show validation error                |
| User skips avatar            | Avatar required             | Block basic completion               |
| User skips optional fields   | Optional missing            | Allow continue                       |
| User edits verified field    | Faculty/major/cohort locked | Show explanation / require review    |
| User hides discovery         | Discovery visibility off    | Exclude from Discovery               |
| Blocked user views profile   | Block relationship exists   | Show unavailable or restricted state |
| Suspended profile viewed     | Account restricted          | Show unavailable/restricted          |
| Profile hidden by moderation | Moderator action            | Show placeholder                     |
| Media unavailable            | Avatar missing/deleted      | Show fallback state and prompt owner |
| Save fails                   | Network/server error        | Keep form data and allow retry       |



### 16.4. Block Interaction

```txt
- Connection is removed or deactivated according to block policy.
- They do not appear in each other's Discovery.
- They cannot message each other.
- They cannot send greetings.
- Profile view should show unavailable/restricted state.
```

## 21. Security & Privacy

### 21.1. Security Rules

```txt
- User can update only own profile.
- Sensitive verified fields require review/admin correction.
- Public profile resource must be sanitized before rendering.
- Block relationship must be checked server-side.
- Suspended/banned user profiles must be restricted.
- Avatar upload must validate MIME and size server-side.
- Do not trust client-side field visibility.
- Admin-only fields must never be included in public payload.
```

### 21.2. Privacy Rules

```txt
- Full MSSV is never public.
- Email is private by default.
- Phone is private by default.
- Verification evidence is never part of profile.
- Admin notes are never part of profile.
- Report history is never visible publicly.
- User can hide from Discovery.
- User can configure visibility for selected profile fields.
```
